# WPP 多 Agent 协同平台优化过程复盘（详细版）

## 1. 文档目的与适用范围
本复盘用于回答三类问题：
- 我们在这次迭代中到底遇到了哪些真实问题。
- 每个问题是如何被拆解、决策、落地和验证的。
- 这些动作带来了什么收益、还遗留了什么风险。

适用对象：管理层复盘、架构评审、研发交接、后续迭代规划。

## 2. 复盘基线（项目起点）
### 2.1 业务背景
- 场景：Windit多部门协同（人力、销售、财务、软研、硬研、测试、诊断）。
- 管理诉求：BigBoss 需要实时看全局进展、依赖冲突、风险态势和部门指标。
- 协同特点：上报频繁、跨部门依赖强、风险与资源变化快。

### 2.2 初始系统状态
- 已有：7 个部门 Agent + 各自 blackboard（人类可读）。
- 缺失：统一编排层、统一状态层、全局治理、可用于管理决策的看板。
- 主要问题：
  - 信息结构不统一，机器难分析。
  - 协同流程缺少协议约束。
  - 历史变化不可视，复盘靠人工记忆。

## 3. 迭代方法（怎么做）
本轮采用“问题驱动 + MVP 递进”的执行方法：
1. 先定协议与数据模型，保证后续不返工。
2. 再打通最小可运行链路（上报 -> 编排 -> 看板）。
3. 然后持续做“用户视角”体验改造（看板布局、历史复盘、可解释性）。
4. 最后补上跨部门联合分析能力（Boss 二级 RAG）。

## 4. 时间线与关键里程碑
| 阶段 | 主要问题 | 关键动作 | 交付结果 |
|---|---|---|---|
| M1 架构定型 | 系统目标不完整 | 定义五层架构（Agent/编排/状态/看板/治理） | 有了可实施蓝图 |
| M2 协议定型 | 上报口径不一 | 统一 WPP 契约与状态机 | 数据可校验、可计算 |
| M3 分布式化 | 过度单机 | 拆成 7 部门独立服务 + BigBoss 聚合 | 具备独立部署能力 |
| M4 看板可用 | UI 可读性低 | 卡片化、图标化、中文化、进度可视 | 可用于日常运营观看 |
| M5 历史复盘 | 历史功能割裂 | 合并历史切换与差异对比流程 | 复盘路径显著缩短 |
| M6 数据可信 | 写死数据与文本堆叠 | 文件系统 + DB 真值源 + 结构化事件 | 可追溯、可审计 |
| M7 部门运营 | 部门端操作弱 | 部门工作台 + Excel 导入 + 纠错闭环 | 部门可持续运营 |
| M8 联合智能 | BigBoss 仅看聚合结果 | 新增 Boss 联合 RAG（检索/分析 API） | 跨部门因果分析可用 |
| M9 外部访问 | 演示访问不稳 | Cloudflare 隧道接入与脚本化排障 | 演示链路可快速恢复 |

## 5. 详细问题-动作-结果（逐阶段）

## 5.1 阶段一：从“部门自报”到“可编排系统”
### 问题现象
- 部门各自维护 blackboard，但 BigBoss 无法统一调度。
- 缺少任务依赖规则，跨部门推进靠人工沟通。

### 根因判断
- 缺少统一的中台编排层和统一状态模型。
- 缺少可机读的跨部门契约定义。

### 改造动作
- 提出并落地五层架构。
- 把“上报”定义为协议事件而非自由文本。

### 结果
- 协同链路从“松散协作”升级为“可控制流程”。

## 5.2 阶段二：协议与数据模型统一
### 问题现象
- 同样是任务，不同部门字段不同、状态定义不同。
- 同步后无法稳定汇总和比较。

### 根因判断
- 没有统一 schema 与状态机约束。

### 改造动作
- 定义统一对象：Task / BlackboardEntry / CollabContract。
- 定义统一状态机：Backlog -> Ready -> In Progress -> Review -> Done（含 Blocked）。
- 定义 WPP 节拍同步与异常处理（超时/重试/降级）。

### 结果
- 数据具备一致性，可用于编排、统计、回放和审计。

## 5.3 阶段三：从单机模式到分布式模式
### 问题现象
- 早期实现集中在单一服务文件，难以匹配“每部门独立 server”的目标。

### 根因判断
- 架构边界不清晰，运行模式与未来部署模式不一致。

### 改造动作
- 明确“每部门独立进程/端口/数据”的部署方式。
- BigBoss 只做编排与聚合，不承载部门私有逻辑。

### 结果
- 架构与未来生产形态对齐，迁移成本可控。

## 5.4 阶段四：看板从“可显示”到“可决策”
### 问题现象
- 用户反馈：界面丑、纯文本、信息理解慢、按钮含义不清。
- 用户关注点：进度百分比、已完成/待开展、风险可视、部门区分。

### 根因判断
- 展示结构按技术模块拆分，不符合管理者任务路径。

### 改造动作
- 看板视觉改造：图标、部门配色、进度条、中文标签。
- 卡片信息改造：负责人、风险、依赖、KPI、完成项/待开展项。
- 交互可解释：增加按钮提示、状态提示、模块说明。

### 结果
- 看板可读性明显提升，管理层能快速定位关注点。

## 5.5 阶段五：历史与复盘能力重构
### 问题现象
- “一级看板历史”和“历史差异对比”分开，操作割裂。
- 用户点了按钮但看不到结果，复盘效率低。

### 根因判断
- 复盘流程被拆成两个功能块，违背单任务闭环原则。

### 改造动作
- 合并为“时间复盘中心”：
  - 历史轮次切换
  - A/B 轮次选择
  - 差异结果同区展示
- 增加历史导航（上一轮/下一轮）和图表趋势。

### 结果
- 历史回放和差异分析形成连续路径，学习成本降低。

## 5.6 阶段六：数据真值源治理
### 问题现象
- 项目详情写死在服务代码中，维护成本高、不可追溯。

### 根因判断
- 配置与业务数据未解耦，缺少“数据层为真值”的设计。

### 改造动作
- 迁移为“文件系统 + 数据库存储”的双层体系：
  - Markdown 用于可读层。
  - DB 用于可计算层。
- 建立映射文档与架构规范文档。

### 结果
- 数据可维护、可版本化、可审计，减少改代码改业务的耦合。

## 5.7 阶段七：部门工作台能力补齐
### 问题现象
- 部门侧虽然可上报，但缺少日常操作入口（录入、修订、反馈、导入）。

### 根因判断
- 早期重点在 BigBoss 视角，部门端产品能力不足。

### 改造动作
- 建设统一部门工作台：
  - 核心文件管理
  - 上报状态管理
  - 事实录入
  - 历史修订
  - 分析反馈闭环
  - Excel 上传导入

### 结果
- 部门端可独立持续运营，BigBoss 不再承担全部数据维护压力。

## 5.8 阶段八：Agent 分析能力深化（四层）
### 问题现象
- 需要“会追溯、会关联、会预警”，不是“会聊天”。

### 根因判断
- 如果只有文本汇报，没有结构化事实和推理链，无法稳定解释“为什么慢了”。

### 改造动作
- 落地四层分析架构：
  - 事实层：结构化事件入库
  - 记忆层：短期变化 + 长期基线
  - 推理层：规则评分 + LLM表达
  - 汇报层：固定口径 + 证据ID
- 增加反馈闭环，不直接篡改历史事实，保留审计链。

### 结果
- 分析输出从“观点”变成“可核验证据链”。

## 5.9 阶段九：BigBoss 联合 RAG（本轮关键）
### 问题现象
- 部门有 RAG，但 BigBoss 只能看聚合结果，缺乏跨部门联合认知。

### 根因判断
- 缺少二级（Boss 级）索引与检索层。

### 改造动作
- 新增 `boss_rag` 模块：
  - 联合快照入库
  - 联合 PageIndex 构建
  - 联合检索（search）
  - 联合分析（analyze）
- 新增 API：
  - `GET /api/v1/boss-rag/state`
  - `POST /api/v1/boss-rag/ingest-round`
  - `GET /api/v1/boss-rag/page-index/latest`
  - `POST /api/v1/boss-rag/search`
  - `POST /api/v1/boss-rag/analyze`
  - `GET /api/v1/boss-rag/insights/latest`
  - `GET /api/v1/boss-rag/insights/history`

### 结果
- BigBoss 可以做跨部门因果分析与预警，具备“联合认知”能力。

## 5.10 阶段十：公网访问与演示稳定性
### 问题现象
- 外部访问不稳定，出现 Cloudflare 530（origin unregistered）。

### 根因判断
- quick tunnel 具备临时性质，且进程生命周期受运行环境影响。

### 改造动作
- 完成 cloudflared 安装与接入。
- 增加一键脚本：
  - `scripts/start_cloudflare_quick_tunnel.sh`
  - `scripts/stop_cloudflare_quick_tunnel.sh`
- 明确生产建议：named tunnel + 固定域名。

### 结果
- 演示链路可快速恢复，生产演进路径清晰。

## 6. 看板优化专题（产品视角）
## 6.1 从技术结构到用户任务结构
### 早期设计问题
- 模块按“后端数据来源”分块，用户需跨多个区域完成同一任务。

### 重构原则
- 一个用户任务尽量在一个区域闭环完成。
- 固定信息与动态信息分层展示。
- 高价值信息优先（风险、冲突、进度、依赖）。

### 关键改造
- 历史模块合并为“时间复盘中心”。
- 固定治理信息合并为 Tab。
- AI 洞察合并为 Tab。
- 去掉低价值展示，聚焦决策信息。

## 6.2 单部门指标可解释性改造
### 问题
- 指标有值但“用途不清晰”。

### 动作
- 把“用途说明”直接放在每个字段卡片内（字段即解释）。

### 收益
- 不用切换说明区，读一个字段即可理解其管理意义。

## 7. Agent 设计亮点（详细）
## 7.1 角色契约化
- 每个 Agent 显式定义：
  - 角色边界
  - KPI
  - 输入契约
  - 输出契约
  - SLA
- 价值：防止“万能 Agent”失控，保证组织职责清晰。

## 7.2 四层可进化分析框架
- 事实层保障“真实数据来源”。
- 记忆层保障“跨周期连续性”。
- 推理层保障“稳定判断 + 可读表达”。
- 汇报层保障“固定口径 + 管理可读”。
- 价值：从“会回答”升级到“会解释、会复盘、会预警”。

## 7.3 技能扩展与共享调用
- 每部门维护自身 `skills_extends`。
- 技能支持权限共享与跨部门调用。
- 价值：形成可积累的“组织能力资产”。

## 7.4 分布式可部署性
- 每部门独立 server/db/rag。
- BigBoss 聚合而不侵入部门私有实现。
- 价值：扩容和迁移成本低，符合企业演进路线。

## 8. 关键技术决策与取舍
| 决策点 | 备选方案 | 采用方案 | 原因 |
|---|---|---|---|
| 上报格式 | 自由文本 | 强制 JSON schema | 可校验、可计算、可审计 |
| 历史存储 | 覆盖式存储 | Event Log + Snapshot | 同时满足追溯与实时查询 |
| 分析策略 | 纯 LLM | 规则 + LLM | 先保稳定，再提升表达质量 |
| 部署形态 | 单体服务 | 分布式服务 | 贴合部门独立部署目标 |
| 公网演示 | 纯内网 | Cloudflare 隧道 | 低门槛快速外部访问 |

## 14. 典型问题闭环案例（问题 -> 决策 -> 实施 -> 验证）

## 14.1 案例 A：同步周期无法精细化管理
### 业务问题
管理侧提出“不能每次都给 7 个部门统一周期”，需要支持单部门粒度调整，并要求“下发后部门端必须生效”。

### 触发信号
- 用户对“应用同步周期/刷新/立即同步/生成演示数据”存在混淆。
- 期望“调整周期”是唯一主控入口，不希望重复控制项增加认知负担。

### 架构判断
- 如果只在 BigBoss 存设置而不下发，部门端 Reporter 实际节拍不会变化。
- 必须建立 BigBoss -> 部门服务的设置分发链路，并在部门端更新运行中的定时器。

### 实施动作
- BigBoss 增加配置下发能力，支持指定部门或全量部门。
- 部门服务提供同步周期接口并在运行时调用 `REPORTER.set_interval(...)` 生效。
- UI 分离“同步周期管理”与“手动同步动作”，降低误操作。

### 关键文件
- `services/bigboss-service/src/server.py`
- `services/department-service/src/server.py`
- `services/orchestrator-service/web/dashboard.js`

### 验证方式
1. 调用 BigBoss `POST /api/v1/settings/sync-interval` 指定单部门。
2. 查询部门 `GET /api/v1/settings` 确认周期变化。
3. 观察该部门上报时间戳间隔是否变化。

### 结果
- 支持全局/单部门两种模式。
- 周期调整从“配置改了但不生效”变为“配置与运行时一致”。

## 14.2 案例 B：历史查看与差异对比割裂
### 业务问题
用户反馈“点了左右轮次没效果”“看板历史与差异对比像两个系统”，希望“一级看板直接切历史 + 同屏显示差异”。

### 触发信号
- 左右导航交互存在，但未驱动主看板数据源切换。
- 差异对比结果未就地呈现，导致用户看不到实际变化。

### 架构判断
- 历史是“维度”，不是“独立功能页”。
- 需要一个统一的轮次状态源，驱动任务列、部门上报卡、差异结果联动刷新。

### 实施动作
- 合并历史与差异逻辑为同一复盘中心。
- 左右轮次按钮驱动主看板重绘。
- 差异区绑定当前 A/B 轮次并展示增量字段变化。
- 项目级历史扩展为趋势化展示（文字 + 折线趋势）。

### 关键文件
- `services/orchestrator-service/web/index.html`
- `services/orchestrator-service/web/dashboard.js`
- `services/bigboss-service/src/server.py`

### 验证方式
1. 切换轮次，验证任务列（Backlog/Ready/In Progress/Blocked/Review/Done）数量变化。
2. 对比两轮，验证差异区输出任务状态和风险变化。
3. 下钻项目详情，验证历史趋势可见。

### 结果
- 历史回看和差异对比从“分裂流程”改为“单流程闭环”。
- 管理层可以在同一屏完成“回看 -> 对比 -> 下钻”。

## 14.3 案例 C：部门 Agent 只有上报，没有可解释分析
### 业务问题
用户强调部门 Agent 核心能力应是“会追溯、会关联、会预警”，并需要财务汇报数字分身能力。

### 触发信号
- 仅有文本上报无法回答“为什么慢了/和上轮关系是什么”。
- 项目数据主要停留在展示层，缺少事件级结构化沉淀。

### 架构判断
- 必须采用四层分析架构，并保持事实与反馈分离：
  - 事实事件不可直接覆盖。
  - 纠错应形成可追溯修订链。

### 实施动作
- 建立事实层数据结构并落库（项目事件）。
- 增加历史修订接口与修订记录展示。
- 引入反馈闭环，让反馈进入后续分析上下文。
- 固定汇报模板输出“结论-依据-影响-建议”。

### 关键文件
- `services/department-service/src/server.py`
- `services/README_DISTRIBUTED_WPP.md`
- `services/orchestrator-service/web/department.html`
- `services/orchestrator-service/web/department.js`

### 验证方式
1. 导入 Excel/事件数据后，检查分析输出中是否带证据与历史关联。
2. 提交反馈，检查下一轮分析是否引用 `recent_feedback`。
3. 执行事件修订，检查修订记录和时间线是否可见。

### 结果
- 部门 Agent 从“上报器”升级为“可解释分析器”。
- 形成“事实 -> 分析 -> 人工反馈 -> 再分析”的闭环。

## 17. 复盘方法论沉淀（可复用到下一项目）
1. 先统一协议再堆功能，避免后续返工。
2. 先做“用户任务闭环”再做“技术模块完备”。
3. 历史能力必须一开始设计，不要后补。
4. AI 能力要以“可核验”优先，避免无证据结论。
5. 分布式边界尽早明确，避免后期拆分成本。

## 18. 补充结论
本次优化过程的核心，不是“功能叠加”，而是把系统从“页面可看”升级为“管理可用”：
- 可用体现在：周期可控、依赖可控、风险可控。
- 可管体现在：历史可查、差异可比、责任可追。
- 可进化体现在：部门独立迭代、技能可沉淀、Boss 可联合分析。
