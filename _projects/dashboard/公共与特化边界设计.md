# 公共与特化边界设计

## 核心原则

这套系统不追求“大统一部门应用”，而是追求：

- 平台层足够稳定
- 部门层足够自由
- BigBoss 只消费标准出口

公共层不是“尽量多放东西”，而是“只放长期稳定、跨部门反复复用且不依赖具体业务语义的能力”。

## 什么必须抽象成公共能力

以下能力应持续留在平台层：

- 登录认证与会话管理
- WPP 协议与网络通信格式
- 配置读取、环境变量展开、运行时路径解析
- 部门注册表与部门发现
- 通用规则 DSL 与规则执行引擎
- 元数据中心与指标/Schema 语义映射骨架
- Query Router / Intent Dispatcher 骨架
- Hybrid Retrieval 与重排序骨架
- 工作台/经营看板的基础壳
- 审计、日志、健康检查、备份恢复框架

这些能力的共同点是：

- 不依赖财务、销售、诊断等具体业务语义
- 换一个部门也成立
- 长期看更趋于稳定，而不是更趋于发散

## 什么应该留在部门产品层

以下能力默认属于部门特化：

- 部门专属数据模型
- 文件上传后的私有解析逻辑
- 部门风险判断补充逻辑
- 工作台私有模块
- 经营看板私有观察模块
- 部门专属前端资源与交互
- 部门私有 RAG 策略与知识组织方式
- 部门私有指标口径、Glossary 与 SQL 示例
- 部门私有 Intent 词典与 Query 路由偏好
- 部门私有 Excel 导入口径与字段别名
- 部门私有默认规则模板与治理写法

这些能力的共同点是：

- 业务语义明显依赖部门
- 后续更可能持续演化，而不是收敛
- 如果放进公共层，会不断出现 `if finance / if sales / if diagnostic`

## 架构判断标准

新增一个能力时，先问 4 个问题：

1. 这个能力换一个部门后还成立吗？
2. 它是否依赖部门私有业务语义？
3. 它未来更可能收敛，还是更可能发散？
4. 如果把它放进公共层，是否会迫使其它部门理解不属于自己的概念？

如果答案偏向：

- “跨部门成立”
- “不依赖私有语义”
- “未来趋于稳定”

那它适合公共化。

如果答案偏向：

- “只在本部门成立”
- “强依赖私有语义”
- “未来会持续分化”

那它应该留在部门产品层。

## 当前改造结论

后续新需求的默认落点应是：

- 先放到部门产品层
- 只有在确认“多个部门稳定复用”之后，才考虑上收平台层

这是避免平台被业务特例拖垮的关键。

## 控制面边界

BigBoss 只应该消费：

- 部门注册表
- 显式网络配置
- WPP 标准上报

不应该在代码中内建“人力一定是 9101、销售一定是 9102”这类部门启动假设。

当前约束已经调整为：

- 部门地址以 `server.yaml` 中的 `department_urls` 为准
- 前端展示以 `/api/v1/departments` 返回的注册表为准
- 不再向前端暴露部门内部 `product_root` 路径

## 私有 API 原则

部门特有模块如果需要后端接口，不应该直接回写到公共 `department-service` 业务逻辑里，而应该通过部门产品扩展暴露。

推荐方式：

- 公共骨架只提供统一挂载点，例如 `/api/v1/product/modules/*`
- 各部门在自己的 `plugins/extension.py` 中实现具体接口
- BigBoss 只消费标准化 WPP 输出，不依赖这些部门私有接口

当前样板：

- 财务私有接口：`/api/v1/product/modules/salary_center`
- 诊断私有接口：`/api/v1/product/modules/turbine_health_grid`

这样做的意义：

- 财务想扩薪资、预算、回款，不需要改别的部门
- 诊断想扩设备健康、根因链，也不需要污染平台公共层
- 公共服务只负责“挂载和鉴权”，不负责“部门业务本身”

## RAG 与查询分流边界

当前 RAG 与查询能力按下面的边界组织：

- 平台层提供：
  - `Metadata Center`
  - `Intent Dispatcher`
  - `Hybrid Retrieval`
  - `Text-to-SQL` 骨架
- 部门产品层提供：
  - 指标口径与 glossary
  - schema 描述
  - SQL 示例
  - intent 词典
  - pageIndex 章节结构
  - Hybrid Retrieval 权重

当前实现落点：

- 元数据中心：`$WPP_ROOT/platform/common/metadata_center.py`
- 查询分流：`$WPP_ROOT/platform/department_kit/query_runtime.py`
- 混合检索：`$WPP_ROOT/platform/department_kit/retrieval_runtime.py`
- 对外接口：
  - `GET /api/v1/wpp/metadata`
  - `POST /api/v1/wpp/query`
  - `GET /api/v1/agents/{agent}/metadata`
  - `POST /api/v1/agents/{agent}/query`

约束原则：

- 平台层只提供查询能力骨架，不定义部门指标口径
- 部门产品层负责提供 glossary / schema / metric / SQL example
- BigBoss 只通过部门标准查询接口消费，不理解部门内部实现细节

## Skills 作为本地运行真值源

当前运行时已经开始优先从各自边界里的 `skills_extends/` 读取行为约束，而不是集中维护一份全局 skill 仓。

当前约束：

- BigBoss 只读取：
  - `$WPP_ROOT/services/bigboss-service/skills_extends/`
- 各部门只读取：
  - `$WPP_ROOT/departments/<department>/skills_extends/`

已经本地化进入运行链的 skill 类别：

- `query-routing`
- `llm-output-guardrails`
- `fact-ingestion`
- `rulebook-authoring`

这意味着：

- Query 路由关键词不再由共享代码写死
- LLM 输出约束不再由共享代码写死
- Excel 导入口径和字段别名不再由共享代码写死
- 默认规则模板不再由共享代码写死

平台层只负责：

- 提供读取这些本地 skill profile 的骨架
- 提供 merge 和兜底逻辑

不负责：

- 替部门决定具体导入口径
- 替部门决定默认规则模板
