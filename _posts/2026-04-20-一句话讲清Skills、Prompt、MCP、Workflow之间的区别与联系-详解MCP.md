# 二、什么是MCP？CLI？



**通过 stdio 的 MCP**（大多数开发人员在本地运行的版本）是问题累积的地方：进程间通信不稳定、环境隔离混乱、令牌开销大。在这种情况下，几乎所有用例都有更好的替代方案。

**而 HTTP 上的 MCP**则不同。企业内部工具平台需要集中的权限管理、统一的 OAuth、标准化的遥测和日志记录。零散的 CLI 工具确实难以提供这些功能。在这种情况下，MCP 的集中式架构具有真正的价值。

Perplexity 实际上放弃的主要是 stdio 用例。丹尼斯-亚拉兹（Denis Yarats）指定的是 "内部"，并没有要求整个行业采用这一选择。这一细微差别在传播过程中丢失了--"Perplexity 放弃 MCP "的传播速度远远快于 "Perplexity 在内部工具集成中将 MCP 排除在 stdio 之外"。

MCP 的出现是因为它解决了一个实际问题：在它出现之前，每个人工智能应用程序都编写自己的工具调用逻辑，没有共享标准。MCP 在适当的时候提供了一个统一的接口，生态系统迅速建立起来。随后，生产经验让局限性浮出水面。这是基础架构工具的正常发展轨迹，而不是死刑判决。



# 三、MCP的厉害之处







## MCP最大的问题在哪？

很多人/公司在2026年已经开始鼓吹“MCP”已死：https://milvus.io/zh/blog/is-mcp-dead-cli-and-skills-for-ai-agents.md

究其原因，最大的问题除了其糟糕的权限管理以外，MCP tools的调用对于上下文窗口的挤占是难以接受的。

![img](https://assets.zilliz.com/is_mcp_dead_cli_and_skills_for_ai_agents_md_3_767d46c583.png)



举个例子，**使用MCP调用文件系统工具，批量读取文件内容并分析**。

1. **工具描述与权限声明**：MCP在调用文件系统前，需要将工具描述（如“文件系统工具”，支持读取、写入、列举文件等）、权限要求（如“只读”）、输入输出Schema等信息传递给模型
2. **操作历史与上下文快照**：若用户已在前几轮对话中多次操作文件系统，MCP会将历史操作、结果快照、任务状态等全部嵌入到新的请求上下文中。
   1. “请帮我依次读取文件A、修改文件B，然后把结果保存到文件C。”MCP工具会这样处理：读取operation_history，知道内容A和内容B已修改分别来自哪里。检查context_snapshot，确认当前文件状态和权限。执行合并操作并保存到fileC.txt。更新operation_history和context_snapshot。
3. **工具调用与响应内容**：MCP返回的响应内容也结构化并详细，包括文件内容、状态码、错误信息等。假如一次批量读取多个文件，响应内容会非常庞大。

如果模型上下文窗口限制为4096 token，可能仅仅两三次MCP工具调用就会把窗口塞满，模型不得不丢弃早期内容或压缩上下文，影响任务连贯性和准确性。



所以这才激生出skills。



![img](https://assets.zilliz.com/is_mcp_dead_cli_and_skills_for_ai_agents_md_4_f80de07814.png)

1.**上下文消耗更小**

- **Skills**通常采用“函数调用”或“API调用”机制，模型只需传递本轮必要参数，不需要每次都带上完整的操作历史和所有环境快照，大幅减少token消耗。
- **MCP**则需要反复传递结构化的历史和快照，token使用量大，窗口容易被“挤爆”。

2. **接口简洁，开发体验友好**

- Skills的接口设计更贴近传统API，开发者只需定义输入输出格式，模型会自动管理上下文，大幅降低开发和维护成本。
- MCP的结构更复杂，开发者需要手动管理历史和快照，增加了实现和调试难度。

3. **扩展性和兼容性更强**

- Skills支持插件化、模块化，可以灵活组合和扩展新能力，易于对接外部系统。
- MCP由于上下文传递臃肿，扩展大量工具时容易遇到性能瓶颈。

4. **响应速度和准确率提升**

- Skills因上下文精简，推理速度更快，模型更专注于当前任务，准确率更高。
- MCP因上下文冗余，容易引入无关信息，影响模型判断。

5. **生态与标准化**

- Skills已成为主流大模型（如OpenAI、百度、讯飞等）支持的标准插件协议，生态丰富，文档完善。
- MCP多见于早期或自定义系统，标准化程度较低。





还有一个问题是，MCP只能单纯完成任务，LLM的能力并没有在MCP中融合。

当我们构建[claude-context](https://github.com/zilliztech/claude-context)时（这是一个 MCP 插件，可为 Claude Code 和其他人工智能编码代理添加[语义搜索](https://zilliz.com/glossary/semantic-search?__hstc=175614333.51d1452db000e5cd352be8b93e8f120c.1776934469409.1776934469409.1776934469409.1&__hssc=175614333.1.1776934469409&__hsfp=8c84e8e4f19b4f556a58866f11b3560b)，让它们从整个代码库中获得深度上下文），我们希望从 Milvus 中检索相关的历史对话片段，并将其作为上下文浮现出来。

检索前 10 个结果，可能有 3 个是有用的。其他 7 个是噪音。如果将所有 10 个结果都交给外部 Agents，噪音就会干扰答案。在测试中，我们发现响应会被无关的历史记录干扰。我们需要在传递结果之前进行过滤。

我们尝试了几种方法。使用小型模型在 MCP 服务器内添加重排步骤：不够准确，相关性阈值需要根据使用情况进行调整。使用大型模型进行重排：技术上可行，但 MCP 服务器作为独立进程运行，无法访问外部 Agents 的 LLM。我们必须配置单独的 LLM 客户端，管理单独的 API 密钥，并处理单独的调用路径。

其实想要的很简单：让外部 Agents 的 LLM 直接参与过滤决策。检索前 10 条，让 Agents 自己判断哪些值得保留，然后只返回相关结果。无需第二个模型。无需额外的 API 密钥。

检索skills可以解决这个问题，它可以再去调用向量搜索（rag-search）前 10 名，让 Agents 自己的 LLM 评估相关性，然后只返回通过的结果。无需额外的模型。Agents 自己完成过滤。

# PS：如何产生生产力？



## 1、前端到底g没g？

当前cli盛行，但是有没有思考过一个问题，当为什么要去掉界面呢？之前我一直在纠结为什么要取代mcp，现在我更觉得，这是在另一种维度取代前端？一方面前端代码不需要人了，另一方面，前端的展示方式也不需要了？(你提到的现象，本质上是**从 GUI（图形用户界面）向 LUI（自然语言用户界面，Language User Interface）或 CUI（对话式用户界面）的演进**。)

之前我对于这个感觉并没有清晰的认知，直到公司运维团队推出了他们的skills，一切可以基于“自然语言”去进行运维的时候（“帮我下掉xx机器”、“帮我分析xx服务容量，并给出缩容和弹性方案”、“帮我查一下xxx服务的主机列表”、“帮我申请域名访问权限”、“帮我评估一下这个集群是否需要扩容”），我恍然大悟&释然了。

这些功能不就是之前各种运维平台做的事情吗？以后都通过cc一句话解决之后，那这些平台的意义在哪呢？而且，这些功能是可以**“重新”**（记住，是重新）用cli解决问题，那为什么不用平台可以更方便的操作呢？hh，不知道开发这些skills的同学，会不会也有这样的思考煎熬，真的非常魔幻。这么做的目的，是真的产生了额外的生产力吗？没感觉出来多少（部分skill，比如评估容量&扩容，确实加速了运维）。同时，再回过头来看，这么做也确实是一次好的探索，旧平台仍然保留，后端逐渐为AI丰富接口。（“我们辛辛苦苦做了几年的平台，最后变成了一个给大模型调用的 API？”）



>  当然，我绝不是顽固的守旧派，我对于AI有疯狂的热枕！

**前端并没有死，而是升维了。** AI 擅长处理逻辑和文本，但人类依然是视觉动物。未来的前端会高度集中在**“AI 无法用文字表达清楚的领域”**，比如：复杂的数据可视化（动态拓扑图、3D 机房展示）、高交互的富文本/白板编辑器、以及沉浸式的空间计算（XR）界面。前端将从“交互的搬运工”变成“数据的视觉解释者”。

但真正的生产力爆发，在于**“跨系统的编排（Orchestration）”**和**“结合上下文的推理（Reasoning）”**：

- **以前排查故障（GUI时代）：** 运维同学需要打开监控平台看 CPU，打开日志平台搜 Error，打开发布平台看最近的变更，然后在脑子里综合这些信息。
- **现在排查故障（Agent时代）：** 一句话“帮我分析 xx 服务为什么报警”。AI 会**自动**去调用监控、日志、发布平台的 Skills，综合分析后告诉你：“因为 10 分钟前张三发布了新版本，导致数据库连接池耗尽，建议立刻回滚。”

**额外的生产力产生于：AI 替代了人类在多个系统之间来回穿梭、搬运数据、并进行基础逻辑推理的过程。**



这场争论的真正分野，不在于技术，而在于开发的组织形态。它区分了两种开发者：单打独斗的“感觉编程”（vibe-coding）信徒，和面向组织的“智能体工程”（agentic engineering）实践者。这才是MCP真正发力的地方——不是本地`stdio`模式的小打小闹，而是作为中心化服务器通过HTTP提供的服务。它把认证（Auth）、安全（Security）、遥测（Telemetry）这些麻烦事一揽子解决了。工程师离职？吊销他的OAuth令牌即可，他从未接触过核心密钥。这对于任何依赖GitHub Actions这类临时运行环境的团队来说，更是刚需。

更有趣的是，连Anthropic和Cloudflare都发现，让LLM直接调用MCP，不如让LLM“写代码去调用MCP”来得更稳、更省。Anthropic的“程序化工具调用”甚至能节省高达98.7%的Token。这说明，MCP的价值在于提供了一个稳定的、可被机器理解的“契约”，而不是一个手感舒适的“玩具”。**所以，当人们在激烈争论MCP和CLI的优劣时，他们实际上在无意中暴露了自己的立场：他们究竟是在构建一个随时可丢弃的个人项目，还是在为一个需要长期维护、多人协作的系统打地基？这根本是两条路线的斗争。**



## 2、到底是真的生产力？还是无聊的工作量？

前线的工作人员为什么现在感觉工作更重了？（“过渡期税”）

在旧系统完全被淘汰之前，开发者被迫在两个时代之间劈叉：

1. **双线作战的负担**：正如你所说，旧的 GUI 平台不能停，新的 AI Skill 又要上。开发者既要修前端 Vue/React 的 Bug，又要去写后端的原子化 API 供大模型调用。
2. **确定性向概率性的妥协**：传统代码是“确定性”的（If A then B）。但接入 AI 后，系统变成了“概率性”的。开发者需要花大量时间去处理大模型的“幻觉”，写大量的防御性代码（Guardrails）、复杂的 Prompt 约束，以及极其严格的权限控制（RBAC），以防 AI 误删库或执行危险操作。
3. **接口描述的额外成本**：以前接口写得烂，前端同事口头沟通一下也能调通。现在接口是给 AI 用的（比如接入 MCP 协议），你必须写极其标准、详尽的 OpenAPI 描述、参数释义，否则 AI 根本不知道怎么调用。



当基础设施完善后，开发的工作量不会是简单的叠加，而是会发生**结构性的转移**。

**1. 前端工作量的大幅削减（CRUD 的消亡）**
过去，业务开发中 60% 以上的工作量是极其枯燥的 CRUD（增删改查）：画表单、写校验逻辑、做列表分页、调接口联调。
当 LUI（自然语言界面）成为主流，这些工作将直接消失。用户一句话，AI 自动提取参数并调用后端。前端不再需要为每一个小功能开发特定的页面，前端的精力将彻底释放，转向极其复杂的交互（如 3D、复杂数据可视化、画布等 AI 无法用语言表达的领域）。

**2. 后端开发的“原子化”与“去胶水化”**
以前，为了配合不同的前端页面，后端需要写大量的 BFF（Backend For Frontend）胶水层代码，把各种数据拼凑成前端想要的形状。
未来，后端只需要提供**最纯粹、最原子的 API（Capabilities）**。后端不需要关心数据怎么展示，不需要写胶水代码，只需要把底层能力暴露出来，剩下的业务编排（Orchestration）全部由 AI Agent 根据用户的自然语言指令动态完成。**后端的接口复用率将达到前所未有的高度。**



### 3、从“流水线工人”到“规则制定者”？

我们不能忽略一个关键变量：**你在给 AI 写接口的同时，AI 也在帮你写代码。**

根据行业观察，[AI 对软件工程的影响是全方位的](https://blog.csdn.net/ChailangCompany/article/details/146476456)。虽然系统架构变复杂了，但开发者手里的工具也变强了：

- **代码生成的效率革命：** 自动化编程工具（如 GitHub Copilot）已经能接管大量重复性劳动，甚至能根据上下文自动生成代码片段或完整函数，极大提高了开发效率。
- **自动化测试与审查：** AI 能够自动生成测试用例、进行智能调试，甚至在代码提交时自动审查潜在的逻辑错误和[性能问题](https://worktile.com/kb/ask/2041932.html)。
- **架构辅助：** AI 甚至能根据海量数据为开发团队提供架构设计建议，预测潜在瓶颈



所以，开发的工作是变重了吗？

- **短期看，是的。** 因为我们正处于新旧交替的泥沼中，既要背负历史包袱，又要探索未知。

- **长期看，不是变重了，而是门槛变高了，性质改变了。**
  未来的开发者将不再是把业务逻辑翻译成代码的“打字员”。那些低价值的、重复性的“搬砖”工作（如写表单、写简单的增删改查）会被彻底消灭。开发者的核心工作将演变为：

1. **定义核心领域模型与原子能力**（写出高质量、高内聚的 API）。
2. **制定规则与边界**（确保 AI 调用的安全性、合规性和权限控制）。
3. **解决极其复杂的系统工程问题**（高并发、底层性能优化、复杂架构设计）。
