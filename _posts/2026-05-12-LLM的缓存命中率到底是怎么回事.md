> 大模型 LLM 缓存机制：KV Cache、PagedAttention 与 Prompt Cache （RadixAttention（基于前缀树的自动缓存）、Claude API 的 cache_control显式缓存断点）技术解析（核心机制：**前缀匹配**）

| 技术           | 解决的问题                      | 关键效果                     |
| -------------- | ------------------------------- | ---------------------------- |
| KV Cache       | 单次推理内部历史 token 重复计算 | 计算复杂度 O(L^2) \to O(L)   |
| PagedAttention | KV Cache 显存碎片化与共享困难   | 显存利用率 ≥ 90%，并发的提升 |
| Prompt Cache   | 跨请求相同前缀重复 Prefill      | TTFT 和推理成本显著降低      |

可引用的实测数据：

- PagedAttention（vLLM 早期版本）vs HuggingFace Transformers：吞吐提升 14–24 倍。
- Anthropic Prompt Cache：书籍级长提示延迟降低约 79%，缓存部分成本降低 90%。
- 跨提供商 Agent 负载测试：成本降低 41%–80%，TTFT 改善 13%–31%。





这个的原始逻辑是，大语言模型的自回归生成过程存在一个基础性性能问题。生成第t+1个 token 时，模型需要基于当前 token 的 Query与所有历史 token 的 Key和 Value进行计算（也就是经典的QKV问题）。如果不做任何优化，每个生成步骤都需要对前 t个 token 重新计算其 Key 和 Value 投影。整体的时间复杂度是O（Len*2），直接导致推理延迟上升和 GPU 算力浪费。

## KV缓存

所以最简单的一个是**KV缓存**（ KV Cache：以显存换计算），已生成 token 的 Key 和 Value 向量在后续步骤中保持不变。KV Cache 将每个 token 在各层的 Key 和 Value 计算结果存储在 GPU 显存中，后续步骤直接从缓存读取，不再重复计算。Prefill（预填充）：对输入 prompt 的所有 token 并行计算 K、V，存入缓存。
Decode（解码）：每生成一个新 token，仅计算该 token 的 K、V，追加至缓存。注意力计算时，新 token 的 Q 与缓存中的全部 K、V 进行交互。

但是当上下文窗口扩展至 128K token，单请求缓存可超过 40 GB，严重限制并发能力。

这是因为传统 KV Cache 分配为每个请求预留连续的显存空间，长度按最大可能输出配置。内部碎片：实际生成长度通常远小于预留上限，未使用空间浪费。外部碎片：不同请求释放的显存块大小不一，形成无法利用的小空洞。共享困难：多个请求的相同前缀无法复用同一份缓存。



## PagedAttention（操作系统的虚拟内存分页思想）

所以引入了PagedAttention（操作系统的虚拟内存分页思想），将 KV Cache 切分为固定大小的 Block（通常每块容纳 16 个 token 的 KV 数据）。这种设计的效果：按需分配，序列每满一个 Block 才申请新块，无内部碎片。所以可以看出来，**PagedAttention 不是和 KV Cache 并列的，而是 KV Cache 的一种"内存管理方案"**

```
标准 Attention 计算：
  Attention(Q, K, V) = softmax(QK^T / √d) × V
  要求 K、V 必须连续存放

PagedAttention 计算：
  把 K、V 分散在多个 Block 里
  计算时逐块处理：
  
  Step1: 取 Query 向量 Q
         ↓
  Step2: 遍历 Block Table，逐块取出 K
         Q × K_block1 → 部分注意力得分
         Q × K_block2 → 部分注意力得分
         Q × K_block3 → 部分注意力得分
         ↓
  Step3: 合并所有部分得分，做 softmax
         ↓
  Step4: 用权重对各 Block 的 V 加权求和
         ↓
  Step5: 得到最终 Attention 输出
```





这个PagedAttention设计，最重要的就是**他对于KVcache共享机制是copy-on-write（写时复制），这也是KV缓冲中最精彩的部分。**

> 在 Unix/Linux 系统中，当父进程创建子进程时，操作系统不会立即复制父进程的所有内存空间。相反，父子进程会共享相同的物理内存页，并将页面标记为“只读”。只有当其中一个进程试图向内存写入数据时，才会触发**缺页中断（Page Fault）**，系统此时才会复制该内存页。
>
> **优势：** 大幅缩短进程创建时间，减少不必要的内存拷贝，避免内存资源浪费。
>
> 举例：Java 并发包中提供了基于 COW 思想的容器，适合**读多写少**的并发场景。
>
> KV Cache 是"爸爸"        → 提出了缓存这件事
>
> PagedAttention 是"儿子"  → 把缓存这件事做得更好 

场景：100个用户同时发请求，System Prompt 都一样。

```
System Prompt（假设很长）：
"你是一个专业的编程助手，你需要...（很长很长）"

用户1问："帮我写个冒泡排序"
用户2问："帮我写个快速排序"  
用户3问："帮我写个二分查找"
...
用户100问："帮我写个链表"
```

```
❌ 没有写时复制：
用户1的KV Cache:
[系统提示的KV... 1000个Block] + [冒泡排序问题的KV]

用户2的KV Cache:
[系统提示的KV... 1000个Block] + [快速排序问题的KV]

用户3的KV Cache:
[系统提示的KV... 1000个Block] + [二分查找问题的KV]

...

用户100的KV Cache:
[系统提示的KV... 1000个Block] + [链表问题的KV]


✅ 有写时复制：

物理内存里：

┌─────────────────────────────────┐
│   系统提示的KV Block（只存1份）   │  ← 100个用户共享这一份
│   Block#1, Block#2, ... #1000   │
└─────────────────────────────────┘
        ↑        ↑        ↑
      用户1    用户2    用户3  ...（都指向同一块物理内存）

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│用户1独有的KV  │  │用户2独有的KV  │  │用户3独有的KV  │
│[冒泡排序的KV] │  │[快速排序的KV] │  │[二分查找的KV] │
└──────────────┘  └──────────────┘  └──────────────┘
```

"写时"才复制，不写不复制。所以，Claude Code 里 Cache: 97% 这么高。原因之一就是：  每次请求的 System Prompt 都一样
  → PagedAttention 的写时复制让这部分KV共享
  → API层识别到前缀一致
  → 直接命中缓存，不重新计算
  → 只收 0.1x 的费用





## Prompt Cache：跨请求前缀复用

在线服务中存在大量跨请求的相同前缀：系统提示词、RAG 注入的长文档、工具定义等。若每次请求都对这类前缀重新进行 Prefill 计算，将产生大量冗余计算。Prompt Cache 通过持久化前缀的 KV Cache，使后续请求直接复用，跳过 Prefill 阶段，从而降低首 token 延迟（TTFT）和推理成本。





具体的实现过程是：

```
┌─────────────────────────────────────────────┐
│                                             │
│   Claude Code 框架层  ←→  Anthropic API层   │
│   （决定"怎么发"）        （决定"怎么算"）    │
│                                             │
└─────────────────────────────────────────────┘
```

**Claude Code 框架负责：构造请求、标记哪些内容可以缓存**

Claude Code 在发送请求时，会在消息结构里加上 `cache_control` 字段，**主动告诉 API："这段内容请帮我缓存"**：

```
{
  "type": "text",
  "text": "...系统提示内容...",
  "cache_control": {
    "type": "ephemeral"   ← 框架加的标记
  }
}
```

框架层做的事：

```
✅ 决定请求结构的排列顺序（稳定内容放前面）
✅ 给 System Prompt 打上 cache_control 标记
✅ 给 Tools 工具列表打上 cache_control 标记
✅ 管理子 Agent 继承父 Agent 的对话历史（保持前缀一致）
✅ 维护 CacheSafeParams（确保关键参数一致性）
```

LLM 内核（Anthropic API）**负责：真正执行缓存的存储与读取**，框架只是"申请缓存"，真正把 token 计算结果存起来、下次复用，是 **API 服务端**完成的：

```
1. 接收到带 cache_control 的请求
       ↓
2. 计算这段 prompt 的哈希/前缀指纹
       ↓
3. 检查缓存池里有没有一样的前缀
       ↓
   命中 → 直接返回缓存的 KV 值（跳过重新计算）
   未中 → 重新计算，并写入缓存池
```

由于缓存是字节级别的，哪怕差一个空格，整段缓存就失效！如“"你好世界"  ✅ 命中，"你好世界 " ❌ 失效（多了个空格）”

这就是为什么 Claude Code 框架要用 `CacheSafeParams` 类型，**显式声明哪些参数必须保持一致**：

```
// Claude Code 源码中的设计
CacheSafeParams = {
  system_prompt,   // 必须一致
  tools,           // 必须一致  
  model,           // 必须一致
  messages,        // 必须一致
  thinking_config  // 必须一致
}
```





## 如何保持高缓存命中率？

```
✅ 好习惯：
1. 不要频繁修改 CLAUDE.md / System Prompt
2. 保持工具列表稳定，不要随意增删
3. 同一任务在一个 session 内完成，不要频繁新开对话
4. 避免频繁使用 --resume 恢复旧会话（有潜在 Bug）
5. 让对话内容尽量连续，不要中断太久（避免5分钟TTL过期）
```



## 缓存的生命周期和层级

一个经典问题，这个缓存是什么层面的，session吗？假如我已经换了session，之前session缓存都会失效吗？

**不是 Session 层面的，是 API 请求层面的，跨 Session 也可以命中！**因为，缓存不在你本地，不在 Session 里，而是存在 Anthropic 的服务器端。服务器不认识"你是谁"、"这是第几个Session"，它只认一件事：

```
本次请求的 Prompt 前缀
        ↓
和服务器缓存里存的前缀
        ↓
是否字节级完全一致？

一致  → 命中！不管你是不是同一个Session
不一致 → 未命中，重新计算
```

所以，缓存能不能跨 Session 命中，**本质上是时间问题，不是 Session 问题**：

```
第1次请求
  → 写入缓存
  → 开始计时 ⏱️
        ↓
   5分钟内有新请求命中
  → 缓存续期，重新计时 ⏱️
        ↓
   5分钟内又命中
  → 再续期...
        ↓
   超过5分钟没有任何请求命中
  → 缓存过期销毁 💨
```



所以可以看到：

```
┌─────────────────┬──────────┬────────────────────────┐
│   内容类型       │ 能否缓存  │ 说明                    │
├─────────────────┼──────────┼────────────────────────┤
│ System Prompt   │  ✅ 能   │ 最稳定，最容易命中        │
│ Tools 工具定义   │  ✅ 能   │ 较稳定，容易命中          │
│ 对话历史        │  ✅ 能   │ 同Session内可命中         │
│ 本次新消息      │  ❌ 不能  │ 每次都变，无法缓存        │
│ 跨Session历史   │  ❌ 不能  │ 新Session历史为空        │
└─────────────────┴──────────┴────────────────────────┘
```

**缓存跟 Session 无关，跟"请求前缀是否一致"以及"5分钟内有没有续期"有关。**
换了 Session，System Prompt 不变的部分照样能命中；但对话历史因为新 Session 是空的，自然就没有了。所以尽量在一个 Session 内完成任务，比频繁新开 Session 要省钱得多（同时也会快很多，推理速度加速）。





## 总结一下

**你在 Claude Code 状态栏看到的 `Cache: 97%`，本质上反映的是 Prompt Cache 的命中情况，而它能高效运作，背后依赖的是 KV Cache 机制，KV Cache 能装下更多请求，靠的是 PagedAttention 的存储优化。**

三者不是并列关系，是**不同抽象层级**的概念：

```
┌─────────────────────────────────────────┐
│            业务应用层                    │
│                                         │
│          Prompt Cache                   │
│   （跨请求复用、计费、TTL管理、API接口）   │
│                                         │
├─────────────────────────────────────────┤
│            推理引擎层                    │
│                                         │
│            KV Cache                     │
│      （缓存K、V向量，避免重复计算）        │
│                                         │
├─────────────────────────────────────────┤
│            内存管理层                    │
│                                         │
│          PagedAttention                 │
│    （决定KV Cache如何在显存里存储）        │
│                                         │
├─────────────────────────────────────────┤
│              硬件层                      │
│           GPU 显存 / CPU 内存            │
└─────────────────────────────────────────┘
```



解决的问题也不一致：

```
KV Cache 解决：
  "同一个请求内，已经算过的Token不要重复算"
  关注点 → 计算效率

PagedAttention 解决：
  "KV Cache占用的显存太碎、太浪费"
  关注点 → 存储效率

Prompt Cache 解决：
  "不同请求之间，相同的前缀不要重复算"
  关注点 → 跨请求复用
```





举例一次真实的请求，

```
你发请求
    ↓
Prompt Cache 层：
  检查 System Prompt 前缀是否命中跨请求缓存
  命中 → 直接返回，收0.1x费用
  未中 → 往下走
    ↓
KV Cache 层：
  计算新Token时，读取已缓存的K、V向量
  避免重复计算当前请求内的历史Token
    ↓
PagedAttention 层：
  把所有K、V向量用分页方式存在显存里
  保证内存利用率接近100%
    ↓
GPU 实际计算
```
