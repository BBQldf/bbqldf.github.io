> 说到数据库，首先是要连接上它！怎么连接，就是用到了**JDBC这种模板方法。**

### -1.2 JDBC为什么要违反“双亲委派”机制？

> 其实不仅是JDBC（驱动加载这一过程）打破了这个流程，Tomcat也打破了。
>
> 我们可以看看JDBC的驱动加载流程，就知道它的问题所在了。
>
> 核心：SPI机制。违反“双亲委派”机制，更多原理可以参考`JavaReview02 - 一、java关键字&java 类加载 - 4.2 哪些操作是违背了“双亲委派”机制的`

细化一下驱动加载的过程：

1. Class.forName(“com.mysql.jdbc.Driver”)：在创建连接之前必须执行该代码，用来加载、并初始化（执行static代码）mysql驱动类。然后**生成驱动实例对象，**并向DriverManager注册（将实例添加到list中）

```java
package com.mysql.jdbc;
import com.mysql.jdbc.NonRegisteringDriver;
import java.sql.DriverManager;
import java.sql.SQLException;
public class Driver extends NonRegisteringDriver implements java.sql.Driver {
	public Driver() throws SQLException {
	}
	static {
		try {
			DriverManager.registerDriver(new Driver());
		} catch (SQLException arg0) {
			throw new RuntimeException("Can\'t register driver!");
		}
	}
}
```

2. 调用DriverManager.registerDriver()方法之前肯定得先初始化DriverManager类，使用的就是静态代码块：

```java
static {
    loadInitialDrivers();
    println("JDBC DriverManager initialized");
}
```

3. loadInitialDrivers方法会通过两种方式加载驱动实现类：

   1. 使用Class.forName加载jdbc.drivers系统变量中指定的驱动实现类； 

   2. 使用SPI加载驱动实现类；

可以看一下代码：

```java
private static void loadInitialDrivers() {
    String drivers;
    try {//1、获取系统变量中的驱动实现类名，然后通过下面的Class.forName加载
        drivers = AccessController.doPrivileged(new PrivilegedAction<String>() {
            public String run() {
                return System.getProperty("jdbc.drivers");
            }
        });
    } catch (Exception ex) {
        drivers = null;
    }
    //2、使用SPI获取驱动实现类
    AccessController.doPrivileged(new PrivilegedAction<Void>() {
        public Void run() {
            ServiceLoader<Driver> loadedDrivers = ServiceLoader.load(Driver.class);
            Iterator driversIterator = loadedDrivers.iterator();
            try{
                while(driversIterator.hasNext()) {
                    driversIterator.next();
                }
            } catch(Throwable t) {
            // Do nothing
            }
            return null;
        }
    });
    //通过Class.forName加载jdbc.drivers系统变量中指定的驱动类
    if (drivers != null && !drivers.equals("")) {
        String[] driversList = drivers.split(":");
        println("number of Drivers:" + driversList.length);
        for (String aDriver : driversList) {
            try {
                Class.forName(aDriver, true,
                        ClassLoader.getSystemClassLoader());
            } catch (Exception ex) {}
        }
    }
}
```

<font color='red'>就是在这里出现了打破委派机制的情况~</font>如果配置了jdbc.drivers变量，使用Class.forName加载驱动类时需要传入系统ClassLoader，因为DriverManager类是boot类加载器加载的，如果直接调用Class.forName(aDriver)，就会使用boot类加载器来加载驱动，显然是加载不到的。

因此，引入了SPI机制，

- 由于高版本jdk中的jdbc使用了SPI加载驱动实现类，所以不需要在前面显示的通过Class.forName("com.mysql.jdbc.Driver");来加载驱动类了。（写了也不会出错）
- 在以前的低版本中如果不写Class.forName("com.mysql.jdbc.Driver");，就需要设置jdbc.drivers系统变量，二者的用途是一样的，都是使用Class.forName方法来加载指定的驱动类；（驱动类一旦加载会执行其static代码，将驱动实例对象注册到DriverManager中）

4. 看上面的代码中的SPI获取驱动实现类的代码：`ServiceLoader.load()`
   - load()函数根据传入的接口类，遍历META-INF/services目录下的以该类命名的文件中的所有类，并实例化返回，同样当加载驱动类时，会调用驱动类的static代码，将其注册到jdbc的DriverManager中。
5. 然后，就是获取驱动连接
   - getConnection()函数：getConnection方法就是从registerdDrivers（是个list）中遍历所有的驱动实现类，驱动内部会根据url协议来判断是否该创建对应的数据库连接；如果所有驱动都无法解析url创建连接，则抛出异常；如果有满足的链接，就去验证一下。
   - isDriverAllowed()方法：该方法是通过“调用类的类加载器”（一般来说就是系统类加载器）再初始化一次驱动类，其目的是为了确保事先注册的驱动与当前的驱动是通过同一个类加载器加载的。

OK！到此，就把所有的驱动过程做完了，但是，似乎还不明确，到底SPI那个机制怎么做？并且，为什么JDBC一开始就是依靠系统级的类加载器（系统级就是应用级）。

其实很简单，

loadInitialDrivers()方法的源码，我们也知道了**JDBC中的DriverManager加载Driver的步骤顺序**依次是：

1. 通过SPI方式，读取 META-INF/services 下文件中的类名，使用线程上下文类加载器加载；
2. 通过System.getProperty(“jdbc.drivers”)获取设置，然后通过系统类加载器加载。

**SPI是怎么解决的：**

- 因为`Class.forName(DriverName, false, loader)`代码所在的类在java.util.ServiceLoader类中，而`ServiceLoader.class`又加载在BootrapLoader中，因此传给 forName 的 loader 必然不能是BootrapLoader(启动类加载器只能加载java核心类库)。
- 这时候只能使用线程上下文类加载器了：把自己加载不了的类加载到线程上下文类加载器中（通过Thread.currentThread()获取），而线程上下文类加载器默认是使用系统类加载器AppClassLoader。

> **总结：**
> 使用jdbc创建具体数据库驱动库的连接，主要思路是：加载驱动类，在驱动类的static代码中按照jdbc规范，将其实例注册到DriverManager中，在创建连接时遍历注册的list，先进行驱动实例的校验，然后根据url协议创建出连接。加载驱动类的顺序为：
>
> - 设置jdbc.drivers的系统变量；
> - 使用Class.forName指定驱动类；
> - 使用SPI加载驱动类；





## 0、一条Sql语句的执行流程

1. JDBC连接器负责跟Mysql客户端建立连接、获取权限、维持和管理连接。验证用户名密码
2. 获得连接后，判断这个语句是不是select语句，如果是，先去看一下查询缓存（大多数时候不建议使用缓存，因为只要一个表更新，这个表上的所有缓存数据就会被清空了。对于那些经常更新的表来说，缓存命中率很低。MYSQL8版本直接将查询缓存的整块功能删掉了。）
3. 缓存没有的话，mysql就要去解析这条语句了，解析成一个AST语法树。然后对这个sql中的关键字、表明进行语法检查。
4. 进入优化器。根据一些策略（数据量、能不能走索引、走哪个索引等），去生成一个执行计划（通过explain查看）
5. 执行器。对权限进行校验，如果有权限，就打开表继续执行。打开表的时候，执行器就会根据表的引擎定义，去使用这个引擎提供的接口，查询并过滤数据后，返回给客户端。
6. 如果是update、delete、insert这类操作，还会产生redo log日志（做持久化的）、bin log（归档日志，是逻辑日志，记录的是这个语句的原始逻辑，比如给“id =1 的一行的某个字段+2”。）。

**以一个select语句为例，mysql的执行器怎么做？**

1. 先根据from，join这些关键字，来确定要查询几张表
   - 并且根据on条件去判断这几个表是按什么字段进行连接的。
2. 再根据where条件去过滤记录。
3. 再根据group by分组
4. having去过滤这些分组
5. 最后根据select去过滤查找哪些列（这个里面也有去重的操作distinct）
6. 最后执行order by排序、limit限制（这里有个优化，如果只需要几条语句，就最好加上，不然有时候会进行全表查询）。



## 1、MVCC原则

> MVCC来保证一致性的思想并不是只在mysql中使用。
>
> 我举一个偏一点的例子，**etcd（一种高可用的的K/V存储和服务发现系统）中follower和leader之间的数据一致性就是通过MVCC思想来控制的。**当然，它的具体实现和MySQL不一样，不是undo日志，read view这些组件。它是使用的treeindex，boltDB，committed日志等，具体可以参考本博客的微服务章节`四、ETCD-高可用的K/V存储和服务发现系统 - 2、[etcd整个写数据流程]`(https://blog.csdn.net/weixin_39540280/article/details/122007678)

MVCC是一种并发控制的方法，一般在数据库管理系统中，实现对数据库的并发访问，在编程语言中实现事务内存。

- 最早的数据库系统，只有读读之间可以并发，读写，写读，写写都要阻塞。
- 引入多版本并发控制之后，只有写写之间相互阻塞，其他三种操作都可以并行，这样大幅度提高了InnoDB的并发度。
- MVCC可以根据事务隔离级别去判断并发隔离
  - 读-写并发：会出现脏读、幻读、不可重复度的问题。
  - 写-写并发：数据更新丢失。（一般用悲观锁解决）

**原理：**

- 数据库记录中的**隐式字段（版本号）、Undo日志和read view**来实现的。
- MVCC是利用在每条数据后面加了隐藏的两列（创建版本号和删除版本号），每个事务在开始的时候都会有一个递增的版本号
- **MVCC手段只适用于Msyql隔离级别中的读已提交（Read committed）和可重复读（Repeatable Read）.**序列化的就交给锁完成。
- Read uncimmitted由于存在脏读，即能读到未提交事务的数据行，所以不适用MVCC.

### 1.1 MVCC在RC和RR的实践

在读已提交（Read committed）这种隔离级别：

比如，一个事务流程：

| 事务1                                | 事务2                                | 事务3                                | 事务4                                                 |
| ------------------------------------ | ------------------------------------ | ------------------------------------ | ----------------------------------------------------- |
| update table set name='A' where id=1 |                                      |                                      |                                                       |
|                                      | update table set name='B' where id=1 |                                      |                                                       |
|                                      | commit                               | update table set name='C' where id=1 |                                                       |
| update table set name='A' where id=1 |                                      |                                      | select name from table where id=1 **readView:[1,3]3** |
|                                      |                                      | commit                               |                                                       |
| commit                               |                                      |                                      | select name from table where id=1 **readView:[1]3**   |

对于上面事务ABCD的流程的undolog日志为：

| record_header | id：1 | name:D | trx_id:1（事务A第二次update） | roll_pointer |
| ------------- | ----- | ------ | ----------------------------- | ------------ |
| record_header | id：1 | name:C | trx_id:3（事务C第一次update） | roll_pointer |
| record_header | id：1 | name:B | trx_id:2（事务B第一次update） | roll_pointer |
| record_header | id：1 | name:A | trx_id:1（事务A第一次update） | roll_pointer |

对于`select name from table where id=1 **readView:[1,3]3`**：去undolog中判断，trx_id和readView中的大小比较<1>从最新记录开始找:：

- 如果当前记录:事务id<未提交事务的最小id,则**可读**
- 如果当前记录:最小id<=事务id<=事务的最大id，则进一步判断事务id是否在未提交事务id的数组中，若在（说明是未提交的事务）则**不可读**(只有自己可读)
- 如果当前记录:事务id> 事务的最大id,，则**不可读**

<font color='red'>对于可重复读的readView，和读已提交的差别就是：</font>

- 读已提交每次都会返回新的readView
- 可重复读的readView则只用第一次记录的readView，后面的都是用第一次的。



## 2、索引

索引就是一种“排好序的数据结构”，可以帮助快速查找数据。

> 如果没有索引，数据库是怎么查询的？ 全表扫描。如果索引可以减少全表扫描，那么索引有效。如果索引导致了比全表扫描更糟糕的结果，那么还不如全表扫描。

**聚簇索引和非聚簇索引：**

- 聚簇索引：将数据存储与索引放到了一块，找到索引也就找到了数据
- 非聚簇索引：将数据存储于索引分开结构，索引结构的叶子节点指向了数据的对应行。即，非聚簇索引叶子节点存储的不再是行的物理位置，而是主键值

**优化：**

- 注意不能全部加上索引，虽然它查询使用优化隐藏器提高了性能，但是也会相应地占用物理空间，从而导致降低增删改的速度，因为操作数据的同时也要“维护索引文件”。

**B+树：**

![在这里插入图片描述](https://img-blog.csdnimg.cn/20200318160522840.png?x-oss-process=image/watermark,type_ZmFuZ3poZW5naGVpdGk,shadow_10,text_aHR0cHM6Ly9ibG9nLmNzZG4ubmV0L3FxXzQxOTk5NDU1,size_16,color_FFFFFF,t_70)

- InnoDB 一棵 B+ 树有1-3层，约 2 千万行数据。
- InnoDB 存储引擎的最小存储单元是页，页可以用于存放数据也可以用于存放键值 + 指针，在 B+ 树中叶子节点存放数据，非叶子节点存放键值 + 指针。
- 索引组织表通过非叶子节点的二分查找法以及指针确定数据在哪个页中，进而在去数据页中查找到需要的数据；
- 单个叶子节点（页）中的记录数 =16K/1K=16。（这里假设一行记录的数据大小为 1k，实际上现在很多互联网业务数据记录大小通常就是 1K 左右）。非叶子节点能存放多少指针:主键 ID 为 bigint 类型，长度为 8 字节，而指针大小在 InnoDB 源码中设置为 6 字节，这样一共 14 字节,即 16384/14=1170。
  - 高度为 2 的 B+ 树，能存放 1170*16=18720 条这样的数据记录。（根节点指针数 * 单个叶子节点记录行数）
  - 高度为 3 的 B+ 树可以存放： 1170\*1170*16=21902400 条这样的记录。（千万级别）
  - 一页能存储的数据量为：16kb / (1KB+8byte+8byte) ≈ 16； 高度为 3 的 B 树能存储 16 x16 x16 = 4096 条数据

### 2.1 常用索引

普通索引（INDEX）、主键索引（PRIMARY）、唯一索引（UNIQUE）、全文索引（FULLTEXT）。（还有一种说法叫联合索引（覆盖索引），但是这个不是这个分类属性）

```
举个例子来说，这个系统有一个会员表
有下列字段：
会员编号 INT
会员姓名 VARCHAR(10)
会员身份证号码 VARCHAR(18)
会员电话 VARCHAR(10)
会员住址 VARCHAR(50)
会员备注信息 TEXT

那么这个 会员编号，作为主键，使用 PRIMARY
会员姓名 如果要建索引的话，那么就是普通的 INDEX
会员身份证号码 如果要建索引的话，那么可以选择 UNIQUE （唯一的，不允许重复）
会员备注信息 ， 如果需要建索引的话，可以选择 FULLTEXT，全文搜索。
二级索引只存储了主键值，相比于聚簇索引，占用的空间要少，意味着innodb的二级索引可以利用这些额外的主键列来做覆盖索引
```

#### 2.1.1 常见索引之间的差别？

> 这个只要是落到了索引这个坑里面，肯定就会追问这个问题！
>
> 一般可以对着“聚簇索引和非聚簇索引有什么区别”这种问题来回答。

- 普通索引（INDEX）：

  - ```sql
    ALTER TABLE `table_name` ADD INDEX index_name ( `column` ) 
    ```

  - 最基本的索引，没有任何限制；二级索引，为的就是指向主键索引

- 主键索引（PRIMARY）：

  - ```sql
    ALTER TABLE `table_name` ADD PRIMARY KEY ( `column` ) 
    ```

  - 一种特殊的唯一索引，不允许有空值。

- 唯一索引（UNIQUE）：

  - ```sql
    ALTER TABLE `table_name` ADD UNIQUE (`column`)
    ```

  - 与"普通索引"类似，不同的就是：索引列的值必须唯一，但允许有空值。

- 全文索引（FULLTEXT）：

  - ```sql
    ALTER TABLE `table_name` ADD FULLTEXT ( `column` )
    ```

  - **仅可用于 MyISAM 表，** 用于在一篇文章中，检索文本信息的, 针对较大的数据，生成全文索引很耗时好空间。 组合索引：为了更多的提高mysql效率可建立组合索引，遵循”最左前缀“原则。

- 组合索引（包含多列的索引，与单行索引相对）:

  ```sql
  ALTER TABLE `table_name` ADD INDEX index_name ( `column1`, `column2`, `column3` )
  ```

#### 2.1.2 普通索引和唯一索引在数据库操作的时候有什么差别？

- 在查询的时候，普通索引和唯一索引没什么区别（因为每次读取的时候内存是一次性读取整个表）
- 但是在**更新、删除、新增**的时候，**普通索引就比唯一索引快了**。因为唯一索引他在更新时要检查唯一性，所以需要将数据页先加载到内存中才能判断，直接操作内存；而普通索引会先检查内存中有没有信息，没有就把更新操作记录到“change buffer”中，等下一次查询操作再进行”change buffer“中的操作（就是他会暂存一下）。这样——> 在写多读少的时候，I/O减少了；当然，如果写完马上就查询，就不需要change buffer了，因为总之是要读内存，还要额外维护change buffer。

### 2.2 索引匹配原则

- 最左匹配原则：对索引中关键字进行对比，一定是从左往右依次进行，且不可跳过
- 联合索引选择原则：
  - 经常用的列优先 【最左匹配原则】
  - 选择性(离散度)高的列优先【离散度高原则】
  - 宽度小的列优先【最少空间原则】
- 覆盖索引：如果查询列可通过索引节点中的关键字直接返回，则该索引称之为覆盖索引。覆盖索引可减少数据库IO，将随机IO变为顺序IO，可提高查询性能
  - 比如`create index name_score on user(name,score);select name,score from user where score = "100"；`在索引的子节点命中了 返回的列，就不需要再继续往B+Tree的底部叶子节点读取数据了，减少了IO操作，提高了查询速度（这里score索引上，会直接命中到name；我们也不需要别的信息，那就直接返回）。 这就是为什么不建议select * from table 的原因，需要什么列就查什么列，可能会命中覆盖索引，这样就会极大的提高查询效率。
- 匹配列前缀原则：匹配列前缀可用到索引 like xxx%，like **%xxx%**、like **%xxx**大部分情况用不到索引(覆盖索引除外，加粗的用到全表索引)；

### 2.3 mysql索引失效的情况

- WHERE字句的查询条件里有不等于号(WHERE column!=…)，MYSQL将无法使用索引
  - 如果WHERE字句的查询条件里使用了函数(如：WHERE DAY(column)=…)，MYSQL将无法使用索引
- 在JOIN操作中(需要从多个数据表提取数据时)，MYSQL只有在主键和外键的数据类型相同时才能使用索引，否则即使建立了索引也不会使用
- 如果WHERE子句的查询条件里使用了比较操作符LIKE和REGEXP，MYSQL只有在搜索模板的第一个字符不是通配符的情况下才能使用索引。比如说，如果查询条件是LIKE 'abc%',MYSQL将使用索引；如果条件是LIKE '%abc'，MYSQL将不使用索引。
- 在ORDER BY操作中，MYSQL只有在排序条件**不是一个查询条件表达式的情况下才使用索引**。
- 如果某个数据列里包含着许多重复的值，就算为它建立了索引也不会有很好的效果。比如说，如果某个数据列里包含了净是些诸如“0/1”或“Y/N”等值，就没有必要为它创建一个索引。（最好是对那些唯一主键进行索引）
- 对于多列索引，不是使用的第一部分，则不会使用索引。查询条件不使用复合索引的首列，均会导致索引失效。（比如`SELECT * FROM `t_index` WHERE key2 = '2' AND key3 = '3';`没有用第一列）
- 如果条件中有or(并且其中有or的条件是不带索引的)，即使其中有条件带索引也不会使用(这也是为什么尽量少用or的原因)。注意：要想使用or，又想让索引生效，只能将or条件中的每个列都加上索引
- 如果列类型是字符串，那一定要在条件中将数据使用引号引用起来,否则不使用索引
- 如果mysql估计使用全表扫描要比使用索引快,则不使用索引

### 2.4  覆盖索引

- 就是select的数据列只用从索引中就能够取得，不必从数据表中读取，换句话说查询列要被所使用的索引覆盖。（避免了回表操作）
- MySQL只能使用B-Tree索引做覆盖索引

### 2.5 mysql为什么不能建很多索引？为什么重复值高的字段不能建索引？

我们在享受索引带来的方便同时，索引有时候也会让查询（特别是增删改）变得耗时。具体而言，有以下两个方面：

1. 空间上：
   - 索引本身是一种物理存在，他和数据放在一起；创建索引和维护索引都要耗费时间，而且数据量越多时间越长。
   - 虽然它查询使用优化隐藏器提高了性能，但是太多的索引也会相应地占用物理空间，从而导致降低增删改的速度，因为操作数据的同时也要“维护索引文件”。
2. 时间上：
   - 非聚集索引需要两次查找，先从非聚集索引中找到主键值，然后再去聚集索引中找到具体数据。如果非聚集索引重复率高(即一个同样的值有多个主键)，那么首先你会从索引中取一半主键值，然后根据主键值再去查询数据，增加了IO，所以特别耗时。
   - 如果数据频繁的更新或删加，就不宜建立索引。



## 3、 InnoDB和MyISAM的区别？

1. InnoDB支持事务，MyISAM不支持，对于InnoDB每一条SQL语言都默认封装成事务，自动提交，这样会影响速度，所以最好把多条SQL语言放在begin和commit之间，组成一个事务； 
2. InnoDB支持外键，而MyISAM不支持。对一个包含外键的InnoDB表转为MYISAM会失败； 
3. InnoDB是聚集索引，使用B+Tree作为索引结构，**数据文件（私有表空间文件：表名.ibd；存储表和视图的结构定义：表名.frm**）是和（主键）索引绑在一起的（表数据文件本身就是按B+Tree组织的一个索引结构），必须要有主键，通过主键索引效率很高。但是辅助索引需要两次查询，先查询到主键，然后再通过主键查询到数据。因此，主键不应该过大，因为主键太大，其他索引也都会很大。
4. InnoDB不保存表的具体行数，执行select count(*) from table时需要全表扫描。而MyISAM用一个变量保存了整个表的行数，执行上述语句时只需要读出该变量即可，速度很快（注意不能加有任何WHERE条件）；
5. Innodb不支持全文索引，而MyISAM支持全文索引，在涉及全文索引领域的查询效率上MyISAM速度更快高；PS：5.7以后的InnoDB支持全文索引了
6. MyISAM表格可以被压缩后进行查询操作
7. InnoDB支持表、行(默认)级锁，而MyISAM支持表级锁
8. InnoDB表必须有唯一索引（如主键）（用户没有指定的话会自己找/生产一个隐藏列Row_id来充当默认主键），而Myisam可以没有

## 4、为什么Mysql要用B+数，不用B数、二叉树、Hash表、红黑树？

其实问这个问题，我们要想，Mysql在哪里用到了这种数据结构——索引。那索引的目的就是快速获取数据。那最好就是排好序，并且查询快。

索引存储方式（<K,V>存储）：key是这个索引元素,比如34,77等,value是这个索引在磁盘上面的文件地址。索引也是落地到磁盘上面的，因为数据是落在磁盘上面的（索引就在他旁边，.idb文件中）,然后根据这个key对应的value去磁盘上面找到对应的值,然后输出。

- 常用的“InnoDB”引擎就是“B+树”。

所以，我们如果用如下几个数据结构，**有什么差别：**

1. 链表：查询太慢

2. 用hash值作为索引？

   - innodb存储引擎会监控对表上二级索引的查找，如果发现某二级索引被频繁访问，二级索引成为热数据，建立哈希索引可以带来速度的提升
   - 自适应hash索引只适合搜索等值的查询，而对于其他查找类型，如范围查找，是不能使用的；
   - Hash 索引不支持联合索引的最左侧原则（即联合索引的部分索引无法使用）：对于联合索引来说，Hash 索引在计算 Hash 值的时候是将索引键合并后再一起计算 Hash 值，所以不会针对每个索引单独计算 Hash 值。因此如果用到联合索引的一个或者几个索引时，联合索引无法被利用。
   - 主要其实就是hash不能用于排序分组：不支持ORDER BY 排序，因为 Hash 索引指向的数据是无序的，因此无法起到排序优化的作用，而 B+ 树索引数据是有序的，可以起到对该字段 ORDER BY 排序优化的作用。
   - 对于等值查询来说，通常 Hash 索引的效率更高，不过也存在一种情况，就是索引列的重复值如果很多，效率就会降低。（考虑limit优化）
   - **Mysql底层实现就是B+树索引加上“hash”索引**

3. B树不行吗？

   - B树不管叶子节点还是非叶子节点，都会保存数据（指的是索引加上真实数据），这样导致在非叶子节点中能保存的指针数量变少（有些资料也称为扇出），指针少的情况下要保存大量数据，只能增加树的高度，导致IO 操作变多，查询性能变低。（一页能存储的数据量为：16kb / (1KB+8byte+8byte) ≈ 16； 高度为 3 的 B 树能存储 16 x16 x16 = 4096 条数据）
   - “B+树”只在叶子结点存储数据，非叶子节点存索引。相比B树，可以存更多的节点，树高更矮，查询次数少，磁盘I/O少，性能更好

4. 跳表可以吗？

   - 面试官直接问我，什么数据结构可以做到O(logn)的范围查询？

     当时没有想到填表，想到了B+树，而且数据库索引确实也是用的B+树。那么问题来了，跳表和B+树在实现难度和性能上有什么区别，在数据量很大的情况下，表现性能如何，为什么redis选跳表？

   - b+树主要是用在外部存储上，为了减少磁盘IO次数。 跳表比较适合内存存储。 实际上，两者本质的设计思想是雷同的，性能差距还是要具体看应用场景。（redis为什么要用跳表，其实是因为它更容易实现，也不涉及到I/O，也就没有B+树减少I/O的核心。这也是为什么redis是单线程的，他不用考虑I/O，如果用了多线程，反而多了上下文切换的时间，得不偿失）

## 5、MySQL如何尽可能的避免死锁呢？

1）以固定的顺序访问表和行。即按顺序申请锁，这样就不会造成互相等待的场面。

2）大事务拆小。大事务更倾向于死锁，如果业务允许，将大事务拆小。

3）在同一个事务中，尽可能做到一次锁定所需要的所有资源，减少死锁概率。

4）降低隔离级别。如果业务允许，将隔离级别调低也是较好的选择，比如将隔离级别从RR调整为RC，可以避免掉很多因为gap锁造成的死锁。

5）为表添加合理的索引。如果不走索引将会为表的每一行记录添加上锁，死锁的概率大大增大。

## 6、什么是事务？

指作为单个逻辑工作单元执行的一系列操作，要么完全地执行，要么完全地不执行。
简单的说，事务就是并发控制的单位，是用户定义的一个操作序列。
而一个逻辑工作单元要成为事务，就必须满足ACID属性。
**事务运行的三种模式：**
1.自动提交事务
    每条单独的语句都是一个事务，每个语句都隐含一个commit
2.显式事务
    以begin transaction 开始，以commit 或 rollback 结束。
3.隐性事务
    在前一个事务完成时，新事务隐式启动，但每个事务仍以commit或rollback显示结束


### 6.1、ACID是靠什么来保证的？

- **原子性** 由 undolog 日志来保证，它记录了需要回滚的日志信息，事务回滚时撤销已经执行成功的sql。
- **一致性** 是由其他三大特性保证，程序代码要保证业务上的一致性。
- **隔离性** 是由 MVCC 来保证。
- **持久性** 由 redolog 来保证，mysql 修改数据的时候会在 redolog 中记录一份日志数据，就算数据没有保存成功，只要日志保存成功了，数据仍然不会丢失。

## 7、事务隔离级别：脏读、不可重复读、可重复读、幻读

- **脏读（读取未提交数据）：**A事务读取B事务尚未提交的数据，此时如果B事务发生错误并执行回滚操作，那么A事务读取到的数据就是脏数据。
- **不可重复读（前后多次读取，数据内容不一致）：**事务A在执行读取操作，由整个事务A比较大，前后读取同一条数据需要经历很长的时间 。而在事务A第一次读取数据，比如此时读取了小明的年龄为20岁，事务B执行更改操作，将小明的年龄更改为30岁，此时事务A第二次读取到小明的年龄时，发现其年龄是30岁，和之前的数据不一样了。
- **幻读（前后多次读取，数据总量不一致）：**事务A在执行读取操作，需要两次统计数据的总量，前一次查询数据总量后，此时事务B执行了**新增**数据的操作并提交后，这个时候事务A读取的数据总量和之前统计的不一样，就像产生了幻觉一样，平白无故的多了几条数据，成为幻读。（幻读是有新增数据，针对insert和delete操作；不可重复读是因为修改了数据，针对update操作）

## 8、数据库的隔离级别

![img](https://ask.qcloudimg.com/http-save/yehe-1261315/j0gc9gmx2e.png?imageView2/2/w/1620)

1. Read uncommited读未提交：可以读取未提交的数据；就是脏读问题
2. Read committed 读提交：要求只能读取已经提交的数据；读不到正在改变的数据，避免了脏读，但是可能会造成不可重复读。（现象：对于同一个数据读到了不一样的数值，比如B先进来，读到了一个数据，A上来修改了数据并提交，B再读，发现已经被修改了，就很疑惑？B在重复读的时候，发生了数据不一致！并且，他也会有幻读的情况发生）
3. Repeatable read 重复读：一旦A事务开始读数据，其他事务（B)不能update修改数据，这样A就可以保证重复读取。
   1. 但是，不可重复读对应的是修改，即UPDATE操作。但是可能还会有**幻读**问题。因为幻读问题对应的是插入INSERT操作，而不是UPDATE操作。
   2. 比如，A先查看一整个表，B在其中新添了几行，并提交。当A提交事务的时候，发现多了几行，对A而言，幻觉出现。
   3. 所以，不可重复读查询的都是同一个数据项（行锁），而幻读针对的是一批数据整体（比如数据的个数）。
4. Serializable 序列化（串行化）：在该级别下，事务顺序执行，不仅可以避免脏读、不可重复读，还避免了幻读。

## 9、一次性解决数据库锁的问题（行锁、表锁）

### 9.1 简单分类

- 表锁是指对一整张表加锁，一般是由 MySQL Server 实现，在执行DDL语句时使用。（数据库模式定义：create、alter、drop、truncate（也是删除））
- **行锁**则是存储引擎实现，不同的引擎实现的不同。
  - `update user set age = 10 where id = 49;update user set age = 10 where name = 'Tom';`使用主键索引需要加一把锁，使用二级索引需要在二级索引和主键索引上各加一把锁。
  - `update user set age = 10 where id > 49;`更新操作涉及多个行，在索引未失效的情况下，InnoDB 引擎会将第一条记录返回并加锁，接着 MySQL Server 发起更新改行记录的 UPDATE 请求，更新这条记录。一条记录操作完成，再读取下一条记录，直至没有匹配的记录为止。**如果索引失效，则会退化成表锁！**（比如`select user set age where id <> 3`），表锁会再进化成行锁（释放掉不满足条件的行锁）
  - `update user set age = 10 where id = -1;`这种情况，id<1,主键不存在，那就直接不锁！

### 9.2 [行锁的具体形式](https://zhuanlan.zhihu.com/p/149228460)

#### 9.2.1 行锁——共享锁（lock in share mode）和排它锁（for update）

- 共享锁（Share locks，简称 S 锁），又称读锁，所有的事务都可以读取，但是不能修改，并且可同时有多个事务对记录加读锁。（可重入）对于普通SELECT语句，InnoDB会自动给涉及数据集加共享锁（S）；
- 排他锁（Exclusive locks，简称 X 锁），或独占锁，又称写锁，只有拥有该锁的事务可以读取和修改，其他事务都不可以读取和修改，并且同一时间只能有一个事务加写锁。（互斥）对于UPDATE、DELETE和INSERT语句，InnoDB会自动给涉及的数据集加排他锁（Ｘ）；
- **实现：**InnoDB行锁是通过索引上的**索引项**来实现的

#### 9.2.2 表锁（意向锁）——意向共享锁（IS）、意向排他锁（IX）、自增锁

为了允许行锁和表锁共存（避免冲突），当你要加表锁时，势必要先遍历该表的所有记录，判断是否加有排他锁。这种遍历检查的方式显然是一种低效的方式，MySQL 引入了意向锁，来检测表锁和行锁的冲突。意向锁是InnoDB自动加的，不需用户干预。

- 意向共享锁（IS）：事务打算给数据行共享锁，事务在给一个数据行加共享锁前必须先取得该表的IS锁。
- 意向排他锁（IX）：事务打算给数据行加排他锁，事务在给一个数据行加排他锁前必须先取得该表的IX锁。
- 当插入表中有自增列时，数据库需要自动生成自增值，它会先为该表加 AUTOINC 表锁，阻塞其他事务的插入操作，这样保证生成的自增值肯定是唯一的。**AI 锁只和意向锁兼容；**

#### 9.2.3 间隙锁

- 当我们用范围条件而不是相等条件检索数据，并请求共享或排他锁时，InnoDB（可重复读、串行化级别下才有效）会给符合条件的已有数据的索引项加锁；
  - 对于**键值在条件范围内但并不存在的记录，**叫做“间隙(GAP)”，InnoDB也会对这个“间隙”加锁。
  - 使用间隙锁可以防止其他事务在这个范围内插入或修改记录，保证两次读取这个范围内的记录不会变，从而不会出现幻读现象。（防止插入，出现幻读；所以如果 id = 49 这条记录不存在，在 RC 隔离级别不会加任何锁，在 RR 隔离级别会在 id = 49 前后两个索引之间加上间隙锁。）

#### 9.2.4 Next-key 锁：行锁（排它锁）+间隙锁

- 指加在某条记录以及这条记录前面间隙上的锁。
- 在默认情况下，mysql的事务隔离级别是可重复读，这时默认采用next-key locks。
- 假设一个索引包含 15、18、20 ，30，49，50 这几个值，可能的 Next-key 锁如下：`(-∞, 15]，(15, 18]，(18, 20]，(20, 30]，(30, 49]，(49, 50]，(50, +∞)`
  - 如果 id 不是主键，而是**二级索引，且不是唯一索引**，那么这个 SQL 在 RR 隔离级别下就会加如下的 Next-key 锁 (30, 49](49, 50)（一个next-key锁+间隙锁）。此时如果插入一条 id = 31 的记录将会阻塞住。之所以要把 id = 49 前后的间隙都锁住，仍然是为了解决幻读问题，因为 id 是非唯一索引，所以 id = 49 可能会有多条记录，为了防止再插入一条 id = 49 的记录。

##### [加锁规则](https://www.jianshu.com/p/d1aba64b5c03)

加锁规则里面，包含了两个“原则”、两个“优化”和一个“bug”。

1. 原则 1：加锁的基本单位是 next-key lock。希望你还记得，next-key lock 是前开后闭区间。
2. 原则 2：查找过程中访问到的对象才会加锁。（绕过覆盖索引的优化）
3. 优化 1：索引上的等值查询，给唯一索引加锁的时候，next-key lock 退化为行锁。
4. 优化 2：索引上的等值查询，向右遍历时且最后一个值不满足等值条件的时候，next-key lock 退化为间隙锁。
5. 一个 bug：唯一索引上的范围查询会访问到不满足条件的第一个值为止。

### 9.3 数据库死锁场景？怎么避免死锁？

发生死锁后，InnoDB一般都能自动检测到，并使一个事务释放锁并退回，另一个事务获得锁，继续完成事务。

- 直接进入等待，直到超时。
- （推荐）主动死锁检测，发现死锁后，主动回滚死锁链条中的某一个事务，让其他事务得以继续执行。
  - 如果出现很多事务都要更新同一行的场景（热点行），每个新来的被堵住的线程，都要判断会不会由于自己的加入导致了死锁，这是一个时间复杂度是O(n)的操作。
  - 对于相同行的更新，在进入引擎之前排队。这样在InnoDB内部就不会有大量的死锁检测工作了。

怎么避免死锁：

- 尽量使用较低的隔离级别（隔离级别低）
- 精心设计索引，并尽量使用索引访问数据，使加锁更精确，从而减少锁冲突的机会。（索引设计）
- 选择合理的事务大小，小事务发生锁冲突的几率也更小。（小事务）
- 给记录集显示加锁时，最好一次性请求足够级别的锁。比如要修改数据的话，最好直接申请排他锁，而不是先申请共享锁，修改时再请求排他锁，这样容易产生死锁。（请求锁要足量）
- 不同的程序访问一组表时，应尽量约定以相同的顺序访问各表，对一个表而言，尽可能以固定的顺序存取表中的行。这样可以大减少死锁的机会。（顺序存取表）
- 多用等值查询。

### 9.4 各种锁的示例

```
CREATE TABLE `t` (
  `id` int(11) NOT NULL,
  `c` int(11) DEFAULT NULL,
  `d` int(11) DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `c` (`c`)
) ENGINE=InnoDB;
 
insert into t values(0,0,0),(5,5,5),
(10,10,10),(15,15,15),(20,20,20),(25,25,25);
```

执行以下语句：select * from t where d=5 for update，触发了什么锁？

- 可重复读隔离级别下：由于d没有索引，因此实际上是加入表锁，InnoDB的表锁形式为next-key lock（主键行锁+间隙锁），即会给数据库中的记录都加上行锁，还同时加上n+个间隙锁(n为已有记录数)。确保无法再插入新的记录。在一行行扫描的过程中，不仅将给行加上了行锁，还给行两边的空隙也加上间隙锁。间隙锁和行锁合称next-key lock，每个next-key lock是前开后闭区间。我们的表t初始化以后，如果用select * from t for update要把整个表所有记录锁起来，就形成了7个next-key lock，分别是 (-∞,0]、(0,5]、(5,10]、(10,15]、(15,20]、(20, 25]、(25, +supremum]。
- 读已提交下：先给所有记录都加锁。语句执行完后，只有符合d=5的行会有行锁

## 10、自适应 Hash 索引、插入缓冲（Insert Buffer）和二次写(Double Write)

### 10.1 什么情况下才会使用自适应Hash索引呢？

- 如果某个数据经常会访问到，当满足一定条件的时候，就会将这个数据页的地址存放到Hash表中。这样下次查询的时候，就可以直接找到这个页面的所在位置。

- InnoDB本身不支持Hash，但是提供自适应Hash索引，不需要用户来操作，而是存储引擎自动完成的。
- 自适应Hash也是InnoDB三大关键特性之一，另外两个分别是**插入缓冲（Insert Buffer）和二次写(Double Write)**。

### 10.2 插入缓冲（Insert Buffer）

- 当我们按照主键a进行插入的时候，对于非聚集索引，也就是常说的二级索引b,它的插入不是顺序的，插入性能必然会下降。
- 针对这种情况，设计了Insert Buffer，对于非聚集索引的插入或者更新操作，不是每一次插入到索引页中，而是先判断插入的非聚集索引页是否在缓冲池中，如果在，则直接插入，如果不在，则先放入一个insert buffer中，告诉数据库这个非聚集的索引已经插入到了叶子节点，实际上并没有插入，只是存放在另外一个位置，然后再以一定的频率和情况进行Insert buffer和辅助索引叶子节点合并操作。这种时候，经常能将多条记录的插入合并到一个操作中，这样就大大提高了非聚集索引离散插入的性能。
- insert buffer需要满足两个条件才能被使用，第一，索引是辅助索引，也就是二级索引，第二，索引不是唯一的（之所以不支持唯一索引，是因为如果辅助索引是唯一索引，那么在插入时需要校验唯一性，校验唯一性的时候就会发生离散读取，从而又增加了开销，那么insert buffer得不偿失）。当满足上述两个条件的时候，就可以使用insert buffer，从而提高数据库的插入操作性能。
- 这里需要注意，如果在程序进行了大量操作的时候发生了MySQL数据库的宕机，那么肯定有大量的insert buffer没有合并到实际的非聚集索引中去，恢复可能会造成很长的时间。

最新的MySQL5.7已经支持**change buffer，**事实上，它在innodb 1.0.x版本已经引入，这个change buffer 可以理解为insert buffer的升级，也就是对常见的DML语言都可以进行缓冲，包含insert delete以及update，对应的分别是insert buffer，delete buffer以及purge buffer。

### 10.3 二次写(Double Write)

- MySQL的数据页默认是16K，而文件系统的数据页是4K，IO操作是按页为单位就行读写的。这就可能出现数据库对一个16k的数据页修改后，操作系统开始进行写磁盘，但是在这个过程中数据库宕机导致没有完全将16K数据页写到磁盘上。数据库重启后，校验数据页，发现有数据页不完整，就起不来了（redo只能加上旧、校检完整的数据页恢复一个脏块，不能修复坏掉的数据页，所以这个数据就丢失了）。
- 针对这种情况，增加一个doubleWrite buffer，作为缓存。doublewrite由两部分组成，一部分为内存中的doublewrite buffer，其大小为2MB，另一部分是磁盘上共享表空间(ibdata x)中连续的128个页，即2个区(extent)，大小也是2M。

![在这里插入图片描述](https://img-blog.csdnimg.cn/f8a59b53cb2b4b77aadbb77818b78cf9.png?x-oss-process=image/watermark,type_d3F5LXplbmhlaQ,shadow_50,text_Q1NETiBAc3RvbmVfbWlu,size_20,color_FFFFFF,t_70,g_se,x_16)

流程：

1、当一系列机制触发数据缓冲池中的脏页刷新时，并不直接写入磁盘数据文件中，而是先拷贝至内存中的doublewrite buffer中；

2、接着从两次写缓冲区分两次写入磁盘共享表空间中(连续存储，顺序写，性能很高)，每次写1MB；

3、待第二步完成后，再将doublewrite buffer中的脏页数据写入实际的各个表空间文件(离散写)；(脏页数据固化后，即进行标记对应doublewrite数据可覆盖)

4、doublewrite的崩溃恢复

如果操作系统在将页写入磁盘的过程中发生崩溃，在恢复过程中，innodb存储引擎可以从共享表空间的doublewrite中找到该页的一个最近的副本，将其复制到表空间文件，再应用redo log，就完成了恢复过程。

因为有副本所以也不担心表空间中数据页是否损坏。

**Q：为什么log write不需要doublewrite的支持？**

A：因为redolog写入的单位就是512字节，也就是磁盘IO的最小单位，所以无所谓数据损坏。
**Q：doublewrite的副作用？**

1. double write带来的写负载（2遍写——本质上就是冗余）
2. double write是一个buffer, 但其实它是开在物理文件上的一个buffer, 其实也就是file, 所以它会导致系统有更多的fsync操作, 而硬盘的fsync性能是很慢的, 所以它会降低mysql的整体性能。

## 11、怎么解决Hash冲突？

> 先做总结：4种方法

1. [开放寻址](https://blog.csdn.net/m0_37531231/article/details/82797919)（线性探索算法、随机探测）（比如，ThreadLocal）；
   - 当冲突发生时，使用某种探测技术在散列表中形成一个探测序列。沿此序列逐个单元地查找，直到找到给定的关键字，或者 碰到一个开放的地址（即该地址单元为空）为止；
   - 比如，我们的关键字集合为{12,67,56,16,25,37,22,29,15,47,48,34},表长为12。 我们用散列函数f(key) = key mod l2。当计算前S个数{12,67,56,16,25}时，都是没有冲突的散列地址，直接存入；计算key = 37时，发现f(37) = 1，此时就与25所在的位置冲突。 于是我们应用上面的公式f(37) = (f(37)+1) mod 12 = 2。于是将37存入下标为2的位置；后面的依次进行。
   - 所以，**好的散列函数=计算简单+分布均匀（计算获得的散列地址分布均匀）**
2. 再哈希（例布隆过滤器，redis）；
   - 这种方式要求有多个不同的Hash函数，当发生冲突时，使用第二个，第三个，….，等哈希函数 计算地址，直到无冲突。虽然不易发生聚集，但是增加了计算时间。
3. 公共溢出区；（没什么用）
   - 将哈希表分为基本表和溢出表两部分，凡是和基本表发生冲突的元素，一律填入溢出表。（类似于链地址）
4. 链地址：hash冲突使用链表（concurrentHashMap，当链表长度大于8时，单向链表转换成红黑树）

### 0. String原理

>redis的String的实现是用的自己的结构体SDS（Simple Dynamic String）

1. 为什么需要？场景驱动！
   - 在有的场景中，需要存入`Hello \0 world! \0`；在C语言中，\0是结束标识，只能转义一下；为了存一些特殊字符，sds就做了一些处理：比如，存一个length字段，标识这个字符串什么时候结束。这种存储方式其实叫做”安全的二进制存储“
   - length字段也能有效避免反复遍历字符串来获得其长度
   - 字符串的扩缩容问题：在C里面（其实Java也是这样。python不是），我在创建字符串的时候，字符串的长度是但创建其实就固定了。
     - 扩容：如果还要在这个字符串后面追加数据（比如Java里面的字符串相加操作），那我们就需要重新申请一个char数组空间，然后把新的字符串拷贝进去，再把原来的空间释放掉；**而在redis中**，sds就会预先多申请一部分空间预留。比如，我创建了一个长度为50的字符串，其实是申请了长度为100的字符空间，这样在新加数据的时候，就可以直接加入；
     - 缩容：在C语言中，一个字符串要变短的话，要立刻进行内存拷贝；在redis中，sds就可以直接修改length字段

2. 大概结构（从上面的分析，我们也能推出来，一个sds基本是什么样子，但实际上还是有点差别）：

```java
struct sds{
	int len;     //记录char数组实际使用了多少个字节
	int alloc;		//分配空间，记录char数组的实际长度
	char buf[];		//存储字符串长度
}
```

3. 源码分析（目录在redis的`src/sds.h`中）

```java
struct sdshdr {
    //记录buf数组中已使用字节的数量
    //等于SDS所保存字符串的长度
    unsigned int len;

    //记录buf数组中未使用字节的数量
    unsigned int free;

    //char数组，用于保存字符串
    char buf[];	//buf 尾部自动追加一个'\0'字符并不会计算在 SDS 的len中
};
```

#### 0.1 String数据结构优化（三种级别，五种类型）

> 我们似乎得到了一个结构不错的 SDS ，但是我们能否继续进行优化呢？因为，在 **Redis3.x** 版本中不同长度的字符串占用的头部是相同的，如果某一字符串很短但是头部却占用了更多的空间，这未免太浪费了。

将 SDS 分为**三种级别**的字符串：

- 短字符串(长度小于32)，len和free的长度用1字节即可；
- 长字符串，len和free的长度用2字节或者4字节；
- 超长字符串，len和free的长度用8字节。

> 共有**五种类型**的SDS（长度小于1字节、1字节、2字节、4字节、8字节），那怎么标识这些类型呢？这肯定要再增加一个字段：
>
> - 在 SDS 中新增一个 type 字段来标识类型，但是没必要使用一个 4 字节的int类型去做！可以使用 1 字节的char类型，通过位运算（3位即可标识2^3种类型）来获取类型。
>
> - 具体而言：**低三位存储类型，高5位存储长度**（小端存储，所以低位在左，高位在右）
>   - <font color='red'>高5位是为了利用好这个char类型的type字段，最多能标识的长度为32；所以短字符串，我们就省略了length字段，所以这也是为什么规定短字符串的长度必定小于32~~</font>

##### 1、字串长度（上面说的5种类型）-512MB

> 在 Reids3.x 版本中len是使用int修饰的，这就会导致 buf 最长就是2^31 -1 = 2147483647，无形中限制了字符串的最大长度。

以sdshdr8和sdshdr16举例：

```c
struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len; /* 已使用，uint8_t是8位 */
    uint8_t alloc; /* 总长度，用1字节存储 */
    unsigned char flags; /* 低3位存储类型, 高5位预留 */
    char buf[];
};
struct __attribute__ ((__packed__)) sdshdr16 {
    uint16_t len; /* 已使用，uint16_t是16位，它表示长度最大为2^16-1 */
    uint16_t alloc; /* 总长度，用2字节存储 */
    unsigned char flags; /* 低3位存储类型, 高5位预留 */
    char buf[];
};
```

- 我们看到 len 和 alloc两个字段的长度区别。如果字符串`abc`同时用sdshdr8和sdshdr16存储的话，仅仅是 len 和 alloc两个字段，sdshdr8就比sdshdr16节省了两个字节；当字符串数量级起来的时候，比如10亿个字符串，一个字符串节省一个字节，那就能省出将近1G的空间！
- redis在创建字符串的时候，会根据字符串的长度来决定使用哪种类型的sds



##### 2、**对齐填充**（redis不对齐）

在 Redis6.x 的源码中 SDS 的结构体为`struct __attribute__ ((__packed__))`与struct有较大的差别，这其实和我们熟知的对齐填充有关。

举个例子：

考虑如下结构体：

```c
typedef struct{
     char  c1;		//1字节
     short s; 		//2字节
     char  c2;		//1字节
     int   i;		//4字节
} s;
```

若此结构体中的成员都是紧凑排列的，假设c1的起始地址为0，则s的地址为1，c2的地址为3，i的地址为4。下面用代码论证一下我们的假设：

```c
#include <stdio.h>

typedef struct
{
    char c1;
    short s;
    char c2;
    int i;
} s;

int main()
{
    s a;
    printf("c1 -> %d, s -> %d, c2 -> %d, i -> %d\n",
           (unsigned int)(void *)&a.c1 - (unsigned int)(void *)&a,
           (unsigned int)(void *)&a.s - (unsigned int)(void *)&a,
           (unsigned int)(void *)&a.c2 - (unsigned int)(void *)&a,
           (unsigned int)(void *)&a.i - (unsigned int)(void *)&a);
    return 0;
}
// 结果为：c1 -> 0, s -> 2, c2 -> 4, i -> 8
//c1后面填了1个空；c2后面填了3个空
```

这就是对齐填充搞的鬼，

- 现代计算机中，内存空间按照字节划分，理论上可以从任何起始地址访问任意类型的变量。

- 但实际中在访问特定类型变量时经常在特定的内存地址访问，这就需要各种类型数据按照一定的规则在空间上排列，而不是顺序一个接一个地存放，这就是对齐。（一些平台对某些特定类型的数据只能从某些特定地址开始存取）
- 比如有些平台每次读都是从偶地址开始（每次读4字节），如果一个int型（假设为 32位）存放在偶地址开始的地方，那么一个读周期就可以读出，而如果存放在奇地址开始的地方，就可能会需要2个读周期，并对两次读出的结果的高低字节进行拼凑才能得到该int数据，导致在读取效率上下降很多。
  - 更具体地，我每次读4字节。如果我要读int类型，我肯定希望每次读的时候都是在int的起始位置，这样我每次都可以正好完成int的读取；但是如果我的int首地址，放到了一个奇数的地址上，那我就要读两次内存，一次读前面部分，另一次读后面部分；
- 一般我们是不需要关注内存对齐，机器会自动对齐

将上述示例代码的结构体更改如下(取消对齐)，再次执行，可以发现取消对齐后和我们的假设就一致了。

```c
typedef struct __attribute__ ((__packed__))
{
    char c1;
    short s;
    char c2;
    int i;
} s;
// 再次执行：c1 -> 0, s -> 1, c2 -> 3, i -> 4
```

综上所述我们知道了对齐填充可以提高 CPU 的数据读取效率，作为 IO 频繁的 **Redis 为什么选择不对齐呢？**

> 思考这个问题，就相当于把redis结构的设计思路完全串起来了~

> 有个细节各位需要知道，即 SDS 的指针并不是指向 SDS 的起始位置（len位置），而是直接指向buf[]，使得 SDS 可以直接使用 C 语言string.h库中的某些函数，做到了兼容，十分nice~。

###### 为什么redis要用flags字段来区分类型？

- 我们看到sdshdrXX中都有一个flag字段，明明我们已经用sdshdrXX结构体来区分5个sds类型吗？直接通过结构体的类型来区分不就好了？这是因为redis在这5个sds结构体的上层又封了一层。
- 在sds.h的源码，我们可以看到一行`typedef char *sds`，typedef 是C语言中用来定义别名的，这里给char指针起了个别名，叫sds。 （所以，大名鼎鼎的sds只是个char指针！）那这个**sds指向的位置，就是sdshdrXX中的一种**！比如，用sdshdr8来存一个字符串，sds指向的就是sdshdr8中buf[]数组中的起始位置。
- 那这个时候我们就直接往后读吗？不是的！我们要知道 len和alloc吧！那意味着sds需要往前读取。所以，我们先让sds指针往前移动一个字节，拿到flags值；通过读这个flags的低三位值，我们就能知道当前sds实例是五种类型的哪一种，那这个就标识了我们用的是哪个sdshdrXX了！
- 接下来，我们再往前读，找到 len和alloc；根据这两个值，我们就可以往后读buf[]数组了

> 所以，可以感受到，sds和5个sdshdrXX之间的关系，很像是Java中的接口和实现类的关系。有了flags的效果，我们就好回答为什么redis不对齐了↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓（对齐了就找不到flags了）



ok，言归正传！

- 如果不进行对齐填充，那么在获取当前 SDS 的类型时则只需要后退一步即可——`flagsPointer = ((unsigned char*)s)-1`；
- 相反，若进行对齐填充，由于 Padding（空白） 的存在，我们在不同的系统中不知道退多少才能获得flags，并且我们也不能将 sds 的指针指向flags，进一步地，我们也无法获得 len 和 alloc 的位置。这样就无法兼容 C 语言的函数了，也不知道前进多少才能得到 buf[]。

#### 0.2 SDS的优势

> 前面我们说了，为什么sds被引入了！O(1)时间复杂度获取字符串长度。
>
> 为了获取一个字符串的长度程序必须遍历这个字符串，直至遇到'0'为止，整个操作的时间复杂度为O(N)。而我们使用SDS封装字符串则直接获取len属性值即可，时间复杂度为O(1)。
>
> length字段太好了，但是这只是它的基本效果。下面简单介绍一下它的其他功能。感兴趣的可以继续，不过面试到这里基本就已经很深入了。

1、O(1)时间复杂度获取字符串长度

2、**杜绝缓冲区溢出**

- C字符串：由于C自身不记录字符串的长度带来一个问题是容易造成缓冲区溢出(buffer overflow)。在`/strcat`函数中，可以将一个字符串拼接到另外一个字符串的末尾。`char *strcat(char *dest,const char *src)`；理想状态下，用户在使用这个函数时，假定C为dest分配了足够多的内存，可以容纳src字符串中的所有内容，而一旦这个假定不成立，就会产生缓冲区溢出

- SDS 的自动扩容机制完全杜绝了发生缓冲区溢出的可能性： 当SDS API需要对SDS进行修改时，API会先检查 SDS 的空间是否满足修改所需的要求，如果不满足，API会自动将SDS的空间扩展至执行修改所需的大小，然后才执行实际的修改操作（自动扩容机制——sdsMakeRoomFor方法）

3、**内存重分配次数优化**——减少修改字符串时带来的内存重分配次数

- C字符串不记录自身长度，每次增长或者缩短字符串长度时，程序都要对这个C字符串数组进行一次内存重新分配操作，不然容易造成内存益出。
- 通过free记录，SDS实现了空间预分配和惰性释放两种优化策略
  - 空间预分配策略：需要对SDS的空间进行扩展时，程序不仅会为SDS分配修改所需要的空间，而且还会为SDS分配额外的空间。规则为：①SDS的长度小于1MB，那么程序会给SDS分配和length一样大的额外空间；②SDS的长度大于1MB，那么程序会给SDS分配额外的1MB空间，举个例子，比如修改后的SDS有30MB的大小，那么程序会分配1MB的未使用空间，SDS的buf数组实际大小将是30MB+1MB+1byte。(额外的一个字节用于保存空字符串)
  - 惰性释放：优化SDS的字符串缩短操作：当SDS的API要缩短SDS保存的字符串时，程序并不需要立即使用内存重分配策略来回收缩短后多出来的字节，而是使用free属性将这些字节记录起来，并等待使用。（真正释放SDS未使用空间的方法sdsRemoveFreeSpace，**避免内存泄漏**）
    - 这个释放我再结合sdsRemoveFreeSpace源码分析**补充一点**。他也不是说直接就去释放内存，然后缩容（sdshdr64 -> sdshdr8） 。我们先找到sds的 len和 free长度；如果没有free空间，就不用缩容；需要缩容的时候，redis会进行一个权衡（if判断）：①先计算存储当前长度的字符串需要什么类型的sds；如果sds类型相较原来没有变化，或者都是大的sds，就没必要更新类型；直接realloc()方法缩一下buf的长度；②如果sdshdrXX的buf[]长度缩的很多，到了sdshdr8，就要malloc()新申请一块内存区，然后考虑buf[]；（设计还是挺优秀的）
    - **为什么能用更小的字符串却不用呢？**这主要是因为sdshdr8以上的字符串，有效负载机会都在buf[]上，len 和 free字段只是8个字节；比如：sdshdr64更新成sdshdr16，len 和 free字段各自节省了6个字节，总共节省了12字节；但是重新申请空间，要拷贝的数据量至少是2^8=256个字节，这就是sdshdr16的存储下限了，字节数再少就要用sdshdr8存储了。所以可以看到，节省空间带来的收益会转化为时间复杂度。

4、**二进制安全**

- C字符串中的字符必须符合某种编码(比如ASCII)，并且除了字符串末尾之外，字符串里面不能包含空字符串，否则最先被程序读入的空字符串将被误认为是字符串结尾。
- SDS API都是二进制安全的，所有SDS API都会以处理二进制的方式来处理存放在SDS buf中的数据，数据写什么样，它被读取时就是什么样子。

### 1. 跳表原理 - SkipList

跳表是一个随机化的数据结构，实质是一种可以进行二分查找的有序链表。**跳表在原有的有序链表上增加了多级索引，**通过索引来实现快速查询。跳表不仅能提高搜索性能，同时也可以提高插入和删除操作的性能。

- 跳表的空间复杂度为O(n)——空间变大了
- 跳表查找任意数据的时间复杂度为O(logn)，这个查找的时间复杂度和二分查找是一样的，但是我们却是基于单链表这种数据结构实现的。

### 2. 压缩列表 - ZipList

**数组的优势占用一片连续的空间可以很好的利用CPU缓存访问数据。**如果我们想要保留这种优势，又想节省存储空间我们可以对数组进行压缩。我们在遍历节点的之后就**知道每个节点的长度(占用内存的大小)，就可以很容易计算出下一个节点再内存中的位置。**这种结构就像一个简单的压缩列表了。

![img](https://images2018.cnblogs.com/blog/747151/201806/747151-20180603165317042-1756608666.png)

参数说明：
zlbytes：记录整个压缩列表占用的内存字节数。（4字节）
zltail：记录压缩列表表尾节点距离压缩列表起始地址有多少字节。（4字节）
zllen：记录了压缩列表包含的节点数量。（**2个字节，最多存2^16-1**; ps:如果ZipList想要存储大于2^16-1也行，这时候zllen全1，然后ZipList就不再把这个直接作为长度统计了，而是自己每次都重新统计，**这似乎是一种攻击？**通过修改这个值，让redis多次计算长度）
entryN：压缩列表的节点，节点长度由节点保存的内容决定。
zlend：特殊值0xFF（十进制255），用于标记压缩列表的末端。

- **当一个列表只包含少量列表项,**并且每个列表项要么就是小整数值,要么就是长度比较短的字符串,那么Redis就会使用压缩列表来做**列表**的底层实现。（Zset也是这样，但是数目增加之后，redis会把压缩列表转化为跳表）

-   **当一个哈希只包含少量键值对,**比且每个键值对的键和值要么就是小整数值,要么就是长度比较短的字符串,那么Redis就会使用压缩列表来做**哈希**的底层实现。

#### 2.2.1 entryN的数据结构

我们已经知道ZipList是通过zltail找到末尾，然后逆序遍历，那怎么区分一个个的entry呢？这是因为entry的数据结构还很复杂。

![](https://raw.githubusercontent.com/BBQldf/PicGotest/master/20220607113407.png)

- prevlen：前一个 entry 的存储大小，主要是为了方便从后往前遍历。
  - 如果 prevlen 数值小于 254 字节，那就只用一个字节来表示长度
  - 如果长度大于等于 254 字节，那就用 5 个字节，第一个字节是固定值 254(FE)，用来标识这是个特殊的数据，剩下的 4 个字节来表示实际的长度。（上面图中写错了）

- len：数据的编码形式（比如：表示的是字符串还是数字，以及对应的长度）
  - 当 entry 是个 string 时，encoding 的第一个字节的前 2 位将保存用于存储字符串长度的编码类型，然后是字符串的实际长度。
    - 第一个字节前2位是00,后6位存data的长度；这种场景最多可以存2^6-1
    - 第一个字节前2位是01，后6位+下一个字节组合,用来存data的长度，最多存2^(6+8) - 1 = 2^14 -1
    - 第一个字节前2位是10,后6位空闲不用，后面四个字节组成32位整数，用来存data的长度；最多存2^32 -1
  - 当 entry 是一个整数的时候，前 2 位设置为 11，在后面的6 位用于指定在这个报头之后将存储哪种类型的整数。
    - 第一个字节低6位全部为0,表示data为int16_t, 占2个字节
    - 第一个字节低6位为01 0000，表示data为int32_t 整数，占4个字节，相当于Java的int
    - 第一个字节低6位为10 0000，表示data为int64_t 整数，占8个字节，相当于Java的long
    - 第一个字节低6位为11 0000,表示data是一个占3个字节，24位的整数
    - 第-个字节低6位为11 1110，表示data是一个占1个字节，相当于Java的byte
    - 第一个字节低6位为11 xxxx,且低4位的取值在[0001, 1101]区间之内，那么这里的xxxx就用来存储data的值，没有额外的data部分了；（注意[0001, 1101]表示不是直接二进制的1-13，而是要减1，表示[0-12]）——极致的内存优化！

- entry-data：实际存储的数据







### 3. 哈希表 - Hash

作为数据库中间件使用时，是最重要的一种数据类型（很类似于java的map，python的dict字典，其实字典是更合适的，我在项目中也用到了字典格式，他可以多级映射，比map方便很多）；Redis hash是一个string类型的**field和value**的映射表，hash特别适合用于存储对象。

> 当hash中的field-value键值对长短端且个数少时（< 512时），使用压缩列表，存放在连续内存中
> 当hash中的field-value键值对个数多时，使用hashtable，hash表中

对于普通的key/value结构来存储，它有如下优势：

1. 如果用普通的key/value结构来存储，每次修改用户的某个属性需要，先反序列化改好后再序列化回去。开销较大。（因为value对象都是序列化后的）；通过 key(用户ID) + field(属性标签) 就可以操作对应属性数据了，既不需要重复存储数据，也不会带来序列化和并发修改控制的问题
2. 用普通的key/value结构来存储，用户ID会数据冗余。比如：对于用户的姓名、年龄、生日；都要创建一个key（id+姓名、id+年龄标签），然后在对应的key里面加入value；如果用hash来存储，key就是id，value里面对应的是{field:value}（姓名：123；年龄：12），可以一次读取完毕。

### 4. BitMap原理和应用

> 额外讲一个高级数据结构BitMap。
>
> 抖音、快手数亿级量级的APP，日活、月活、留存、漏斗分析、多维分析等是如何做到秒级响应的呢 ？这其中就是BitMap技术。
>
> **对比hashmap的优势：以bit为单位，占用内存小；hashMap是以字节为单位。**
>
> 举个例子：每存储⼀亿个 email地址，就需要 1.6GB的内存（⽤哈希表实现的具体办法是将每⼀个 email地址对应成⼀个⼋字节的信息指纹，然后将这些信息指纹存⼊哈希表，由于哈希表的存储效率⼀般只有 50%，因此⼀个 email地址需要占⽤⼗六个字节。⼀亿个地址⼤约要 1.6GB，即⼗六亿字节的内存）

- BitMap可以理解为通过一个bit数组来存储特定数据的一种数据结构；由于bit是数据的最小单位，所以这种数据结构往往是非常节省存储空间。
- Java SDK里边有提供BitMap的实现：java.util.BitSet
- Bloom Filter可以看做是更复杂的BitMap
- **作用：大量数据的快速排序、查找、去重**
  - 快速排序：0-7内的5个元素(4,7,2,5,3)排序（这里假设这些元素没有重复）,只需要8个Bit（1Bytes），把相应的位置上置为1即可。（但是一个缺点是，所有的数据不能有重复的）
  - 快速去重：20亿个整数中找出不重复的整数的个数，内存不足以容纳这20亿个整数。其实这个问题很简单，一个数字的状态只有三种，分别为不存在，只有一个，有重复。因此，我们只需要2bits就可以对一个数字的状态进行存储了，**假设我们设定一个数字不存在为00，存在一次01，存在两次及其以上为11。那我们大概需要存储空间2G左右**。 接下来的任务就是把这20亿个数字放进去（存储），如果对应的状态位为00，则将其变为01，表示存在一次；如果对应的状态位为01，则将其变为11，表示已经有一个了，即出现多次；如果为11，则对应的状态位保持不变，仍表示出现多次。 最后，统计状态位为01的个数，就得到了不重复的数字个数，时间复杂度为O(n)。
  - 快速查找：也很简单，对于一个int类型的数，一个元素是4字节占32位，那么除以32就知道元素的下标，对32求余数（%32）就知道它在哪一位，如果该位是1，则表示存在。
- **[一些更现实一点的场景](https://baijiahao.baidu.com/s?id=1671471006888482946&wfr=spider&for=pc)**：
  - 比如想知道某个应用，每天有多少个独立用户使用了该应用？可以根据该应用的用户访问日志，每天生成一个BitMap；每个用户对应BitMap里的一个位置，如果当天访问了，该位置就置为1，否则为0。这样要知道当天这个应用的总独立用户数，只需要看看那天的BitMap里边有多少个1。
  - 想知道某个应用，昨天使用过的用户中，有多少今天也使用了（回访情况）？可以在例子1（每天保存一个独立活跃用户的BitMap）的基础上，将昨天的BitMap和今天的BitMap进行AND操作，然后数一下生成的BitMap里有多少个1即可。
- <font color='red'>**难点：怎么将用户映射到BitMap里边的某个位置？**</font>（两种方法）
  - 采用外部数据（比如在数据库里保存用户到BitMap位置的映射），在数据库里边给用户分配一个数值型的用户ID，而用户ID的生成规则采用自增量的方式来产生（ID1就映射到bitmap的1,2就映射到2，以此类推）；这种方法，对于用户量较少的情况下，很好，bitmap位数少
  - 采用固定的规则（比如计算用户名的hash code，最常使用）。比如要求将用户映射到一个1024位的BitMap：用户A的hashcode是101，101除1024取余数是101，所以用户A就对应BitMap的第101位；而用户B的hashcode是1234567，1234567除1024取余数是647，用户B就对应BitMap的第647位。（这种不需要做外部查询数据库，很方便；但是bitmap的位数不好确定，并且可能有hash冲突的问题和一致性哈希问题，解决方法可以讲一下，参考本博客`Mysql数据库-11 4种hash冲突解决`和`分布式-8. 一致性Hash问题`）

#### 4.2 BitMap的优缺点？扩展分析它的局限性？

**优点：**

- BitMap空间的浪费少（不过引入压缩技术后优势应该不明显）要是应用每天都有新增用户，从BitMap的大小，基本就可以知道应用的总用户数
- BitMap里边直观的包含了每天的新增用户信息，便于做用户回访/保持分析。

**缺点：**

- 需要维护更大的用户到ID的映射表；
- 由于这个表在处理中有很大的查询量，有可能对该表的查询成为性能瓶颈很难满足跨应用的分析，比如计算两个应用的重合用户。（说到这个，甚至可以提一句联邦学习~~）

拿计算用户回访为例，需要计算APP1昨天新增的用户，今天有多少人继续使用了APP1？

- 对于使用全局用户ID的方案，需要通过数据库，查询出APP1昨天新增的用户是哪些，他们的用户ID是多少，由此构造一个昨天新增用户的BitMap；然后将该BitMap和今天的活跃用户的BitMap进行一个AND操作，就可以知道其中哪些用户今天继续使用了APP1；然后做一个计数就可以得出人数。

- 而对于使用应用级别用户ID的方案，假如前天APP1的用户总数是1000，昨天是2000，那么就知道昨天新增的用户在BitMap里边是位置1000-2000；那么要计算今天的回访，只需要加载今天的BitMap，看看1000-2000这个位置范围内有多少个1就可以了。

- 当长度超过64位时，比如需要表示0010000..0（68位），会用两个long类型的数据表示该BitSet，即实际的001000..00（68位） = [001000..0(64位)，000000000(64位)]
- 当两个长度不同的BitSet进行运算时，比如001（3位）和0010000..0（68位）进行运算，最终它们的长度均为68位，实际占有空间的大小为128bit。

### 5. Redis的删除策略（LRU、LFU、Random、TTL）

redis是一种内存级数据库，所有的数据均存储在内存中，通过TTL指令可以查看数据的状态。
xx:具有时效性的数据；
-1：代表永久性数据；
-2：已经过期的数据 或 已经删除了的数据 或 未定义的数据；

expires空间相当于是redis开辟的一个存放值的地址和对应过期时间的空间，删除策略相当于是维护下面这个空间中的东西，上面存的值不变。

- **定时删除**：创建一个定时器，当key设置有过期时间，且过期时间到达时，由定时器立即执行对key的删除。到点了就干掉expires中和上面的key值。用处理器性能换内存空间。（这可能会存在一个时间点上大量的数据同时过期，但是缓存雪崩的问题）	
- **惰性删除：**数据到达过期时间，不做处理。等下次访问该数据时，发现未过期，则返回值；发现已经过期，删除（删除expires空间和key值），并返回不存在。
- **定期删除**：周期性轮询redis库中的时效性数据，采用随机抽取策略，利用过期数据占比的方式控制删除的频度。

#### 2.4.1 list的内存优化 - ZipList 

> 上面我们说到，redis的list结构使用quicklist实现的，它里面主要包含next、prev指针，value字段；
>
> 这就有个几个问题，①如果每个node里面的value指针指向的数据很小，比如只存储了一个int值，那next、prev指针就占了node里面的绝大多数空间，真正存储数据的有效负载就很低。②链表list节点分配的话，就会导致很多不连续的内存碎片（因为表不是连续内存空间）；③链表查询很慢，需要从一端开始；

为了优化上面的问题，redis把node节点的value指针指向了ZipList实例。ZipList是一块连续的空间，里面可以存储多个list的元素。

也就是说，ZipList用一块连续的内存空间实现了类似链表的效果~。这样，quicklist相比于简单链表结构

- node节点更少，内存碎片也就更少；
- 而且一个node里面存放了多个元素，next、prev指针占的空间也就可以忽略了

ZipList的数据结构可以参考`二、redis - 2. 压缩列表`

### 2.6 有序集合（ZSET）：

有序集合它保留了集合不能有重复成员的特性，但不同的是有序集合中的元素可以排序。它和列表使用索引下标作为排序依据不同的是，**有序集合给每个元素设置一个分数（score）作为排序的依据。**

有序集合的元素不能重复，但是score可以重复。就像班里同学的学号不能重复，但是分数可以重复一样。

- zadd添加成员：向有序集合u:kaoshi添加用户test01和他的分数98`zadd u:kaoshi 98 test01`

- zscore计算某个成员的分数,如果成员不存在则返回nil:`zscore key member`：计算有序集合key中，用户member的分数

- zrank计算成员的排名（zrank是分数从低到高返回排名，zrevrank是分数是从高到低返回排名，返回排名值）：`zrank u:kaoshi test01`(返回 3)

- zrem删除成员：允许一次删除多个成员，返回结果为成功删除的个数`zrem key member1 member2`

- zrange 和 zrevrange 返回指定排名范围的成员，如果加上withscores选项，同时会返回成员的分数：

  - ```
    zrange key min max 
    zrange key min max withscores
    ```

- zrangebyscore 返回指定分数范围的成员：`zrangebyscore key min max [withscores]`

- zcount返回指定分数范围成员个数：`zcount key min max`

#### 2.6.1 ZSet的低层实现

1、ZSet 当数据比较少时，采用 ziplist 存储，每个 member/score 元素紧凑排列，节省内存

2、当数据超过阈值（zset-max-ziplist-entries、zset-max-ziplist-value）后，转为 hashtable + skiplist 存储，降低查询的时间复杂度

3、hashtable 存储 member->score 的关系，所以 ZSCORE 的时间复杂度为 O(1)



## 3、redis并发竞争key问题如何解决？（分布式锁的问题）

本质上就是多个用户同时去set一个key！

那其实就是要考虑怎么去做分布式锁！

1. **乐观锁：**乐观锁适用于大家一起抢着改同一个key，**对修改顺序没有要求的场景。**watch 命令会监视给定的每一个key，当 exec 时如果监视的任一个key自从调用watch后发生过变化，则整个事务会回滚，不执行任何动作。
2. **分布式锁：**SETNX和EXPIRE两种
3. **时间戳：**适合有序需求场景，例如 A 需要把 key 设置为 a，然后 B 设置为 b，C 再设置为 c，最后的值应该是 c。在写入时保存一个时间戳，写入前先比较自己的时间戳是不是早于现有记录的时间戳，如果早于，就不写入了。
4. **消息队列：**在并发量很大的情况下，可以通过消息队列进行串行化处理。

## 4、如何解决数据库与缓存的数据一致性问题？（延时双删+消息队列）

本身Redis就是用户访问数据库的一个缓存中间件，当用户去取数据的时候，先访问Redis，再访问Mysql。只去查没问题，但是这个不一致的问题主要就是发生在**数据更新（包括删除）**的时候，由于更新有先后顺序，并且这时候也没有Mysql的多表事务操作，约束了ACID的特性！

有这么几种情况：**（“双更新的都有问题，一般是库更新再删缓存”）**

1. 先更新数据库，再更新缓存：
   - A更新完数据库，但缓存更新失败，后来的请求全部读的是错误信息。只有当缓存「失效」后，才能从数据库中得到正确的值。
   - 并发更新的问题：A更新数据库后（X = 1），B紧接着更新数据库（X = 2），然后B更新缓存（X = 2），最后A 更新缓存（X = 1）——最终 X 的值在缓存中是 1，在数据库中是 2，发生不一致。
2. 先更新缓存，再更新数据库
   - A更新完缓存，但数据库更新失败，后来的请求一开始读的是正确信息。一旦缓存「失效」，就会从数据库中读取到「旧值」，重建缓存也是这个旧值。、
   - 并发更新的问题：A更新缓存后（X = 1），B紧接着更新缓存（X = 2），然后B更新数据库（X = 2），最后A 更新数据库（X = 1）——最终 X 的值在缓存中是 2，在数据库中是 1，发生不一致。
   - <font color='red'>还有一些问题：</font>
     - 每次数据发生变更，都「无脑」更新缓存，但是缓存中的数据不一定会被「马上读取」，这就会导致缓存中可能存放了很多不常访问的数据，浪费缓存资源。
     - 写到缓存中的值，并不是与数据库中的值一一对应的，很有可能是先查询数据库，再经过一系列「计算」得出一个值，才把这个值才写到缓存中。(时间差的存在)
     - 这种「更新数据库 + 更新缓存」的方案，不仅缓存利用率不高，还会造成机器性能的浪费。
     - 所以此时我们需要考虑另外一种策略：删除缓存。
3. 先删除缓存，再更新数据库（删除策略解决了更新后，读出错的问题，但是并发的问题还是存在）
   - 请求B请求一个空的缓存，然后去数据库拿数据，补录到redis；但是A还没更新成功数据库，导致B拿到错误数据。
   - 最简单的解决办法就是**延时双删**的策略，保证事务提交完以后再进行删除缓存。但是**无法解决Mysql的主从同步问题**（从数据库还没更新好，就强制将其指向主库进行查询）
4. 先**更新数据库，再删除缓存**：最终一致性策略。
   - 更新数据库成功了，但是在删除缓存的阶段出错了没有删除成功
   - 参考下面↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓

解决方案：

- 核心思想：就是分布式事务中的最终一致性（异步重试）！（那这一块就很熟了）——针对的都是“**先更新数据库，在删除缓存**”这种方式：
  1. 基于RocketMQ、RabbitMQ、kafka这类消息中间件，将Redis 的 key 作为消息体发送到消息队列中，然后从队列中获取需要删除的数据
     - 这样的操作会对业务代码进行大量的侵入，但是不需要增加的系统，不需要增加整体的服务的复杂度。
  2. 基于Mysql的canal数据库同步组件（阿里的），当用户更新数据库后，从binlog日志中加载数据，再同步到redis中。（本质上也是用的中间件的思想）
     - 订阅 binlog 日志进行搭建独立系统操作 Redis，这样的缺点其实就是增加了系统复杂度。
- 可以做到强一致吗？
  - 很难！！要想做到强一致，最常见的方案是 2PC、3PC、Paxos、Raft 这类一致性协议，但它们的性能往往比较差，而且这些方案也比较复杂，还要考虑各种容错问题。（缓存的性能就被削减了，舍本逐末的感觉）。
  - 如果非要追求强一致，那必须要求所有更新操作完成之前期间，不能有「任何请求」进来，比如，分布式锁。

**然后延展一下：**就是MySQL中存在主从数据库的情况，然后用户**查数据**也会存在不一致的问题。（这个也是分布式带来的问题）上面的是更新操作，解决方案是引入了一些策略；而针对查询操作，解决方案是引入了一些技术。

所以有一些操作被引入，比如**分布式锁，消息队列**，这些都可以用Zookeeper和Redis来实现。

### 4.1 思考：如何解决本地缓存和数据库的一致性？

> 一般情况下，问一致性的问题，只用回答上面即可。但是可以补充上“本地缓存和数据库的一致性”问题解决方案。
>
> 场景题：服务涉及到二级缓存，对于热点数据（活动数据），放在本地缓存；对于非热点数据，放在redis缓存。接口优先查询一级缓存。

本地缓存常见的是hashMap、currentHashMap、guara cache、coffeine等。

解决方法：

1. MQ方案（推荐，通用）：比如：
   1. 请求通过nginx.的负载均衡策略转发到某一合应用A-1.
   2. 应用A-1收到请求，更新db,同时更新库用自己的本地缓存
   3. 应用A-1发送更新my广播消息
   4. 应用A-2和应用A-3收到消息，查询db,更新本地缓存
   5. 这个时候应用A-1,A-2A-3与DB数据就保持一致
2. 比较本地JVM变量的版本号和redis版本号（简单，但不实用；如果没有mq的情况下可以使用）：
   1. 在应用本地定义一个成员版本变量LongAdder localVersion = new LongAdder();
   2. 比较redis版本号与本地版本变量localVersion是否相等;
   3. 如果两个值相等，不处理:
   4. 如果不相等，说明别的应用的本地缓存已经更新，读取db数据更新到本应用的本地缓存，同时将redis版本号赋值给本地版本变量localVersion,使他们数据一致;

可以看到，这个方法的解决思路和MySQL、Redis的解决方案几乎一致。



### 4.2 分布式锁的实现——[基于Mysql、Redis、Zookeeper](https://blog.csdn.net/zhaisharap/article/details/122471322)？

既然都讲到了分布式锁，那就展开讲一下他们的具体实现吧！

- 在分布式环境中，对资源进行上锁有时候是很重要的，比如抢购某一资源，这时候使用分布式锁就可以很好地控制资源。

##### **基于数据库的实现方式：**

- 核心思想是：在数据库中创建一个表，表中包含方法名等字段，并在方法名字段上创建唯一索引，想要执行某个方法，就使用这个方法名向表中插入数据，成功插入则获取锁，执行完成后删除对应的行数据释放锁。（利用索引和insert语句带来的行锁）
- 成功插入则获取锁，执行完成后删除对应的行数据释放锁：`delete from method_lock where method_name ='methodName';`
- 上面只是最基本的一种实现，仍然存在的问题：
  - 不具备可重入的特性，因为同一个线程在释放锁之前，行数据一直存在，无法再次成功插入数据，所以，需要在表中新增一列，用于记录当前获取到锁的机器和线程信息
  - 没有锁失效机制，因为有可能出现成功插入数据后，服务器宕机了，对应的数据没有被删除，当服务恢复后一直获取不到锁，所以，需要在表中新增一列，用于记录失效时间，并且需要有定时任务清除这些失效的数据；
  - 不具备阻塞锁特性，获取不到锁直接返回失败，所以需要优化获取逻辑，循环多次去获取。

##### **基于Redis的实现方式：**

> [Redis实现分布式锁的7种方案](https://blog.csdn.net/fengyuyeguirenenen/article/details/123752418#comments_23249323)

1、核心思想：使用SETNX命令。

- `SETNX key val`：当且仅当key不存在时，set一个key为val的字符串，返回1；若key存在，则什么都不做，返回0。
- expire key timeout：为key设置一个超时时间，单位为second，超过这个时间锁会自动释放，避免死锁。delete key：删除key
- 获取锁的时候，使用setnx加锁，并使用expire命令为锁添加一个超时时间，超过该时间则自动释放锁，锁的value值为一个随机生成的UUID。释放锁的时候，通过UUID判断是不是该锁，若是该锁，则执行delete进行锁释放。
- 然后额外开辟一个线程，用这个线程去把锁的时间延长。
- 存在的问题：setnx和expire两个命令分开了， 「**不是原子操作**」 。如果执行完setnx加锁， **正要执行expire设置过期时间时，进程crash**。
  或者要重启维护了，那么这个锁就“长生不老” 了，「别的线程永远获取不到锁啦」 。

2、核心思想：SETNX + value值是（系统事件+过期时间）

- 为了解决上面“发生异常锁得不到释放的场景”，可以把过期时间放到setnx的value值里面，如果加锁失败，再拿出来value值校验一下

```java
long expires = System.currentTimeMillis() + expireTime; //系统时间+设置的过期时间
String expiresStr = String.valueOf(expires);

// 如果当前锁不存在，返回加锁成功
if (jedis.setnx(key_resource_id, expiresStr) == 1) {
        return true;
} 
// 如果锁已经存在，获取锁的过期时间
String currentValueStr = jedis.get(key_resource_id);

// 如果获取到的过期时间，小于系统当前时间，表示已经过期
if (currentValueStr != null && Long.parseLong(currentValueStr) < System.currentTimeMillis()) {

     // 锁已过期，获取上一个锁的过期时间，并设置现在锁的过期时间（不了解redis的getSet命令的小伙伴，可以去官网看下哈）
    String oldValueStr = jedis.getSet(key_resource_id, expiresStr);
    
    if (oldValueStr != null && oldValueStr.equals(currentValueStr)) {
         // 考虑多线程并发的情况，只有一个线程的设置值和当前值相同，它才可以加锁
         return true;
    }
}
        
//其他情况，均返回加锁失败
return false;
}
```

- **存在的缺点：**
  - 过期时间是客户端自己生成的（System.currentTimeMillis()是当前系统的时间），必须要求分布式环境下，每个客户端的时间必须同步
  - 如果锁过期的时候，并发多个客户端同时请求过来，都执行jedis.getSet()，最终只能有一个客户端加锁成功，但是该客户端锁的过期时间，可能被别的客户端覆盖
  - **该锁没有保存持有者的唯一标识，可能被别的客户端释放/解锁**（不是真正意义上的独占）
    - 假设线程a执行完后，去释放锁。但是它不知道当前的锁可能是线程b持有的（线程a去释放锁时，有可能过期时间已经到了，此时线程b进来占有了锁）。那线程a就把线程b的锁释放掉了，但是线程b临界区业务代码可能都还没执行完呢。

3、核心思想：分布式锁算法——Redlock

- redis一般都是集群部署的：如果线程一在Redis的master节点上拿到了锁，但是加锁的key还没同步到slave节点。恰好这时，master节点发生故障，一个slave节点就会升级为master节点。线程二就可以获取同个key的锁啦，但线程一也已经拿到锁了，锁的安全性就没了。
- 假设有5个master节点，RedLock的实现步骤如下：
  - 按顺序向5个master节点请求加锁
  - 线程的请求会根据设置的超时时间来判断，是不是要跳过该master节点
  - 如果大于等于3个节点（N/2+1，这里是5/2+1=3个节点）加锁成功，并且使用的时间小于锁的有效期，即可认定加锁成功啦
  - 如果获取锁失败，解锁！



##### **基于ZooKeeper的实现方式：**

- 核心思想：利用Zookeeper的目录结构，它内部是一个分层的文件系统目录树结构，规定同一个目录下只能有一个唯一文件名。具体的流程为：
  - 创建一个目录mylock；
  - 线程A想获取锁就在mylock目录下创建临时顺序节点；
  - 获取mylock目录下所有的子节点，然后获取比自己小的兄弟节点，如果不存在，则说明当前线程顺序号最小，获得锁；
  - 线程B获取所有节点，判断自己不是最小节点，**设置监听**比自己次小的节点；
  - 线程A处理完，删除自己的节点，线程B监听到变更事件，判断自己是不是最小的节点，如果是则获得锁。
- 优点：具备高可用、可重入、阻塞锁特性，可解决失效死锁问题。
- 缺点：因为需要频繁的创建和删除节点，性能上不如Redis方式。

 当然，在具体使用中，还需要考虑很多因素，比如超时时间的选取，获取锁时间的选取对并发量都有很大的影响，上述实现的分布式锁也只是一种简单的实现，主要是一种思想





PS：

ZooKeeper一般搭配Kafka这种中间件，统一配置管理、统一命名服务、分布式锁、集群管理。（他只是一个组件，但是干的事很多啊）

ZooKeeper的数据结构，跟Unix文件系统非常类似，可以看做是一颗树，每个节点叫做ZNode。每一个节点可以通过路径来标识



## 5、Redis是单线程还是多线程？（和Nginx类似）

单线程！！内部使用epoll多路复用。不负责数据的读写，只管理读写的事件。

- redis的操作是原子性的，采用独立的进程，彼此之间不会影响，提高可靠性。即使一个线程崩溃，其他线程也没事。（不会和Mysql一样，会有一致性的问题。如果指令执行失败，一般是指令写错了！）失败就是失败，也没有回滚操作；
- 无论什么版本，redis都是单线程。但是，在服务器端进行IO多线程之后，把输入/输出放到更多的线程里去并行处理，可以缩短执行时间，压榨系统及硬件的资源（比如，网卡能够高效的使用）
- 注意：
  - 少使用事务（`wait 3 300`这种做同步，防止不一致性的指令）
  - 缓存本身就是不严谨的，如果担心redis挂了，给后端数据库带来大压力，我们可以考虑**数据分片**！！把数据拆到若干台redis服务器中，一个挂了，另外的还能用。

### 5.1、Redis 的线程模型了解么？ 

- Redis 内部使⽤⽂件事件处理器 file event handler ，这个⽂件事件处理器是单线程的，所以 Redis 才叫做单线程的模型。

- 它采⽤ IO 多路复⽤机制同时监听多个 Socket，根据 Socket 上的事件来选择对 应的事件处理器进⾏处理。

### 5.2、[Redis怎么解决大Key问题？](https://blog.csdn.net/yangbindxj/article/details/123207063)

#### 5.2.1、 **定义**

- 什么是大key？这里就有个误区了，所谓的大key问题是某个key的value比较大，所以本质上是大value问题。
- key往往是程序可以自行设置的，value往往不受程序控制，因此可能导致value很大。



#### 5.2.2、 **场景：**

> 在线音乐app中，某个歌单有很多用户收藏，假如有这样的数据结构：
>
> - 歌单和用户之间的映射关系采用redis存储
> - redis的key是歌单ID，长度可控且很小
> - redis的value是个list，list包含了用户ID
> - 用户可能很多，就导致list长度不可控

redis中有常见的几种数据结构，每种结构对大key的定义不同，比如：

- value是String类型时，size超过10KB
- value是ZSET、Hash、List、Set等集合类型时，它的成员数量超过1w个



#### 5.2.3、 知道了场景，基本就知道了**大Key是怎么产生的？**

- 本质上就是业务设计不合理，没有预见vaule的动态增长问题，一直往value塞数据，没有删除机制，迟早要爆炸
- 数据没有合理做分片，将大key变成小key



#### 5.2.4、 **大key有什么影响？**

redis的一个典型特征就是：核心工作线程是单线程。

单线程中请求任务的处理是串行的，前面完不成，后面处理不了，同时也导致分布式架构中内存数据和CPU的不平衡。

- 执行大key命令的客户端本身，耗时明显增加，甚至超时
- 执行大key相关读取或者删除操作时，会严重占用带宽和CPU，影响其他客户端
- 大key本身的存储带来分布式系统中分片数据不平衡，CPU使用率也不平衡
- 大key有时候也是热key，读取操作频繁，影响面会很大
- 执行大key删除时，在低版本redis中可能阻塞线程

这样看来大key的影响还是很明显的，**最典型的就是阻塞线程，并发量下降，导致客户端超时，服务端业务成功率下降**。



#### 5.2.5、 **如何找到大key**

- 核心：增加内存&流量&超时等指标监控
- bigkeys命令：使用bigkeys命令以遍历的方式分析Redis实例中的所有Key，并返回整体统计信息与每个数据类型中Top1的大Key
- redis-rdb-tools：使用redis-rdb-tools离线分析工具来扫描RDB持久化文件，虽然实时性略差，但是完全离线对性能无影响。



#### 5.2.6、**如何解决大key问题**

根据大key的实际用途可以分为两种情况：

- 可删除：
  - 如果发现某些大key并非热key就可以在DB中查询使用，则可以在Redis中删掉
  - 当Redis版本大于4.0（惰性版本）时，可使用UNLINK命令安全地删除大Key，该命令能够**以非阻塞的方式，逐步地清理传入的Key**
    - 与 DEL 命令相同，表示删除指定的 key，如果指定 key 不存在，命令则忽略。
    - 与 DEL 命令不同在于它是异步执行的，因此它不会阻塞
  - 当Redis版本小于4.0（渐进式版本）时，避免使用阻塞式命令KEYS，而是建议通过SCAN命令执行增量迭代扫描key，然后判断进行删除
- 不可删除：（压缩和拆分）
  - 当vaule是string时，比较难拆分，则使用序列化、压缩算法（LZF压缩算法，核心就是对重复值压缩）将key的大小控制在合理范围内，但是序列化和反序列化都会带来更多时间上的消耗
  - 当value是string，压缩之后仍然是大key，则需要进行拆分，一个大key分为不同的部分，记录每个部分的key，使用multiget等操作实现事务读取
  - 当value是list/set等集合类型时，根据预估的数据规模来进行分片，不同的元素计算后分到不同的片

### 5.3 Redis 的大 Key 对持久化有什么影响？

当 AOF 写回策略配置了 Always 策略，如果写入是一个大 Key，主线程在执行 fsync() 函数的时候，阻塞的时间会比较久，因为当写入的数据量很大的时候，数据同步到硬盘这个过程是很耗时的。

AOF 重写机制和 RDB 快照（bgsave 命令）的过程，都会分别通过 `fork()` 函数创建一个子进程来处理任务。会有两个阶段会导致阻塞父进程（主线程）：

- 创建子进程的途中，由于要复制父进程的页表等数据结构，阻塞的时间跟页表的大小有关，页表越大，阻塞的时间也越长；
- 创建完子进程后，如果父进程修改了共享数据中的大 Key，就会发生写时复制，这期间会拷贝物理内存，由于大 Key 占用的物理内存会很大，那么在复制物理内存这一过程，就会比较耗时，所以有可能会阻塞父进程。

**大 key 除了会影响持久化之外，还会有以下的影响（实际性的）：**

- **客户端超时阻塞：**由于 Redis 执行命令是单线程处理，然后在操作大 key 时会比较耗时，那么就会阻塞 Redis，从客户端这一视角看，就是很久很久都没有响应。
- **引发网络阻塞：**每次获取大 key 产生的网络流量较大，如果一个 key 的大小是 1 MB，每秒访问量为 1000，那么每秒会产生 1000MB 的流量，这对于普通千兆网卡的服务器来说是灾难性的。
- **阻塞工作线程：**如果使用 del 删除大 key 时，会阻塞工作线程，这样就没办法处理后续的命令。
- **内存分布不均：**集群模型在 slot 分片均匀情况下，会出现数据和查询倾斜情况，部分有大 key 的 Redis 节点占用内存多，QPS 也会比较大。

#### 5.3.1 如何避免大 Key 呢？

最好在设计阶段，就把大 key 拆分成一个一个小 key。或者，定时检查 Redis 是否存在大 key ，如果该大 key 是可以删除的，不要使用 DEL 命令删除，因为该命令删除过程会阻塞主线程，而是用 unlink 命令（Redis 4.0+）删除大 key，因为该命令的删除过程是异步的，不会阻塞主线程。



## 6、redis集群——读写分离

为什么要用redis读写分离：单机情况下redis能承受大约2万的QPS（具体数据因机器配置与业务场景而异），如果想要承接更高数值的QPS（10万以上），则需要用到读写分离的redis集群。

### 1、redis是单线程的，我们现在服务器都是多核的，那不是很浪费？

- 他是单线程的，但是，我们可以通过在单机开多个Redis实例（就相当于集群的思想）
- 可以进行多IO操作：我们⽤到了集群的部署⽅式也就是Redis cluster，并且是主从同步读写分离，类似Mysql的主从同步，Redis cluster ⽀撑 N 个 Redis master node，每个master node都可以挂载多个 slave node。

Redis**至少需要存在三个数据分片**，每个分片称为master，分片之前的关系为完全图，假设整个cluster有N个节点，那么每个节点都和其他N-1个节点保持连接和心跳，节点之间采用Gossip协议进行通信。通信主要确认节点是否存活、节点的数据版本、投票选择新的master等内容。

- Redis Cluster着眼于扩展性，在单个redis内存不⾜时，使⽤Cluster进⾏分⽚存储。Redis-Cluster采用无中心结构，每个节点保存数据和整个集群状态,每个节点都和其他所有节点连接。
  - 当主节点宕机了，整个集群就没有可写的节点了。 由于从节点上备份了主节点的所有数据，那在主节点宕机的情况下，如果能够将从节点变成一个主节点，是不是就可以解决这个问题了呢？
    - Redis Sentinal着眼于⾼可⽤，在master宕机时会⾃动将slave提升为master，继续提供服务。

### 2、集群之间怎么交互？

- 集群中所有的Master节点都可以进行读写数据，不分主次，即redis集群式去中心化的。
- 由于Cluster架构中无Proxy层，客户端是直接与集群中的任意可用节点直接交互的，【话是这么说，但是一个请求是怎么找到集群中的一个节点的呢，redis有多种策略，一般使用CRC16去hash(key)计算改请求要分配到具体的哪一个节点上。然后才是 客户端与节点的直接操作】对象保存到Redis之前先经过CRC16哈希到一个指定的Node上。
  - 可能会问负载均衡的问题？！！



### 3、怎么保证Redis集群的高可用！？——主从+哨兵；本质：读写分离

> Redis中哨兵集群中的哨兵领导者选举和Redis-Cluster中的主节点选举都是基于**Raft算法**实现的。
>
> 核心就两个：
>
> 1. 选主 Leader Election
> 2. 复制日志 Log Replication

1. redis主从复制（提到这个，就跟我前⾯提到的 数据持久化的RDB和AOF有着⽐密切的关系了。）
   - 原理：让master机器去写， 数据同步给别的slave机器，他们都拿去读。
   - 启动⼀台slave 的时候，他会发送⼀个psync命令给master ，如果是这个slave第⼀次连接到master， 他会触发⼀个全量复制。master就会启动⼀个线程，⽣成RDB快照，还会把新的写请求都缓存在内存 中，RDB⽂件⽣成后，master会将这个RDB发送给slave的，slave拿到之后做的第⼀件事情就是写进本 地的磁盘，然后加载进内存，然后master会把内存⾥⾯缓存的那些新命名都发给slave。
2. 哨兵机制sentinel。
   - 集群监控：负责监控 Redis master 和 slave 进程是否正常⼯作。
   -  消息通知：如果某个 Redis 实例有故障，那么哨兵负责发送消息作为报警通知给管理员。
   -  故障转移：如果 master node 挂掉了，会⾃动转移到 slave node 上。
   -  配置中心：如果故障转移发⽣了，通知 client 客户端新的 master 地址。

### 4、Redis的同步机制了解么？

Redis可以使⽤主从同步，从从同步。第⼀次同步时，主节点做⼀次bgsave，并同时将后续修改操作记 录到内存buffer，待完成后将RDB⽂件全量同步到复制节点，复制节点接受完成后将RDB镜像加载到内 存。加载完成后，再通知主节点将期间修改的操作记录同步到复制节点进⾏重放就完成了同步过程。后续的增量数据通过AOF⽇志同步即可，有点类似数据库的binlog。

**哨兵+主从并不能保证数据不丢失，但是可以保证集群的高可用。**



## 7、Redis持久化？

### 7.1 RDB快照

：RDB 持久化机制，是对 Redis 中的数据执⾏周期性的持久化。 （理解为全量备份）

- 原理：fork和cow。fork是指redis通过创建⼦进程来进⾏RDB操作，cow指的是 copy on write，⼦进程创建后，⽗⼦进程共享数据段，⽗进程继续提供读写服务，写脏的⻚⾯数据会逐 渐和⼦进程分离开来。
- RDB对Redis的性能影响⾮常⼩，是因为在同步数据的时候他只**是fork了⼀个⼦进程去做持久化的**，⽽ 且他在数据恢复的时候速度⽐AOF来的快。
- 有一个问题就是，RDB在⽣成数据快照的时候，如果⽂件很⼤，客户端可能会暂停⼏毫秒甚⾄⼏秒，你公司在做 秒杀的时候他刚好在这个时候fork了⼀个⼦进程去⽣成⼀个⼤快照，哦豁，出⼤问题。

### 7.2 AOF日志

AOF 机制对每条写⼊命令作为⽇志，以 append-only 的模式写⼊⼀个⽇志⽂件中，因为这 个模式是只追加的⽅式，所以没有任何磁盘寻址的开销，所以很快，有点像Mysql中的binlog。（增量备份）

- 是AOF是⼀秒⼀次去通过⼀个后台的线程 **fsync 操作**，那 最多丢这⼀秒的数据。**快！**
- ⼀样的数据，AOF⽂件⽐RDB还要⼤。**文件大！**

Redis 提供了 3 种 AOF 日志写回硬盘的策略，分别是：

- Always，这个单词的意思是「总是」，所以它的意思是每次写操作命令执行完后，同步将 AOF 日志数据写回硬盘；
- Everysec，这个单词的意思是「每秒」，所以它的意思是每次写操作命令执行完后，先将命令写入到 AOF 文件的内核缓冲区，然后每隔一秒将缓冲区里的内容写回到硬盘；
- No，意味着不由 Redis 控制写回硬盘的时机，转交给操作系统控制写回的时机，也就是每次写操作命令执行完后，先将命令写入到 AOF 文件的内核缓冲区，再由操作系统决定何时将缓冲区内容写回硬盘。

两种⽅式都可以把Redis内存中的数据持久化到磁盘上，然后再将这些数据备份到别的地⽅去，**RDB更 适合做冷备，AOF更适合做热备**。



## 8、Redis 和 Memcached 有啥区别，为啥选择⽤Redis作为你们的缓存中间件？

- Redis ⽀持复杂的数据结构；
- Redis 原⽣⽀持集群模式；
- 由于 Redis 只使⽤单核，⽽ Memcached 可以使⽤多核，所以平均每⼀个核上 Redis 在存储⼩数据时 ⽐ Memcached 性能更⾼。⽽在 100k 以上的数据中，Memcached 性能要⾼于 Redis，虽然 Redis 最近也在存储⼤数据的性能上进⾏优化，但是⽐起 Remcached，还是稍有逊⾊。

## 9、 缓存雪崩、穿透、击穿？怎么解决？

- 缓存雪崩：
  - 当缓存服务器重启或者大量缓存集中在某一个时间段失效，这样在失效的时候，也会给后端系统(比如DB)带来很大压力。
  - 解决方案：
    - 设置缓存失效时间，不要再统一时间失效；（随机初始化失效时间）
    - 利用redis集群，让热点key均匀地分布在不同的节点上
    - 定时刷新缓存（比如设置了3小时失效，每接近3小时的时候就去刷新一下缓存）
- 缓存穿透：(针对恶意攻击)
  - key对应的数据在数据源并不存在，每次针对此key的请求从缓存获取不到，请求都会到数据源，从而可能压垮数据源（数据库会进行全表查询）。比如用一个不存在的用户id获取用户信息，不论缓存还是数据库都没有，若黑客利用此漏洞进行攻击可能压垮数据库。
  - 解决方案：
    - 如果发生了穿透，数据库无论查询什么结果，都把它缓存到redis中（针对同一个id）
    - 拉黑ip
    - 对参数合法性进行校验。
    - 使用布隆过滤器（缓存穿透，海量数据快速去重）。简单的说⼀下就是我们先把我们数据库的数据都加载到我们的过滤器中。检索时，我们只要看看这些点是不是都是1就（⼤约）知道集合中有没有它了：如果这些点有任何⼀个0，则被检元素⼀定不在；如果都是1，则被检元素很可能在。
- 缓存击穿：
  - **key对应的数据存在，但在redis中过期**，此时若有大量并发请求过来，这些请求发现缓存过期一般都会从后端DB加载数据并回设到缓存，这个时候大并发的请求可能会瞬间把后端DB压垮。（热点key被长时间访问，导致超过了key的过期时间）
  - 解决方案：
    - 直接设置不过期
    - 补偿类型（已经出现了缓存击穿）的，分布式锁（互斥锁）；（甚至可以是锁机制，一个进程从数据库拿到数据后，就把它加载到缓存中）
    - 甚至一些**限流机制**也可以用上来

⼀般避免以上情况发⽣我们从三个时间段去分析下：

- 事前：Redis ⾼可⽤，主从+哨兵，Redis cluster，避免全盘崩溃。（先用大集群先准备好）
- 事中：本地 ehcache 缓存 + Hystrix 限流+降级，避免MySQL 被打死。（赶快补救，退化为高并发的为，要降级处理）
- 事后：Redis 持久化 RDB+AOF，⼀旦重启，⾃动从磁盘上加载数据，快速恢复缓存数据。（恢复一下）

### 9.2 避免缓存击穿的利器之BloomFilter

> ⼀般都会在回答缓存击穿，或者海量数据去重这个时候引出来。
>
> Guava中就提供了⼀种Bloom Filter的实现

- 它实际上是⼀个很⻓的⼆进制向量和**⼀系列**随机映射函数。
- 作用：⽤于检索⼀个元素是否在⼀个集合中。
- 它的优点是空间效率和查询时间都远远超过⼀般的算法，缺点是有⼀定的误识别率和删除困难。
- **工作原理：**当⼀个元素被加⼊集合时，通过K个散列函数将这个元素映射成⼀个位数组中的K个点，把它们置为1。检索时，我们只要看看这些点是不是都是1：
  1. 如果都是1，则被检元素很可能在。
  2. 如果这些点有任何⼀个0，则被检元素⼀定不在；（很像hashCode和equals的区别，hashCode不相同的元素一定不相等）

> **做个对比：**Bloom Filter跟单哈希函数Bit-Map不同之处在于：Bloom Filter使⽤了k个哈希函数，每个字符串跟k个bit对应。（从⽽降低了冲突的概率。）

说了这么多，讲一下它的**实际应用场景：**

- **所谓缓存穿透：**简单来说就是你数据库的id都是1开始然后⾃增的，那我知道你接⼝是通过id查询的，我就拿负数去查询**（拿一个缓存中肯定没有的值，直接穿透了redis）**，这个时候，会发现缓存⾥⾯没这个数据，我⼜去数据库查也没有，⼀个请求这样，100个，1000个，10000个呢？你的DB基本上就扛不住了。
- 如果我在缓存里面额外存一个id的BloomFilger，判断没这个数据就不去查了，直接return⼀个数据为空不就好了嘛。
  - 比如：先把我们数据库的数据都加载到我们的过滤器中，对于id为`9864`，经过三次hash之后，把过滤器的二进制向量中相应位置变成1；下次我进来查询如果id也是1，那我就把1拿去三次hash 发现跟上⾯的三个位置完全⼀样，那就能证明过滤器中有1的，反之如果不⼀样就说明不存在了

**PS：一些其他的应用场景：**

- 爬⾍过滤已抓到的url就不再抓，可⽤bloom filter过滤（这个可以展开说一下）
- 防火墙领域里面cerberus（看门狗watchDog）在收集监控数据的时候, 有的系统的监控项量会很⼤, 需要检查⼀个监控项的名字是否已经被记录到db过了, 如果没有的话就需要写⼊db。（安全类扩展）

**缺点：**

- 存在误判，可能要查到的元素并没有在容器中，但是hash之后得到的k个位置上值都是1（特别是当数据量大的时候）。如果bloom filter中存储的是⿊名单，那么可以通过建⽴⼀个⽩名单来存储可能会误判的元素。
- 删除困难。⼀个放⼊容器的元素映射到bit数组的k个位置上是1，删除的时候不能简单的直接置为0，可能会影响其他元素的判断。可以采⽤Counting Bloom Filter

## 10、Redis VS Memcache

> 说到缓存技术，就会想到redis和memcache。他们都是内存型数据库。但互联网用到的缓存技术绝大部分都是使用redis。

### 1、支持的**数据类型**不同（最大的区别）

- memcache支持string类型，图片，视屏等缓存；

- redis不仅支持简单的k/v类型，还提供list，hash，set，zset等数据类型。
- 区别分析：memcache只支持字符串，也就是说它的value值只能是String类型，这就比较受限。
  - 如果你要存储一个集合类型，比如hash结构，就要把它先序列化成一个JSON或者别的字符串结构
  - 然后要修改hash里面的值，就需要把整个结构读入到java结构中去修改
  - 然后再序列化整个集合，再通过网络I/O写回到Memcache中。
  - 那这样，我其实就改了一个元素，但是网络I/O却传了整个数据，那有效负载就很低。
  - Redis天生支持各种数据结构，要修改某个元素，只需要传目标值就行。如果项目里面需要这些复杂结构的话，基本就淘汰了。不过memcache还可用于缓存其他东西，例如图片、视频等等。

### 2、是否支持**持久化**（比较重要的区别）

- memcache不支持持久化，数据都是在内存中，一旦停电则会造成数据丢失，且不可以恢复；（都有分布式功能，能在一定程度上解决）
- redis支持持久化，通过RDB/AOF两种方式来将数据持久化到磁盘，若停电还可以恢复数据。

### 3、**内存管理**——内存空间和key数量

- redis：它有专门的内存模型
  - 不同的对象类型在内存中的编码方式不同
  - redis的key和valve均不能超过512MB
  - Redis当物理内存用完时，可以将一些很久没用到的value 交换到磁盘.——也就是**虚拟内存**(这也意味着redis需要调用系统函数，会浪费一定的时间)
  - 还有一些内存回收算法，比如LRU、LFU这些
  - 既然都说到内存回收了，就不得不提一下**内存碎片**：
    - 如果Redis服务器中的内存碎片已经很大，可以通过安全重启的方式减小内存碎片：因为重启之后，Redis重新从备份文件中读取数据，在内存中进行重排，为每个数据重新选择合适的内存单元，减小内存碎片。
    - jemalloc作为Redis的默认内存分配器，在减小内存碎片方面做的相对比较好。jemalloc在64位系统中，将内存空间划分为小、大、巨大三个范围；每个范围内又划分了许多小的内存块单位；当Redis存储数据时，会选择大小最合适的内存块进行存储。
    - 如果内存碎片率过高（jemalloc在1.03左右比较正常），说明内存碎片多，内存浪费严重；这时便可以考虑重启redis服务，在内存中对数据进行重排，减少内存碎片。
    - 如果内存碎片率小于1，说明redis内存不足，部分数据使用了虚拟内存（即swap）；由于虚拟内存的存取速度比物理内存差很多（2-3个数量级），此时redis的访问速度可能会变得很慢。因此必须设法增大物理内存（可以增加服务器节点数量，或提高单机内存），或减少redis中的数据。
  - 也能设置过期时间，例如expire 设定，例如expire name 10；还有相应的过期策略
- memcache
  - memcache在set时就指定，例如set key1 0 0 8,即永不过期。
  - memcache的key最长250个字符，value不能超过1MB（可以通过配置文件修改）
- 其实两者都是很优秀的内存管理方案，

### 4、**线程模型**——这是重点（解决了为什么这么快！）

- 先说memcache吧，它是**多线程的**，在多核机器上，memcache会比redis的性能好一点；（`在本blog的二、redis - 6、redis集群`中介绍了为什么redis会是单线程，如何应对多核问题）
  - 尤其是在碰到大Key问题（某个key的value比较大，所以本质上是大value问题——产生的原因：key往往是程序可以自行设置的，value往往不受程序控制，因此可能导致value很大）的时候，因为大key会把redis的单线程堵死；
- redis都是单线程，然后都是原子操作

### 5、集群实现

- memcache实现：
  - memcache实例之间是不通信的，它是依赖客户端对key进行的一致性哈希，然后请求到Memcache集群的单台实例
- redis实现：
  - 天生就支持多种分布式的模式，有主从的、有哨兵模式、还有cluster

## 1、mysql的悲观锁和乐观锁？

悲观锁与乐观锁是人们定义出来的概念，你可以理解为一种思想，是处理并发资源的常用手段。**不要把他们与mysql中提供的锁机制(表锁，行锁，排他锁，共享锁)混为一谈。**

- 利用 select .... for update 的悲观锁语法锁住记录；右边（事务B）再想使用将被阻塞，同时，阻塞被解除后事务B能看到事务A对数据的修改
- **乐观锁认为一般情况下数据不会造成冲突，所以在数据进行提交更新时才会对数据的冲突与否进行检测。**如果没有冲突那就OK；如果出现冲突了，则返回错误信息并让用户决定如何去做。当修改了某个文件需要提交的时候，它会检查文件的当前版本是否与服务器上的一致，如果一致那就可以直接提交，如果不一致，那就必须先更新服务器上的最新代码然后再提交
  - 常见的做法有两种：版本号控制及时间戳控制。





## 6、TCP链接中状态对比？

TCP协议规定，对于已经建立的连接，网络双方要进行四次握手才能成功断开连接，如果缺少了其中某个步骤，将会使连接处于假死状态，连接本身占用的资源不会被释放。网络服务器程序要同时管理大量连接，所以很有必要保证无用连接完全断开，否则大量僵死的连接会浪费许多服务器资源。

常用的三个状态是：**ESTABLISHED** 表示正在通信，**TIME_WAIT** 表示主动关闭，**CLOSE_WAIT** 表示被动关闭。

- TIME_WAIT 是主动关闭链接时形成的，等待2MSL时间，约4分钟。主要是防止最后一个ACK丢失。 由于TIME_WAIT 的时间会非常长，因此server端应尽量减少主动关闭连接
- CLOSE_WAIT是被动关闭连接是形成的。根据TCP状态机，服务器端收到客户端发送的FIN，则按照TCP实现发送ACK，因此进入CLOSE_WAIT状态。但如果服务器端不执行close()，就不能由CLOSE_WAIT迁移到LAST_ACK，则系统中会存在很多CLOSE_WAIT状态的连接。此时，可能是系统忙于处理读、写操作，而未将已收到FIN的连接，进行close。

![](https://raw.githubusercontent.com/BBQldf/PicGotest/master/20220308174753.png)

为什么客户端的 TIME-WAIT 状态必须等待 2MSL ？

- 确保 ACK 报文能够到达服务端，从而使服务端正常关闭连接。第四次挥手时，客户端第四次挥手的 ACK 报文不一定会到达服务端。服务端会超时重传 FIN/ACK报文，此时如果客户端已经断开了连接，那么就无法响应服务端的二次请求，这样服务端迟迟收不
  到 FIN/ACK 报文的确认，就无法正常断开连接。
- 防止已失效的连接请求报文段出现在之后的连接中。TCP 要求在 2MSL 内不使用相同的序列号。客户端在发送完最后一个 ACK 报文段后，再经过时间2MSL，就可以保证本连接持续的时间内产生的所有报文段都从网络中消失。

## 7、tcp 如何最大程度利用带宽？（就是TCP的拥塞控制）

慢启动——>加法增（拥塞窗口cwnd+1）——>快重传（比如接收方在收到一个失序的报文段后就立即发出重复确认，而不要等到自己发送数据时捎带确认。）——>乘法减！

**如何检测拥塞？**

- TCP认为网络拥塞的主要依据是它重传了一个报文段。上面提到过，TCP对每一个报文段都有一个定时器，**称为重传定时器(RTO)**，当RTO超时且还没有得到数据确认，那么TCP就会对该报文段进行重传，当发生超时时，那么出现拥塞的可能性就很大，某个报文段可能在网络中某处丢失，并且后续的报文段也没有了消息。
- TCP会把：1）ssthresh降低为cwnd值的一半；2）把cwnd重新设置为1；3）重新进入慢启动过程。
