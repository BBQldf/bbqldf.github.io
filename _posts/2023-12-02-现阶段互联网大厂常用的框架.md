### 1、对于大型的服务治理平台，要实现的功能主要是上面的这五块（或者说四块）：

1. **服务发现和配置管理系统：**所有服务的API接口要能由统一的管理中心进行归类，方便其他的APP去调用这些API。
2. **分布式协调系统：**有了各个服务的API，要回答一个关键问题，**如何在任一时间点确定哪些服务器活着并且在工作中**？在分布式的环境中，网络失败、带宽限制、可变延迟连接、安全问题，甚至跨多个数据中心时可能发生的错误时，有状态服务迁移问题，如何解决？就近访问的路由策略，领导人选举、协同工作流和配置服务同步，以及广义的分布式数据结构如锁、队列、屏障（Barrier）和锁存器（Latch）这些都是非常有价值的设计，但是怎么去选型，是一个问题。（meituan是怎么做的，需要进一步探索）
3. **RPC调用（Remote Procedure Call）：**解决了服务本身与服务管理中心之间的ping问题，就要回答服务之间怎么调用了。要做的是，服务依赖梳理（上下游关系），调用失败之后的告警，多消息协议（Thrift/Protobuf）和多交互方式（Ping-Pong/Oneway/ Streaming），TP99 延迟指标等都是需要关注的点
4. **容器化：**容器适用于轻量、弹性、无状态等业务场景，使用容器云或容器化 PaaS 来支撑微服务架构业务应用，实现敏捷的业务应用服务开发和迭代，快速构建一致性的开发、测试环境，支持弹性伸缩应对突发流量，扩展服务治理增强安全管控，提供统一的日志、监控、审计等企业级能力中心等。（这一块OCTO提供了负载均衡、容错、灰度发布等治理功能，这些功能刚好K8s就自带）







### 2、要对比Thrift和gRPC的区别，这句话或许总结得好！

- **GRPC主要就是搞了个ProtoBuf，然后采用HTTP协议，所以协议部分没有重复造轮子，重点就在ProtoBuf上（这一部分在底层会变成序列化数据，本质上就是通信双方协商好一个统一的结构化数据存储格式规范，在反序列化后可以正常使用）。**
  - 由于gRPC基于HTTP 2.0标准设计，带来了更多强大功能，如多路复用、二进制帧、头部压缩、推送机制。这些功能给设备带来重大益处，如节省带宽、降低TCP连接次数、节省CPU使用等。gRPC既能够在客户端应用，也能够在服务器端应用，从而以透明的方式实现两端的通信和简化通信系统的构建。

- **Thrift的数据格式是用的现成的，没有单独搞一套，但是它在传输层和服务端全部是自己造轮子，所以可以对<font color='red'>协议层、传输层有多种控制要求。</font>**
  - Thrift使用一种IDL（Interface Definition Language）来定义数据结构和服务接口，类似于其他IDL语言（例如Protocol Buffer（紧凑型））。但是它又做了一些**扩展：**Thrift支持多种传输协议，如TBinaryProtocol（二进制协议）、TCompactProtocol（紧凑协议）和TJSONProtocol（JSON协议）等。
  - 服务端创新设计：①Thrift支持多种服务器模型，包括单线程服务器、多线程服务器和线程池服务器等。这些服务器模型可以根据应用的负载和并发需求进行选择，以获得更好的性能和资源利用率；②Thrift提供了可插拔的服务处理器接口，使开发者可以根据实际需求自定义服务的处理逻辑。这样的设计使得Thrift在服务端的扩展性更强，能够满足不同业务逻辑的需求。
    - TSimpleServer: 简单的单线程服务模型，常用于测试；
    - TThreadPoolServer: 多线程服务模型，使用标准的阻塞式IO；
    - TNonBlockingServer: 多线程服务模型，使用非阻塞式IO(需要使用TFramedTransport数据传输方式);
    - THsHaServer: THsHa引入了线程池去处理，其模型读写任务放到线程池去处理，Half-sync/Half-async处理模式，Half-async是在处理IO事件上(accept/read/write io)，Half-sync用于handler对rpc的同步处理；

  - 错误处理机制：Thrift在IDL中允许显式地定义异常类型，并可以在服务接口中声明可能抛出的异常。这使得Thrift在错误处理方面更加灵活。Protocol Buffers没有内置的异常机制，因此在处理错误时需要采用其他方式。




**gRPC 特点：**

- 语言中立，支持多种语言；
- 基于 IDL 文件定义服务，通过 proto3 工具生成指定语言的数据结构、服务端接口以及客户端 Stub；
- 通信协议基于标准的 HTTP/2 设计，支持双向流、消息头压缩、单 TCP 的多路复用、服务端推送等特性，这些特性使得 gRPC 在移动端设备上更加省电和节省网络流量；
- 序列化支持 PB（Protocol Buffer）和 JSON，PB 是一种语言无关的高性能序列化框架，基于 HTTP/2 + PB, 保障了 RPC 调用的高性能。

**如何选型：**

什么时候应该选择gRPC而不是Thrift：

- 需要良好的文档、示例
- 喜欢、习惯HTTP/2、ProtoBuf
- 对网络传输带宽敏感

什么时候应该选择Thrift而不是gRPC：

- 需要在非常多的语言间进行数据交换
- 对CPU敏感
- 协议层、传输层有多种控制要求
- 需要稳定的版本
- 不需要良好的文档和示例





# 二、数据库存储系统

> MySQL大家的设计大同小异，反倒是分布式的Redis，各个公司有自己的设计。
>
> 在2015 年， Redis 官方正式发布了集群版本 Redis Cluster。所以，紧跟社区步伐，并结合内部需求做了很多开发工作，演进出了全内存、高吞吐、低延迟的 KV 存储 Squirrel。另外，基于 Tair，还加入了很多自研的功能，演进出持久化、大容量、数据高可靠的 KV 存储 Cellar 。

MySQL已经是非常成熟的框架了，开发了很多代，现在所有的公司基本都是使用得这一套模型，从95年由Michael Widenius和David Axmark创建提供基本的数据存储和检索功能，至今演变出了**MySQL5.X**，和**MySQL8.X版本**这两个非常稳定的版本。

- MySQL 3.0（1997年）：MySQL 3.0版本引入了重要的特性，如支持多表联接（JOIN）和子查询（Subquery），这使得MySQL的查询功能得到了显著增强。
- MySQL 4.0（2001年）：MySQL 4.0版本带来了更多的功能改进，包括支持存储过程（Stored Procedures）、触发器（Triggers）、视图（Views）和游标（Cursors）。这使得MySQL可以更好地用于复杂的业务逻辑和应用程序开发。
- MySQL 5.0（2003年）：MySQL 5.0版本是一个重要的里程碑，它引入了许多新特性，如支持ACID事务、存储过程和触发器的完整实现，以及更好的性能优化。此版本还引入了InnoDB存储引擎作为默认存储引擎，提供更高级的事务支持和数据完整性。
- MySQL 5.1（2008年）：MySQL 5.1版本继续改进和优化了MySQL的性能和功能。它包含一些重要的新特性，如分区表（Partitioning）和Row-based Replication。
- MySQL 5.5（2010年）：MySQL 5.5版本进一步改进了性能和稳定性，并引入了半同步复制（Semi-Synchronous Replication）和存储过程的性能优化。
- MySQL 5.6（2013年）：MySQL 5.6版本带来了更多的性能和功能改进，包括全文搜索（Full-text Search）的增强、InnoDB存储引擎的改进和新的NoSQL接口等。
- MySQL 5.7（2015年）：MySQL 5.7版本引入了JSON数据类型、多源复制（Multi-Source Replication）、在线DDL操作（Online DDL）等重要功能，进一步提高了MySQL的性能和可用性。
- MySQL 8.0（2018年）：MySQL 8.0版本是目前最新的稳定版本，它引入了许多重要的改进，如事务数据字典、更好的JSON支持、全局事务标识符（GTID）复制等。此版本还通过优化查询执行和存储引擎等方面进一步提高了性能和扩展性。



**Squirrel，Cellar其实是两个不同设计目的的产物，**在实际应用上，如果业务的数据量小，对延迟敏感，建议用 Squirrel ；如果数据量大，对延迟不是特别敏感，建议用成本更低的 Cellar 。

- Squirrel是基于Redis-Cluster模式进行二次开发的大型分布式缓存系统。
- Cellar通过完全的分布式设计，并结合内存+固态硬盘的混合存储引擎，致力于为业务提供大容量、高可靠、高性能的KV存储服务


**适合使用Cellar的场景：**

- 不能容忍数据丢失
- 数据量大，内存放不下的服务

**不适合使用Cellar的场景：**

- 使用复杂数据结构（map/set），map/set中元素很多（1000以上）

**适合使用Squirrel的场景：**

- 需要使用复杂数据结构（map, set），map/set中元素很多（1000以上）
- 延迟敏感服务

不适合Squirrel的场景：

-   数据量超过100GB（数据太多，全内存太浪费资源）

## 1、数据模型

> 如何涉及到分布式存储系统，我们习惯问的问题：**where is my data！？？？数据是如何分布的？**这个问题在 KV 存储领域，就是 **Key 是怎么分布到存储节点上的。**

我们来看一下，这两个系统的数据模型：

![image-20230724000717457](https://ask.qcloudimg.com/http-save/7469656/usm4gkp0p2.png)

Squirrel 跟 Cellar 是一样的。当组件拿到一个 Key 后，用固定的哈希算法拿到一个哈希值，然后将哈希值对 Slot 数目取模得到一个Slot id，

两个 KV 现在都是预分片16384个 Slot 。得到 Slot id 之后，再根据路由表就能查到这个 Slot 存储在哪个存储节点上。这个路由表简单来说就是一个 Slot 到存储节点的对照表。

## 2、数据容灾

> 解决这个问题之后，需要追问一句，如果出现了节点故障，客户端的访问有什么变化？对于 Redis 集群而言，节点宕机已经有完备的处理机制了。

![Squirrel 架构](https://ask.qcloudimg.com/http-save/7469656/rsecgxqs4.png)

Squirrel 架构中间部分跟 Redis 官方集群是一致的。它有主从的结构， Redis 实例之间通过 Gossip 协议去通信。在右边多了一个**集群调度平台，**包含调度服务、扩缩容服务和高可用服务等，它会去管理整个集群，把管理结果作为元数据更新到 ZooKeeper。客户端会订阅 ZooKeeper 上的元数据变更，实时获取到集群的拓扑状态，直接在 Redis 集群进行读写操作。

不管是网络抖动，还是发生了宕机（比如说 Redis 2 ），它可以实时更新 ZooKeeper ，告诉 ZooKeeper 去摘除 Redis 2 ，客户端收到消息后，读流量就直接路由到 Redis 3上。

- 如果 Redis 2 只是几十秒的网络抖动，过几十秒之后，如果 HA 节点监控到它恢复后，会把它重新加回
- 如果过了一段时间，HA 判断它属于一个永久性的宕机，HA 节点会直接从 Kubernetes 集群申请一个新的 Redis 4 容器实例，把它加到集群里



**Cellar通过多个副本来实现容灾：**

- Cellar保障同一个Bucket的不同副本分布在不同的物理设备上。当某台DataServer发生故障时，ConfigServer就是失去这台DataServer的心跳，随后重新构建路由表，DataServer根据路由表来进行补副本操作，并依靠没有故障的副本为Client提供无间断的服务
- 根据同一个Bucket的不同副本分布的机房不同，Cellar单个集群的部署可以分为跨机房容灾和非跨机房容灾的模式，目前同地域的机房默认都是跨机房容灾的，即使单个机房故障也可以保障可用性。

Cellar也提供异地双机房部署的方式，默认有北京、上海数据同步的业务都是异地双机房的（双机房部署也支持set化）。





## 3、数据迁移

> 问一句，如果出现了节点故障，它的数据去哪了？？特别是有状态服务，怎么解决节点宕机后，
>
> - 数据完整性和一致性？
> - 迁移时间窗口？（对于有状态服务，迁移期间可能需要暂停或减少服务的访问。选择合适的时间窗口可以减少对用户的影响，怎么迁移更快？（就近原则））
> - 数据验证和测试？（比较源和目标环境的数据来实现，确保数据的一致性）
> - 依赖关系和服务联动？
> - **回滚计划**？（持久化重构：减少全量重传；减少并控制RDB；减少AOF写盘抖动；）
> - 迁移后续工作？（如更新服务配置、重定向域名或IP地址、更新DNS记录等，以确保所有用户和客户端都能正确连接到新环境）

**Squirrel 智能迁移**：

![image-20230724003637138](https://ask.qcloudimg.com/http-save/7469656/6qwrfwrhyq.png)

核心是“就近原则”，比如说同机房的两个节点做迁移肯定比跨机房的两个节点快。迁移任务生成之后，会把任务下发到一批迁移机上。迁移机迁移的时候，有这样几个特点：

1. 会在集群内迁出节点间做并发，比如同时给 Redis 1、Redis 3 下发迁移命令。
2. 每个 Migrate 命令会迁移一批 Key。
3. Squirrel会用监控服务去实时采集客户端的成功率、耗时，服务端的负载、QPS 等，之后把这个状态反馈到迁移机上。迁移数据的过程就类似 TCP 慢启动的过程，它会把速度一直往上加，若出现请求成功率下降等情况，它的速度就会降低，最终迁移速度会在动态平衡中稳定下来，这样就达到了最快速的迁移，同时又尽可能小地影响业务的正常请求。

接下来，看一下大 Value 的迁移，Squirrel实现了一个异步 Migrate 命令，该命令执行时，Redis 的主线程会继续处理其他的正常请求。如果此时有对正在迁移 Key 的写请求过来，Redis 会直接返回错误。这样最大限度保证了业务请求的正常处理，同时又不会阻塞主线程。





为了防止临时的节点故障而导致大量数据迁移，Cellar引入了<font color='red'>**fault handoff机制**</font>。

- 如果ConfigServer接收不到某个DataServer的心跳了，并不会立刻进行数据迁移，而是构建一个handoff路由表，相应的DataServer仅仅把流量切到没有故障的副本上，并且在没有故障的副本上记录增量log
- 如果在一定时间内（默认24小时），故障的DataServer恢复了，会进行resume handoff，把记录的增量log再迁移到恢复的DataServer上
- 如果超过24小时还没有恢复，ConfigServer再进行路由表重建，并迁移数据(一般24小时以内就进行人工处理了，把故障的设备下掉或者替换个正常的设备)

类似于故障handoff，Cellar为了线上无损运维、升级引入了manual handoff机制。即可以人为的把某台DataServer的流量切走，等运维、升级进行完毕，类似于fault handoff补增量log，再把流量切回来。



> 做好了节点容灾以及跨地域容灾后，业务又对服务提出了更高要求：强一致存储。
>
> 之前的数据复制是异步的，在做故障摘除时，可能因为故障节点数据还没复制出来，导致数据丢失。但是对于金融支付等场景来说，它们是不容许数据丢失的。
>
> 有状态服务的数据迁移该怎么解决？目前业界主流的解决方案是基于 Paxos 或 Raft 协议的强一致复制。Cellar最终选择了 [Raft 协议](https://juejin.cn/post/7143541597165060109)。



## 4、热点Key

> 热点key问题，其实是在大规模并发访问下，某些特定的key（通常是缓存或数据库中的数据索引）被频繁地访问，导致该key所在的存储节点成为系统的**性能瓶颈**，从而影响整个系统的性能和可伸缩性。
>
> 热点key问题可能导致请求堆积、延迟增加和服务不稳定（缓存击穿的风险）等现象。（**感兴趣的朋友可以取了解一下，解决Redis缓存雪崩、穿透、并发、预热、*击穿*、*热点Key*等问题**）
>
> 比较常用的策略是：
>
> - 缓存预热：在系统启动或访问量较小时，提前将部分热门数据加载到缓存中，减少热点数据访问时的延迟
> - 缓存降级：当热点key访问过于频繁时，可以采取缓存降级策略，暂时关闭或限制该key的缓存，避免造成更大的影响
> - 缓存策略优化：选择合适的缓存策略，如LRU（Least Recently Used）或LFU（Least Frequently Used），以更好地适应热点key的访问模式

对于Squirrel，有如下设计：

1. 有请求进来读写普通节点时，节点内会同时做请求 Key 的统计
   - 如果某个 Key 达到了一定的访问量或者带宽的占用量，会自动触发流控以限制热点 Key 访问，防止节点被热点请求打满
2. 同时，监控服务会周期性的去所有 Redis 实例上查询统计到的热点 Key
   - 如果有热点，监控服务会把热点 Key 所在 Slot 上报到Squirrel的迁移服务
   - 迁移服务这时会把热点主从节点加入到这个集群中，然后把热点 Slot 迁移到这个热点主从上
   - 因为热点主从上只有热点 Slot 的请求，所以热点 Key的处理能力得到了大幅提升

![Squirrel热点key](https://ask.qcloudimg.com/http-save/7469656/07bgg7xp00.png)

通过这样的设计，Squirrel可以做到实时的热点监控，并及时通过流控去止损；通过热点迁移，Squirrel能做到自动的热点隔离和快速的容量扩充。（核心就是，**“单人单列，特事特办”**）

Cellar 热点 Key 解决方案有一点差别，每个存储节点右普通数据区域和热点数据区域；中心节点加了一个职责，多了热点区域管理，它现在不只负责正常的数据副本分布，还要管理热点数据的分布

![Cellar 热点 Key 解决方案](https://ask.qcloudimg.com/http-save/7469656/bwshfagpkv.png)

1. 如果这个 Key 是一个热点，那么它会在做集群内复制的同时，还会把这个数据复制有热点区域的节点，也就是图中的 C、D 节点
2. 同时，存储节点在返回结果给客户端时，会告诉客户端，这个 Key 是热点，这时客户端内会缓存这个热点 Key
3. 当客户端有这个 Key 的读请求时，它就会直接去热点区域做数据的读取

通过这样的方式，Cellar可以做到只对热点数据做扩容，**不像 Squirrel ，要把整个 Slot 迁出来做扩容**。

有必要的话，中心节点也可以把热点区域放到集群的所有节点上，所有的热点读请求就能均衡的分到所有节点上（数据分布更加地均衡）。另外，通过这种实时的热点数据复制，Cellar很好地解决了类似客户端缓存热点 KV 方案造成的一致性问题。

# 三、消息中间件

> 消息队列已经逐渐成为企业IT系统内部通信的核心手段。它具有低耦合、可靠投递、广播、流量控制、最终一致性等一系列功能，成为异步RPC的主要手段之一。 当今市面上有很多主流的消息中间件，如老牌的ActiveMQ、RabbitMQ，炙手可热的Kafka，阿里巴巴自主开发的Notify、MetaQ、RocketMQ等。
>
> 使用mq的场景有很多，最常用的几种，是做业务解耦/最终一致性/广播/错峰流控等。反之，如果需要强一致性，关注业务逻辑的处理结果，则RPC（直接通信）显得更为合适。



设计消息中间件时需要关注以下几个关键点：

1. 可靠性和持久化： 消息中间件应该能够保证消息的可靠传递，即使在发生故障或意外情况时也不丢失消息。持久化是实现可靠性的关键，消息应该被持久化存储在中间件中，直到成功被消费者处理。
2. 高性能和低延迟： 消息中间件需要具备高性能和低延迟的特性，以应对高并发和大规模的消息处理。这包括优化消息的传输、存储和消费过程，以提供快速的消息传递和响应。
3. 消息分发和路由： 中间件需要支持灵活的消息分发和路由机制，以便根据不同的业务场景将消息传递给正确的消费者。这可以通过主题（Topic）、队列（Queue）等方式来实现。
4. 可伸缩性： 中间件应该具备良好的可伸缩性，能够应对不断增长的消息负载。这可以通过水平扩展和负载均衡等手段来实现。
5. 安全性： 消息中间件应该提供安全机制，确保消息在传输和存储过程中的机密性和完整性。这可能包括身份验证、授权、加密等功能。
6. 监控和管理： 中间件需要提供全面的监控和管理功能，方便运维人员实时监控中间件的状态和性能指标，以及进行故障排查和性能调优。
7. 支持多种协议和编程语言： 消息中间件应该支持多种协议（如AMQP、MQTT、STOMP等）和编程语言（如Java、Python、Go等），以方便不同系统和平台之间的通信。
8. 顺序性保证： 在某些业务场景下，消息的顺序性是非常重要的，中间件需要能够保证消息在传递和消费过程中的顺序性。
9. 高可用性和容错性： 消息中间件应该具备高可用性和容错性，即使在部分节点发生故障时也能继续提供服务，不影响整个系统的稳定性和可用性。



**<font color='red'>消息中间件可以实现消息的解耦，确保消息的一致性传递，实现消息的广播效果，进行消息错峰与流控，从而提高系统的性能、可靠性和可伸缩性：</font>**

1. 消息解耦：
   - 发布-订阅模式：消息中间件采用发布-订阅模式，生产者将消息发布到主题（Topic）中，而消费者可以订阅感兴趣的主题，从而实现生产者与消费者之间的解耦。生产者不需要知道具体的消费者，只需要发布消息到主题即可，消息中间件会负责将消息传递给订阅了该主题的消费者。
2. 消息一致性：
   - 事务消息：消息中间件支持事务消息，允许生产者在发送消息时，将消息和业务数据放在一个事务中，要么同时成功，要么同时失败。消费者也可以使用事务来确认消息的消费，确保消息的可靠传递和消费。
3. 消息广播：
   - 主题广播：消息中间件通过主题广播功能，将一个消息发送给多个消费者，实现消息的广播效果。这样可以确保所有订阅了主题的消费者都能接收到相同的消息。
4. 消息错峰与流控：
   - 限流策略：消息中间件可以通过限流策略来控制消息的流量，防止由于高并发访问导致系统崩溃。例如，可以设置每秒最大处理消息数量，或者根据系统负载动态调整限流策略，确保消息的稳定处理。
5. 消息重试和延时重试：
   - 消息中间件支持消息重试机制，当消息消费失败时，可以根据配置进行自动重试。同时，还可以设置延时重试策略，使得消息在一定时间后再次进行重试，避免瞬时故障影响整个系统。



**常用的消息中间件选型：**

Kafka适用场景：

- 高吞吐量：Kafka的设计目标之一是实现高吞吐量的消息传递，适用于需要处理大量数据流的场景，如日志收集、实时数据处理等。
- 数据持久性：Kafka采用持久性存储，可保证消息不丢失，适用于需要高可靠性的消息传递场景。
- 分布式系统：Kafka具有良好的分布式特性，支持多个节点构成的集群，适用于大规模、分布式的系统架构。
- 流式处理：Kafka提供了流处理的支持，可以与流处理框架（如Apache Flink、Spark Streaming等）结合使用，实现实时数据处理和分析。

RocketMQ适用场景：

- 顺序消息：RocketMQ可以保证单个队列内消息的有序性，适用于需要严格保持消息顺序的场景，如订单处理等。
- 事务消息：RocketMQ从早期就支持事务消息，适用于需要确保消息的事务性和一致性的场景。
- 大规模部署：RocketMQ的设计目标之一是支持大规模部署，适用于高并发、大规模消息传递场景。
- 多租户：RocketMQ支持多租户的部署，不同租户可以使用独立的命名空间和资源，适用于云平台和多用户场景。

Apache Pulsar适用场景：

- 特点：分层架构、高性能、低延迟，适用于大规模数据流处理和持久化。
- 适用场景：实时数据处理、事件处理、消息持久化等大数据场景。
