# cpu 告警

说两个遇到关注这两个系统性能指标的场景：

1. 对系统压测，进行系统评估的时候cpu.idle和cpu.load是两个主要参考指标
2. 服务正常运行时，会配置host指标告警；近期某个线上服务晚上9点左右，会有周期性的告警；本质上都是cpu服务不过来，单机服务压力大，具体可能是这么几种情况：
   1. 上游数据激增
   2. 代码问题，部分逻辑没有做好，导致如果任务的执行时间非常长（处理流程耗时较多，计算复杂度高）
   3. mafka消息队列的分区少，都让这台机器处理

## 一、什么是cpu利用率——cpu.idle和cpu.load的区别

> 在Linux系统中，通过top命令可以查看cpu.idle和cpu.load。

结论先行：

1. cpu.idle表示系统的空闲进程占比，这个<font color='red'>**越高**</font>说明cpu压力越小
2. cpu.load表示系统的额负载，这个<font color='red'>**越低**</font>越好！



## 1、进程切换的逻辑

![进程切换](https://pic2.zhimg.com/80/v2-d831951a5e41bbfb1e0e1151a8a2b649_720w.webp)

在Linux内核中，每个进程都会被分配一个固定的时间片，默认为10ms，在这10ms中，该进程享有cpu的所有权。10ms看上去很短，但以2.6GHz的Intel处理器为例，10ms能够处理5000w条指令，对于绝大多数的应用这已经足够长了。

![img](https://pic3.zhimg.com/80/v2-440bb1699b2fa0f0340b38eabcbd7452_720w.webp)

如果该进程用完了10ms,或者有其他优先级高的进程发出请求，系统会触发一个中断，内核重新接管cpu，内核分配cpu给其他进程（所以，**一次系统调用的过程，其实是发生了两次 CPU 上下文切换**。（用户态-内核态-用户态））。10ms的分片让用户，也就是我们觉得我们的系统运转非常流畅，尽管我们可能同时开了很多的应用。



## 2、cpu.idle空闲进程比（非利用率）

> 空闲进程的主要目的是**消除操作系统调度程序可能遇到的特殊情况**(并不是事情都不做！)，让任何时候调度器都能找到一个处于“就绪”状态的线程。
>
> 在Windows系统中，系统空闲进程包含一个或多个的线程，这些线程始终处于最低优先级以及“就绪”状态，每个逻辑内核都有一个对应的线程。当操作系统发现没有高优先级的线程需要运行时，就会切换到空闲线程，以维持调度算法的一致性。
>
> 

CPU利用率主要分为用户态，系统态和空闲态，分别表示CPU处于用户态执行的时间，系统内核执行的时间，和空闲系统进程执行的时间，三者之和就是CPU的总时间，当没有用户进程、系统进程等需要执行的时候，CPU就执行系统缺省的空闲进程。

CPU的利用率就是指非空闲进程占用时间的比例，即CPU执行非空闲进程的时间 / CPU总的执行时间。cpu.idle是基于/proc/stat计算出来的。因为/proc/stat中的数值都是从系统启动开始累计到当前时刻的积累值，所以需要在不同时间点t1和t2取值进行比较运算，当两个时间点的间隔较短时，就可以把这个计算结果看作是CPU的即时利用率。

### 2.1 cpu.idle过低排查方式

1. top 或者 jps 或者 ps -ef | grep java,假定我们找到的是java 进程 PID=9527
2. 使用命令：top -H -p 9527，找到java进程中的线程情况
3. 找到占用cpu较高的排名靠前的几个线程 PID 假如找到的耗CPU较高的线程 PID=95271
4. 查看该线程的堆栈信息，使用命令：jstack -l 95271 或者 jstack -F 95271
5. 分析堆栈的信息，进一步进行排查代码逻辑



## 3、cpu.load(系统平均负载,通过top、uptime、和falcon可以查看)

系统平均负载被定义为在特定时间间隔内运行队列中(在CPU上运行或者等待运行多少进程)的平均进程数。如果一个进程满足以下条件则其就会位于运行队列中：

- 它没有在等待I/O操作的结果
- 它没有主动进入等待状态(也就是没有调用’wait’)
- 没有被停止(例如：等待终止)

在Linux中，进程分为三种状态，一种是阻塞的进程blocked process，一种是可运行的进程runnable process，另外就是正在运行的进程running process。

进程可运行状态时，它处在一个运行队列run queue中，与其他可运行进程争夺CPU时间。 系统的load是指正在运行和准备好运行的进程的总数。比如现在系统有2个正在运行的进程，3个可运行进程，那么系统的load就是5（2 + 3）。

### 3.1 cpu.load大于多少没有性能够问题呢？

理想情况下cpu不空闲，但每个线程又都能及时获得cpu时间，没有等待的线程，任务处理完，紧接着出现另外一个可运行线程来获得cpu时间，此时cpu.load/cores=1。如果cpu.load过大则表示，有部分线程在等待获得cpu资源，过小表示cpu资源比较空闲。

对于cpu.load多少开始出现性能问题，外界有不同的说法：

1. 有的认为cpu.load/cores最好不要超过1
2. 有的认为cpu.load/cores最好不要超过3
3. 有的认为cpu.load不超过2*cores-2即可

**cpu.load太高会影响接口响应时间。**进行压测时，发现load低时，接口TP999响应时间<50ms，而cpu.load/cores>4时，TP999响应时间约为200ms左右，多出来的时间可以理解为线程等待cpu处理的时间，可见cpu.load/cores过高时是影响接口响应时间的。

看了一下这个函数的内部实现，发现他使用线程池 `asynThreadPool` 提交一个任务，任务中调用 `syncMainProcess` 方法同步POI信息。

```
private static ThreadPoolExecutor asynThreadPool = new ThreadPoolExecutor(
        10,
        20,
        30, TimeUnit.SECONDS,
        new ArrayBlockingQueue&lt;Runnable>(1000),
        new ThreadPoolExecutor.CallerRunsPolicy());
```

这段代码是创建一个线程池 `asynThreadPool`，用于异步执行任务。线程池的参数配置如下：

1. `corePoolSize`：核心线程数为10，即线程池中始终存活的线程数。
2. `maximumPoolSize`：最大线程数为20，即线程池中允许的最大线程数。
3. `keepAliveTime`：非核心线程的空闲存活时间为30秒，如果超过这个时间，非核心线程会被销毁。
4. `unit`：空闲存活时间的单位为秒。
5. `workQueue`：工作队列为一个容量为1000的阻塞队列，用于存储等待执行的任务。
6. `rejectedExecutionHandler`：拒绝策略为`CallerRunsPolicy`，当线程池和工作队列都满了，新提交的任务会由调用线程自己执行。

这个线程池的配置适用于处理中等数量的异步任务，**如果任务数量过大，可能会导致工作队列满，新提交的任务由调用线程执行**，影响调用线程的性能。

### 4.1 使用线程池 `asynThreadPool` 可能存在以下风险：

1. **内存溢出**：如果任务提交速度远大于处理速度，工作队列可能会迅速填满，导致大量任务堆积，消耗大量内存，最终可能导致内存溢出。
2. **系统资源耗尽**：线程池中的线程数可能会占用过多的系统资源，尤其是在多线程环境下，可能会导致CPU过载或上下文切换过于频繁，影响系统性能。
3. **性能瓶颈**：`CallerRunsPolicy` 拒绝策略可能会导致调用线程直接执行任务，如果调用线程本身还有其他重要的任务要处理，这可能会成为性能瓶颈。
4. **任务执行延迟**：当线程池中的所有线程都在忙碌时，新的任务只能在队列中等待，这可能会导致任务执行延迟。
5. ~~**错误处理**：如果任务执行过程中抛出未捕获的异常，可能会导致线程终止，如果没有适当的错误处理机制，这可能会导致线程泄漏。~~
6. **线程泄漏**：如果任务长时间运行或者无法完成，可能会导致线程一直被占用，无法处理新的任务，从而导致线程泄漏。

为了降低这些风险，可以采取以下措施：

- 对提交的任务进行监控和控制，确保任务的产生速度和处理速度相匹配。
- 根据系统资源和实际需要合理配置线程池的大小。
- 实现适当的错误处理和异常捕获机制，确保线程的稳定运行。
- 定期进行性能测试，以确保线程池配置能够满足生产环境的需求。



#### 4.1.1 为什么要使用CallerRunsPolicy策略？

`CallerRunsPolicy` 是 `ThreadPoolExecutor` 提供的一种拒绝策略。当线程池的所有运行线程都被占用（即，**线程池满**），且**工作队列已满**时，如果还有新任务提交，这些新任务将不会被放入线程池中，也不会被放入队列中等待。相反，`CallerRunsPolicy` 策略会让提交任务的线程（即**调用者线程**）自己执行这个任务。

使用 `CallerRunsPolicy` 的原因可能包括：

1. **降低任务丢失风险**：相比于直接丢弃任务的拒绝策略，`CallerRunsPolicy` 可以保证每个任务都会被执行，不会发生丢失。
2. **反压调用者**：当线程池无法处理更多任务时，`CallerRunsPolicy` 通过让调用者线程执行任务，从而减缓新任务的提交速度，这种方式可以起到一定的反压作用，防止线程池被过度消耗。
3. **简化错误处理**：使用 `CallerRunsPolicy` 可以避免处理任务被拒绝的复杂逻辑，因为这种策略不会抛出 `RejectedExecutionException` 异常。

然而，这种配置也有其缺点，尤其是当调用者线程本身有其他重要任务或者是时间敏感的操作时，这种策略可能会导致调用者线程的性能下降，因为它必须等待任务执行完成。因此，是否采用 `CallerRunsPolicy` 需要根据具体的应用场景和性能要求来决定。





### 4.2 线程池参数rejectedExecutionHandler

`ThreadPoolExecutor` 提供了四种拒绝策略（`RejectedExecutionHandler`），当任务无法被线程池处理时，这些策略定义了不同的处理方式：

1. **AbortPolicy**：
   - **定义**：这是默认的拒绝策略。当任务无法处理时，会抛出一个 `RejectedExecutionException` 异常。
   - **优点**：简单明了，能够立即知道线程池无法接受任务，避免了系统资源的过度消耗。
   - **缺点**：可能导致提交的任务丢失，除非调用者捕获异常并处理。抛出异常可能导致上层服务不稳定，需要额外的错误处理逻辑。

2. **CallerRunsPolicy**：
   - **定义**：当任务无法处理时，会让提交任务的线程自己执行该任务。
   - **优点**：提供了反压机制，减缓任务提交速度，避免了任务丢失。不需要额外的资源，因为任务在调用者线程中执行。
   - **缺点**：可能会影响调用者线程的性能，特别是当调用者线程本身有其他重要任务时。如果所有的调用者线程都很忙，可能会导致整个系统的响应速度变慢。

3. **DiscardPolicy**：
   - **定义**：当任务无法处理时，不做任何处理，直接丢弃任务。
   - **优点**：系统最简单的处理方式，不会抛出异常，也不会执行任务。避免了因为异常处理导致的系统复杂性。
   - **缺点**：任务会被无声地丢弃，调用者无法得知，可能会丢失重要的任务。

4. **DiscardOldestPolicy**：
   - **定义**：当任务无法处理时，会丢弃工作队列中最老的一个任务，然后尝试提交当前任务。
   - **优点**：尝试为新任务腾出空间，使得新任务有机会执行。相比于 `DiscardPolicy`，它至少尝试了一种补救措施。
   - **缺点**：会丢弃队列中最老的任务，如果这个任务很重要，那么可能会导致问题。同样可能导致任务丢失，且调用者不会得到任何通知。

选择哪种拒绝策略应根据具体的业务需求和系统的容错性要求来决定。例如，如果任务不允许丢失，可以选择 `CallerRunsPolicy` 或者自定义拒绝策略来确保任务一定会被执行。如果系统对任务的实时性要求很高，可能需要使用 `AbortPolicy` 并配合适当的异常处理机制来确保系统的响应性。在某些情况下，可能需要实现自定义的 `RejectedExecutionHandler` 来满足特定的业务需求。



## 二、如何利用火焰图（flame graph）来分析异常

火焰图是基于 perf 结果产生的 SVG 图片，用来展示 CPU 的调用栈。

y 轴表示调用栈，每一层都是一个函数。调用栈越深，火焰就越高，顶部就是正在执行的函数，下方都是它的父函数。

x 轴表示抽样数，如果一个函数在 x 轴占据的宽度越宽，就表示它被抽到的次数多，即执行的时间长。注意，x 轴不代表时间，而是所有的调用栈合并后，按字母顺序排列的。

**火焰图就是看顶层的哪个函数占据的宽度最大。只要有"平顶"（plateaus），就表示该函数可能存在性能问题。**
