# 	一句话讲清RAG vs Memory

> 2025年开始RAG太火了，arXiv上面关于rag的论文从24年的967篇，飙升到25年的2786篇，年初的时候,GitHub上RAG相关的开源项目有35个之多。但是事实上，RAG（Retrieval Augmented Generation，检索增强生成）并不是一个新技术，只不过当年LLM面临一个很大的问题是，他的模型知识永远是基于训练那一刻之前的，新数据拿不到这一点太“致命”。
>
> 大模型 1.0 时代（2022-2023）
> ├─ 问题：模型能力强，但数据陈旧、无法定制
> └─ 结果：企业观望，不敢大规模商用
>
> RAG + Memory 2.0 时代（2024-2025）
> ├─ 问题被解决：实时数据 + 个性化 + 私有数据保护
> ├─ 应用场景爆发：客服、内部工具、行业应用
> └─ 结果：AI 产品从"玩具"变成"生产力工具"

# 一、一句话总结

**核心区别**：RAG 是"问什么查什么"，Memory 是"记住你说过什么"。

**RAG 火起来的根本原因：**

1. **大模型的"天花板"问题**
   - ChatGPT/Claude 训练数据有截止时间，无法实时更新
   - 企业发现"我花钱用大模型，却回答不了我最新的产品信息"
   - 这直接影响商业价值，催生了对 RAG 的需求
2. **隐私和成本压力**
   - 企业不敢把敏感数据喂给云端大模型训练（合规风险、竞争优势泄露）
   - RAG 让企业能用大模型能力，同时保护自己的私有数据
   - 比重新训练模型便宜 100 倍
3. **垂直应用爆发**
   - 医疗、法律、金融这些高准确度行业需要"有出处的答案"
   - RAG 提供可追溯的来源，符合监管和审计需求

**Memory 火起来的根本原因：**

1. **用户体验预期升级**
   - 用户已经习惯了 Siri/Alexa 记住他们的习惯
   - 每次都要重新解释背景信息，用户体验糟糕
   - 这变成了产品竞争力的必需项
2. **付费用户的粘性需求**
   - 免费用户用完就走，付费用户需要持续价值
   - Memory 让 AI 与用户的关系变深，形成"个人助手"的感觉
   - ChatGPT Plus、Claude Pro 都强调这一点
3. **长上下文能力的成熟**
   - 早期模型上下文窗口小（4K、8K），记不了什么
   - Claude 200K、GPT-4 Turbo 128K，现在能记住足够的对话历史 
   - 技术能力成熟了，成本下来了，商业需求也出现了

| 维度         | RAG                          | Memory                 |
| ------------ | ---------------------------- | ---------------------- |
| **数据来源** | 外部知识库（文档、数据库等） | 对话历史、上下文       |
| **存储方式** | 向量索引、语义搜索           | 对话记录、摘要         |
| **使用时机** | 每次查询时动态检索           | 跨轮次复用积累的信息   |
| **应用场景** | 回答特定领域问题、事实查询   | 上下文连贯、个性化交互 |
| **更新频率** | 知识库更新时迭代             | 每轮对话动态更新       |

# 二、什么是RAG？

## 1、RAG其实就是让模型知道额外的检索信息

**RAG的演进：**

RAG 1.0（2023）: 拼命优化检索算法
RAG 2.0（2024）: 怎么组织上下文让模型用好

例如：
- 要不要包含相似的反例？
- 要不要加入检索相关度分数？
- 怎么排序检索结果？（按时间/相关度/用户偏好？）
- 怎么写 prompt 让模型知道信息来自哪里？



一个常见的RAG场景：

```
场景：用户问"我的订单什么时候到？"

❌ 表面 RAG 思路：
1. 检索 → 找到订单信息 ✓
2. 生成 → AI 回答
问题：AI 拿到 100 条订单记录，却混淆了哪个是用户的

✅ 本质上下文工程思路：
1. 检索 → 找到订单信息
2. 构建上下文 → 精心组织提示词：
   "用户ID: xxx
   当前订单（最新）: xxx
   请只关注这一条订单，回答..."
3. 生成 → AI 精准回答
```

1. **真实的性能瓶颈在这里**
   - 一个糟糕的 prompt 工程，检索完美也白搭
   - 好的 prompt 工程，即使检索有噪音也能work
   - 业界经验：70% 的 RAG 性能问题源自 prompt，只有 30% 源自检索

RAG 的核心能力其实是 **"让模型在正确的上下文里做决策"**，而不是 **"去查个信息"**。

这就像：

- ❌ 医生诊断的本质不是"查医学资料"
- ✅ 医生诊断的本质是"整合患者信息、医学知识和临床经验，做出判断"



**提升 RAG 效果的顺序：**

1. 第一阶段：优化上下文组织（50% 效果提升）
   └─ 写好 system prompt、组织检索结果格式
2. 第二阶段：优化检索算法（30% 效果提升）
   └─ 改进向量库、BM25、混合检索
3. 第三阶段：优化数据质量（20% 效果提升）
   └─ 清洗知识库、分块策略

## 2、深层理解RAG：把知识转化为模型能理解的语言

| 角度         | 表面理解                              | 本质理解                          |
| ------------ | ------------------------------------- | --------------------------------- |
| **名字**     | 检索（Retrieval）+ 生成（Generation） | 上下文工程（Context Engineering） |
| **重点**     | 怎么检索信息                          | 怎么把信息组织成模型能理解的形式  |
| **成败因素** | 检索算法准不准                        | 提示词写得好不好                  |
| **难点**     | 找对信息                              | 让模型用好信息                    |

一个能干活的Agent,需要精心组装三类上下文:

**1. 领域知识(Knowledge)**

- 企业内部文档、产品手册、历史案例
- 这就是传统RAG的强项

**2. 工具描述(Tools)**

- API文档、函数说明、调用示例
- 当工具数量>100个,如何选择成为大问题

**3. 交互历史(Memory)**

- 对话历史、用户偏好、任务状态
- 本质也是一种检索问题

**这三类数据的管理,本质上都是检索问题**。RAG的技术栈(向量索引、混合检索、重排序)可以完美复用。



最有效的 RAG 不是"更好的检索"，而是：

1. 检索 → 找到相关文档
2. 重排/重写 → 用 LLM 把文档改写成"最适合回答这个问题的形式"
3. 融入上下文 → 精心设计 prompt 让模型聚焦
4. 生成 → 模型回答

这里第 2 步（LLM 改写文档）比传统的"精准检索"更重要。

不是"我去查了个信息"，而是"我用模型的语言重新诠释了知识库，让它能自己理解"。

<font color='red'>这就是为什么 Claude、GPT-4 这样的大模型在 RAG 上表现这么好 —— 不是因为检索好，而是因为它们**能用自然语言理解复杂的上下文**，给它足够清晰的 prompt，它就能自己做出判断。</font>

所以说，<font color='red'>**RAG不会死**,而是会从"应用"变成"基础设施"。 就像数据库之于应用开发,RAG(或者说Context Engine)会成为所有AI应用的标配。</font>



# 三、memory的做工之处

>  **Memory就是特殊的RAG**，这两者的核心都一样，检索增强。
>
> - **数据来源不同** - RAG处理静态文档,Memory处理动态对话
> - **技术栈相同** - 都是存储、索引、检索
> - **目标互补** - 一个提供领域知识,一个提供个性化上下文
>
> **2025年LLM的演进特点是不是由模型规模或更多GPU驱动，而是由记忆驱动**。因为现在光堆参数提升的效果疫情不想去年那样那么明显了，最节省资源和由用户感知的是让模型真正"记住"用户。

## 1、memory解决了什么问题？

**表面理解 vs 本质理解：**

表面理解：Memory = 存储对话历史，下次用时调出来

本质理解：Memory = **"用户在系统中的身份演化过程"**

**三个递进的理解层次：**

**第一层：存储（Storage）**

```
简单记录：
用户说过什么 → 存起来 → 下次拿出来
问题：只是重放，没有理解
```

**第二层：提炼（Extraction）**

```
不是记录对话原文，而是提炼关键信息：

用户原话："我最近在忙一个电商系统的性能优化"
↓ 提炼
关键信息：{用户_职业: "工程师", 关注领域: "性能优化", 行业: "电商"}

用户说："我讨厌 MySQL 的慢查询"
↓ 提炼
关键信息：{用户_痛点: "SQL性能", 态度: "强烈不满"}
```

**第三层：身份塑造（Identity Shaping）**

```
本质上，Memory 是在回答三个问题：

1. "这个用户是谁？"
   → 职位、行业、技能、性格
   
2. "用户的世界观是什么？"
   → 价值观、偏好、约束条件
   
3. "用户要去哪里？"
   → 目标、痛点、发展方向

每次交互都在**更新**这个用户模型
```

**更深的理解：Memory 其实在做"关系演化"**

```
对话 1：
用户："我是新手，怎么学 Python？"
AI 记住：这是初学者
↓
对话 2：
用户："列表推导式怎么用？"
AI 理解：不需要再解释基础了，可以用高阶例子
↓
对话 3：
用户："怎么优化 list comprehension 的性能？"
AI 识别：这个用户已经进阶了，现在关心性能
↓
对话 4：
用户："我想用并行处理..."
AI 主动推荐：基于你的发展轨迹，这里有个高级模式...

关键：AI 在追踪用户的**成长曲线**
```

**Memory 的三种形态：**

1. **短期记忆（Session Memory）**
   - 当前对话的上下文
   - 用处：保持对话连贯性
   - 技术：直接放在 prompt 里
2. **中期记忆（Conversation Memory）**
   - 最近的几次对话
   - 用处：追踪用户的思路演化
   - 技术：summarization + 检索
3. **长期记忆（User Profile Memory）**
   - 用户的持久身份属性
   - 用处：个性化、推荐、决策
   - 技术：embedding + 向量库 + 定期更新



那回到agent本身的能力分解上面来看，agent = LLM+ Planning（长短期记忆） + Feedback + Tool use（搜索、网页浏览、代码执行、文件系统、数据库）。

![img](https://pic2.zhimg.com/v2-952bd51693e9ec4342bd1771bdbdabf9_1440w.jpg)

1. 作为原始输入（包括文本、图像或其他形式）的学习嵌入表征的`感官记忆`;
2. `短期记忆`就是**上下文学习**（in-context learning），非常短且影响范围有限，受到Transformer的上下文窗口长度的限制。
3. `长期记忆`作为智能体在查询时可用的外部向量存储，可通过快速检索访问。



但是，memory也有一些悖论问题：

* **悖论 1：信息越多，反而可能表现越差**
  理由：过时信息、矛盾信息、噪音太多
  解决：需要智能的遗忘（什么该记，什么该忘）

* **悖论 2：完全依赖历史，会固化用户认知**
  理由：用户可能在变化、改变想法
  解决：需要主动更新和挑战（"你最近提到XXX，还是这么想吗？"）

* **悖论 3：隐私 vs 个性化的矛盾**
  理由：记得越多越好用，但越隐私
  解决：本地存储、用户可控

每次交互不是独立的，而是在不断地问："这个人现在是谁？他想要什么？他能理解什么程度的内容？"

这才是为什么 Memory 会成为产品竞争力的核心 —— 不是因为"记得住"，而是因为**"能通过记忆理解和陪伴用户的成长"**。所以<font color='red'>**RAG 是"知识的语言"，而 Memory 是"关系的进化"**。</font>

## 2、龙虾openClaw为什么牛逼？

> OpenClaw = Agent 框架         = Tool 调用能力         = RAG（通过工具获取实时数据）         + Memory（记住用户意图和上下文）         + 推理（怎么组织工具调用）

### 2.1 工作区文件一览

| 文件                   | 用途                                         | 加载时机                          |
| :--------------------- | :------------------------------------------- | :-------------------------------- |
| `AGENTS.md`            | 操作指令：告诉龙虾"怎么做事"和如何使用记忆   | 每次会话开始                      |
| `SOUL.md`              | 人设：性格、语气、边界                       | 每次会话开始                      |
| `USER.md`              | 用户资料：你是谁、怎么称呼你                 | 每次会话开始                      |
| `IDENTITY.md`          | 龙虾身份：名字、风格、表情符号               | 每次会话开始                      |
| `TOOLS.md`             | 工具使用备注（不控制工具开关，只是使用建议） | 每次会话开始                      |
| `HEARTBEAT.md`         | 心跳任务清单（可选，保持简短）               | 心跳运行时                        |
| `BOOT.md`              | 启动清单（可选，Gateway 重启时执行）         | Gateway 启动时                    |
| `BOOTSTRAP.md`         | 首次引导仪式（完成后删除）                   | 仅首次运行                        |
| `MEMORY.md`            | 长期记忆（可选）                             | 仅主会话                          |
| `memory/YYYY-MM-DD.md` | 每日记忆日志                                 | 今天+昨天：会话开始；其余按需读取 |
| `skills/`              | 工作区级技能（可选）                         | 按需加载                          |

> **重要**：这些文件在每次对话时都会注入到 AI 模型的上下文窗口中，**会消耗 Token**。保持文件简洁，尤其是 `MEMORY.md`——它会随时间增长，导致上下文使用量增大和更频繁的压缩。

> `memory/YYYY-MM-DD.md` 中**今天和昨天**的文件会在会话开始时自动读取，其余日期的文件通过 `memory_search` 和 `memory_get` 工具按需访问，不占用上下文窗口。

**工作区文件注入规则**

- 空文件会被跳过
- 大文件会被截断并标注[truncated]
  - 单文件上限：`agents.defaults.bootstrapMaxChars`（默认 20,000 字符）
  - 所有文件总上限：`agents.defaults.bootstrapTotalMaxChars`（默认 150,000 字符）
- 缺失的文件会注入一行 "missing file" 标记
- 子 Agent 会话只注入 `AGENTS.md` 和 `TOOLS.md`（其他文件被过滤以保持子 Agent 上下文精简）
- 截断时可注入警告，通过 `agents.defaults.bootstrapPromptTruncationWarning` 控制（`off` / `once` / `always`，默认 `once`）

运行 `/context list` 或 `/context detail` 可以查看每个文件的原始大小 vs 注入大小、是否被截断。



### 2.2 记忆系统

> 来源：https://datawhalechina.github.io/hello-claw/cn/adopt/chapter6/#_6-%E8%AE%B0%E5%BF%86%E7%B3%BB%E7%BB%9F

OpenClaw 的记忆是**纯 Markdown 文件**——文件就是唯一真相来源，模型只"记住"被写到磁盘上的内容。

| 文件                   | 用途                             |
| :--------------------- | :------------------------------- |
| `MEMORY.md`            | 精选长期记忆（每次会话加载）     |
| `memory/YYYY-MM-DD.md` | 每日日志（按需读取，不占上下文） |

写记忆的原则：说了"记住这个"，龙虾必须立即写入文件——"脑子里记着"熬不过会话重启。

搜索工具：`memory_search`（语义搜索）和 `memory_get`（读取指定文件）。

#### 2.2.1 记忆系统进阶（向量搜索、混合搜索、时间衰减、QMD 后端）

```
Layer 4: 业务应用层
         ↑
         │
Layer 3: 混合搜索层（向量 + 关键词 + 时间衰减）
         ↑
         │
Layer 2: 存储层（向量库 + QMD + 传统数据库）
         ↑
         │
Layer 1: 提炼层（从对话中提取什么该记）
```

1. 自动记忆刷新

   - 会话接近压缩上限时，OpenClaw 自动触发静默记忆刷新，提醒模型将重要信息写入磁盘

2. 向量记忆搜索

   - 默认启用，自动监控记忆文件变化，通过语义相似度搜索（即使措辞不同也能匹配）。

3. 混合搜索（BM25 + 向量）

   - 结合语义匹配和关键词匹配（精确 ID、代码符号、错误信息）

   - **MMR 去重**（减少重复结果，`lambda`：0=最大多样性，1=最大相关性，默认 0.7）：

     

     ```
     { "hybrid": { "mmr": { "enabled": true, "lambda": 0.7 } } }
     ```

     - **时间衰减**（优先近期记忆，半衰期 30 天：今天 100%、30 天前 50%、90 天前 12.5%）：

     

     ```
     { "hybrid": { "temporalDecay": { "enabled": true, "halfLifeDays": 30 } } }
     ```

   - `MEMORY.md` 和非日期命名文件不受衰减影响。

   - ```
     {
       "agents": {
         "defaults": {
           "memorySearch": {
             "query": {
               "hybrid": {
                 "enabled": true,
                 "vectorWeight": 0.7,
                 "textWeight": 0.3
               }
             }
           }
         }
       }
     }
     ```

     

4. 



**QMD (Quantum Memory Database)**是 Shopify 联合创始人兼 CEO Tobias Lütke (Tobi) 开发的本地语义搜索引擎。（<font color='red'>注意：QMD对电脑硬件CPU｜GPU是有要求的！它是个本地模型</font>）

**核心价值**：

- 不要把整个文件塞给 AI，而是先用本地搜索找到最相关的片段（通常只有 2-3 句话），再把这些精准内容传给 AI
- 解决传统记忆系统把整个 MEMORY.md 文件直接塞进上下文导致的"上下文爆炸"问题

**实际效果**：

- 📊 Token 削减：60-97%（平均 95% 以上）
- ⚡ 响应速度提升：5-50 倍
- 💰 成本降低：90-99%
- 🎯 精准度：93%（纯语义搜索仅 59%）

**测试问题**："我们三个月前讨论的那个项目，最后用的什么方案？"

| 对比项     | 启用前            | 启用后    | 改善幅度     |
| :--------- | :---------------- | :-------- | :----------- |
| 上下文大小 | 8 万+ tokens      | 削减 95%+ | -            |
| 响应时间   | 45 秒（超时失败） | 2 秒      | 快 20+ 倍    |
| API 成本   | $2.4              | $0.01     | 降低 200+ 倍 |
| 成功率     | 失败              | 成功      | ✅            |

**结论**：速度快了 20+ 倍，成本降低 200+ 倍，而且不会失败。



### 2.3 上下文压缩与会话修剪（context级别的“短期记忆”）

对话越来越长时，OpenClaw 会**自动压缩**旧对话（总结成摘要），保持上下文窗口可用。

手动压缩：`/compact Focus on decisions and open questions`（可附带指令引导重点）。（`/compact`命令）

自动压缩默认开启。`/status` 可查看压缩次数。

压缩后结构：`压缩摘要 + 最近消息 + 当前提问`

配置（可指定不同模型做压缩总结）：

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "openrouter/anthropic/claude-sonnet-4-5",
        "identifierPolicy": "strict"
      }
    }
  }
}
```

**压缩可靠性保障**：

- 大型会话压缩时自动延长运行截止时间，避免压缩到一半被超时中断
- 压缩后自动修复孤立的 `tool_result` 块，防止残余数据导致后续请求报错
- 空会话不会触发无意义的压缩循环

还有一个比较新颖的技术值得注意，**会话修剪（Session Pruning）**：

修剪只临时裁剪发送给模型的内容，不修改磁盘记录。

| 对比   | 压缩           | 修剪            |
| :----- | :------------- | :-------------- |
| 作用   | 总结旧对话     | 裁剪旧工具结果  |
| 持久化 | 写入 JSONL     | 仅内存中        |
| 触发   | 接近上下文上限 | 每次 LLM 调用前 |

修剪默认关闭。启用（`cache-ttl` 模式）：

```
{
  "agents": {
    "defaults": {
      "contextPruning": {
        "mode": "cache-ttl",
        "ttl": "5m"
      }
    }
  }
}
```

修剪规则：只修剪 `toolResult` 消息；保护最近 3 条助手消息后的工具结果；含图片的不修剪；软裁剪（保留头尾加 `...`）或硬清除（替换为 `[Old tool result content cleared]`）。

默认参数：`ttl` 5 分钟，`keepLastAssistants` 3，`softTrimRatio` 0.3，`hardClearRatio` 0.5，`minPrunableToolChars` 50,000。







# PS：如何产生生产力？

## 1、多模态RAG？

**理论上**,多模态RAG很有价值:

- 医疗文献中的图表
- 设计文档中的示意图
- 视频中的关键帧

**工程上**,有两大拦路虎:

1. **Token爆炸** - 用ColPali处理一页PDF,生成1024个token,每个token 128维,一页就要500KB存储。百万页文档库需要TB级索引。
2. **检索效果** - 纯文本向量检索已经很成熟,但图文混合检索的效果还不够稳定。





## 2、如何在memory上做出有意思的研究呢？

> 来源：个性化llm的memory方向怎么样?https://www.zhihu.com/question/1957619016504967717

已经有人说了现在agent里最常用的显式记忆＋（明文记忆）大概率不是最终的memory形态，我比较赞同。不过在我看来明文记忆本身不是什么坏方向，但是如果只搞明文记忆那就是24K烂泥扶不上墙的坏方向。

Machine Learning最重要的是Learning，问题现在大部分agent里采用的明文记忆模式是把database 里的东西retrieve 回来送进上下文的模式，也就是说几乎不包含任何learning（你非要说in-context learning也是learning那我也没话说），因此如果你要按这个逻辑做的话，那不仅毫无创新点，而且几乎全是工程，没有一丁点算法，搞这种科研不如早点进大厂打螺丝。

从做科研的角度上说，纯做prompting和RAG几乎可以说是LLM research里最烂的那档了，prompt你随便拉个高中生都能写，那你如何说服审稿人收你的稿子？或者说，写prompt真能算”科研"吗？



当然有人可能要说，哎呀 ReAct*，CoT*，AutoPE+之类的也能发，但是

1. 他们已经不是纯prompt，是提出了新的“范式"；
2. 时代不同了。

所以总结一下就是，在我看来memory有很多东西可以做，思考“压缩的形式"是一个非常promising的方向。哪怕你做不到设计一种新的压缩范式，做其他人提到的知识图谱+LLM其实也不那么incremental，至少能给审稿人一个收你稿子的理由。

如果你只是想"实现一个能用的memory模块"，prompting和RAG确实足够了。只不过这相当无趣，甚至几乎不能称得上科研—每天就是看log，拉badcase，修prompt。如果memory的research搞成这样的话，那就变成堆砌工程了，不仅文章难出，而且做这玩意很枯燥的（别问我怎么知道的，说多了都是泪）。
