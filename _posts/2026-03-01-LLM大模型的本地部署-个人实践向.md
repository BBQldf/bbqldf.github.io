# 大模型的本地部署

> 随着模型规模的不断扩大和应用场景的持续深化，完全依赖云端服务暴露出了诸多问题：高昂的API调用成本、数据传输延迟、敏感数据隐私风险，以及特定场景下的网络连接限制等。

# 一、一句话总结

**一句话总结：项目部署推荐使用ollama，个人实践先用LM studio**.

现在市面上工具特别多，我们尽量地提炼出这些工具的核心能力，有些迭代慢，有些功能缺，我们找出一条路之后就可以基本完成技术选型了，最后的形式应该是**“****LLM +** **各种接入工具”。**



# 二、why本地部署

这个没得选，未来想要在生产环境使用，必须要进行本地化，我写了几个必要的点，供参考。数据安全性和模型灵活性好理解，额外的一个认知是我们在实践过程中总结出来的，调用deepseek这类最大的模型，我们自部署成本怎么算都是更高的。**自部署主要的好处是“吞吐量够大”**，外部API的推理服务吞吐量限制的很厉害。**基于开源模型去做出我们自己的优秀模型是最重要的路径**。

 

**1.** **数据安全与隐私保护**

在线大模型通常需要将数据上传至云端处理，这对于企业核心数据、商业机密或个人隐私信息来说存在泄露风险。本地化部署可以确保数据始终留在本地，避免因数据传输和云端存储带来的安全隐患。例如，金融、医疗、政府等行业对数据安全有极高要求，本地模型成为唯一可行选择。

 

**2.** **行业定制与模型微调**

通用大模型虽然强大，但在某些垂直细分领域（如法律、医疗、制造业）难以满足专业需求。企业可利用本地化部署，结合自身行业数据进行模型微调和改造，使其更懂业务、更贴合场景。例如，企业可以用自己的知识库、业务流程数据训练模型，获得更高准确率和业务适配性。

 

**3.** **高性能与低延迟需求**

本地部署大模型能够极大提升响应速度，减少网络延迟，适合对实时性要求极高的场景，如智能客服、边缘计算、自动驾驶、工业控制等。云端模型受限于网络状况，可能无法满足极端实时需求，而本地模型可做到毫秒级响应。

 

**4.** **降低长期成本**

对于高频使用场景，云端大模型按量计费或订阅费用较高。长期来看，本地部署可以降低运维和使用成本，尤其是企业自有硬件资源充足时，能减少对云服务商的依赖。

 

**5.** **系统自主性与可控性**

本地化部署让企业和个人对模型有更高自主权，可以自由调整、升级、控制模型版本，避免被云厂商“锁定”，也便于合规审计和本地法规适配。比如，某些国家或行业有数据本地化法律要求，只有本地部署能满足。

 





# 三、哪些方案是比较好的



根据模型规模和性能需求，选择以下工具之一：

### **2.1 Ollama（新手友好）**

- **适用场景**：快速部署开源模型（如Llama3、Qwen、DeepSeek）。

- 步骤

  ：

  1. 安装Ollama：`brew install ollama`。
  2. 拉取模型：`ollama pull llama3:8b`（支持量化版本如`llama3:8b-instruct-q4_K_M`）。
  3. 启动服务：`ollama serve`，通过API或WebUI（如Open WebUI）访问。

### **2.2 vLLM（生产级高性能）**

- **适用场景**：需要高吞吐量的生产环境（如M4 Pro芯片）3。

- 步骤

  ：

  1. 安装依赖：`pip install vllm`（需Python 3.10+）。
  2. 启动服务：`python -m vllm.entrypoints.api_server --model Qwen/Qwen2-7B-Instruct`。
  3. 优化参数：通过`--tensor-parallel-size`启用多芯片并行（M4 Pro支持）。



Qwen3在Apple Silicon上通过MLX框架实现最佳性能，相比PyTorch快2倍，支持4-bit、8-bit量化优化([AI Development on Roundfleet Mac Mini M4: Mastering Apple's MLX](https://www.roundfleet.com/blog/ai-development-mac-mini-m4-mlx))。

| 部署工具  | 优势                                  | 适用场景                      | 安装配置                                                     |
| :-------- | :------------------------------------ | :---------------------------- | :----------------------------------------------------------- |
| MLX       | Apple Silicon优化最佳，比PyTorch快2倍 | 高性能推理，M4芯片首选        | `pip install mlx`                                            |

#### vLLM 的三大核心技术

##### **PagedAttention**

借鉴操作系统虚拟内存分页思想，将 KV Cache 切分为固定大小的"页面"分散管理，显存利用率提升 3-5 倍，支持变长序列混批，抗 OOM 能力更强。

而且这一切对开发者完全透明。你看下面这段代码，根本不需要手动操作显存：

```
from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-2-7b-chat-hf",
    tensor_parallel_size=2,
    dtype='half',
    enable_prefix_caching=True  # 启用前缀缓存，重复 prompt 直接复用
)

sampling_params = SamplingParams(max_tokens=200)
outputs = llm.generate(["你好，请介绍一下你自己。"], sampling_params)
print(outputs[0].text)
```





##### **连续批处理（Continuous Batching）**

还记得传统批处理是怎么工作的吗？等齐一队人再发车，哪怕有人早到了也只能干等。这就是所谓的“尾延迟陷阱”：一个长请求拖垮整批性能。

而 vLLM 的连续批处理，玩的是“动态流水线”：

* 新请求随时插入正在运行的批次；
* 某个请求 finish 了？立刻释放它的页面，不影响别人；
* GPU 几乎永远满载，利用率拉到 90%+。

内置异步引擎，天生适合高并发场景：

```
import asyncio
from vllm import AsyncLLMEngine
from vllm.sampling_params import SamplingParams

engine = AsyncLLMEngine(
    model="Qwen/Qwen-7B-Chat",
    tensor_parallel_size=2,
    max_model_len=32768  # 支持 32K 超长上下文！
)

async def generate_response(prompt: str):
    sampling_params = SamplingParams(max_tokens=150)
    async for result in engine.generate(prompt, sampling_params, request_id=f"req-{hash(prompt)}"):
        if not result.finished:
            print("Streaming:", result.outputs[0].text)  # 流式输出，用户体验拉满
    return result.outputs[0].text

# 并发处理多个请求
async def main():
    prompts = ["写首诗", "解释量子纠缠", "推荐一本好书"]
    await asyncio.gather(*[generate_response(p) for p in prompts])

asyncio.run(main())
```



##### **原生量化支持（GPTQ/AWQ）**

INT4 量化开箱即用，模型体积缩小 4 倍，让 7B 模型能跑在消费级显卡（如 RTX 3090）上。

```
# 直接加载 AWQ 量化模型
llm = LLM(
    model="Qwen/Qwen-7B-Chat-AWQ",
    quantization="AWQ",
    gpu_memory_utilization=0.9  # 最大利用 90% 显存

# 或者 GPTQ
llm_gptq = LLM(

```



### 2.3 Triton（高阶用法-系统部署）

> 信息来源：
>
> 1、vLLM的优势：https://blog.csdn.net/weixin_42581846/article/details/155259082
>
> 2、Triton部署：https://blog.csdn.net/gitblog_00985/article/details/151338316

你有没有遇到过这种情况：好不容易训好的大模型，一上线推理就卡成PPT？多显存爆了、延迟飙升、吞吐上不去.明明是A100集群，跑起来还不如家用显卡流畅。这背后，往往不是硬件不行，而是推理引擎没选对。

Transformer 解码有个致命特点：**每生成一个 token，都要把前面所有 token 的Key/Value 缓存（KV Cache）留在显存里。序列越长，缓存越多，显存占用像滚雪球一样越来越大**。

更要命的是，传统做法是给每个请求分配一块连续的显存块。这就导致：

* 请求长度不同——大量内存碎片；
* 批处理只能等最慢的那个完成——GPU 空转；
* 小请求浪费资源，大请求直接 0OM （Out of Memory）...

结果就是：显存利用率不到 30%，GPU 成了“电暖气”。

Triton本质是一个“全能选手”：支持 TensorFlow、PyTorch、ONNX，能跑图像分类也能跑语音识别。所以现在系统化部署更加地全面。

| 特性            | Triton Inference Server              | FastAPI              | 传统Flask+TorchServe方案 |
| :-------------- | :----------------------------------- | :------------------- | :----------------------- |
| **多框架支持**  | 支持TensorFlow/PyTorch/ONNX等10+框架 | 无限制（需自行集成） | 仅支持PyTorch            |
| **性能优化**    | 动态批处理/模型并行/显存优化         | 异步I/O/自动数据验证 | 需手动实现批处理         |
| **API开发效率** | 无业务API层                          | 自动生成OpenAPI文档  | 需手动编写文档           |
| **资源占用**    | 较高（完整推理引擎）                 | 轻量（~20MB内存）    | 中等                     |
| **适用场景**    | 高性能推理核心                       | 业务API编排          | 简单演示场景             |

#### Triton客户端封装（triton_client.py）

```python
import numpy as np
import tritonclient.grpc as grpcclient
from tritonclient.utils import InferenceServerException
 
class TritonClient:
    def __init__(self, url="localhost:8001"):
        self.client = grpcclient.InferenceServerClient(url=url)
        self.model_metadata = {}
        
    def get_model_metadata(self, model_name: str):
        """获取模型元数据"""
        if model_name not in self.model_metadata:
            self.model_metadata[model_name] = self.client.get_model_metadata(model_name=model_name)
        return self.model_metadata[model_name]
    
    async def infer(self, model_name: str, input_data: dict):
        """执行推理请求"""
        inputs = []
        outputs = []
        
        # 构造输入张量
        for name, data in input_data.items():
            input_tensor = grpcclient.InferInput(name, data.shape, "FP32")
            input_tensor.set_data_from_numpy(data.astype(np.float32))
            inputs.append(input_tensor)
        
        # 获取输出张量信息
        metadata = self.get_model_metadata(model_name)
        for output in metadata.outputs:
            outputs.append(grpcclient.InferRequestedOutput(output.name))
            
        # 执行推理
        try:
            response = self.client.infer(
                model_name=model_name,
                inputs=inputs,
                outputs=outputs
            )
            return {name: response.as_numpy(name) for name in response.outputs}
        except InferenceServerException as e:
            raise ValueError(f"Triton推理错误: {str(e)}")
```

- **Triton** 适合企业级多模型推理部署，不仅能服务 LLM，也能部署图像、语音等多种 AI 模型，并提供更丰富的管理和监控功能。

**一句话：vLLM 不是要取代 Triton，而是在 LLM 这个垂直场景上"降维打击"。**

- **纯 LLM 业务**（客服、写作助手、代码生成）→ 直接用 vLLM，完全够用
- **多模型混合调度**（视觉+语音+文本+推荐）→ 继续用 Triton
- **最佳组合**：用 Triton 做路由，把 LLM 请求转发给 vLLM 处理，强强联合



# 四、如何产生生产力？相比于阿里云部署？

<font color='red'>**一句话总结：vibe coding是实现AI agent的形态之一，是实现快速接入AI的手段。**</font>

一些逻辑，AI现在有很多“健忘症”、“幻觉”等问题，这里的问题除了模型基座的人要思考，我们也要积极思考并跟进解决方案，因为这里面的一些点，未来结合我们的底座模型，或许可以做出属于本公司的“护城河”。



**一句话总结：windows、linux、macos不同的架构会涉及到不同的模型优化方案，短期我不倾向于自研底层优化逻辑，基于包装好的容器能力，部署本地服务后，对外提供整体的“自有”大模型能力**

l **模型架构设计：**不同大模型采用的底层架构（如Transformer、MoE、MoR等）直接影响模型的计算效率、可扩展性、推理速度和适用场景。例如，我们在用的一些MoE（专家混合模型）通过稀疏计算提升大规模模型的推理效率，适合高并发场景，而MoR（递归混合模型）通过动态计算深度降低延迟，更适合中小规模应用

l **硬件与算力适配：**不同模型对硬件资源（GPU、TPU、CPU、边缘设备）的需求不同，系统架构需考虑算力分配、异构计算、内存管理等因素。

l **业务场景适配性：**实时对话系统、API服务、多模态推理、边缘部署等不同场景，对系统架构有不同要求

l **可扩展性与长期维护：**这一块是我单独拎出来的一点，本次试验过程中，就发现像codebuddy这种背靠腾讯的工具平台，在半年内迭代得非常好了，而有些工具就贡献度很低，关注模型和系统架构的可扩展性、升级便利性和运维成本，确保未来业务发展和技术迭代能顺利支持
