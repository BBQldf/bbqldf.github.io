> 一句话把秒杀系统说完全的话，就是**将请求尽可能拦截在上游，通过限流、缓存、异步削峰，保护下游数据库，确保系统在高并发下的稳定性与数据一致性。**
>
> 相对具体一点的话，**多级缓存 + 限流熔断 + 异步队列 + 库存预减**，让真正打到 DB 的请求降到最低。
>
> 秒杀系统，是电商系统的一个特殊场景。所以我们先解决电商系统中的简单并发问题，再去解决秒杀系统中的高并发问题



# 电商系统的概览

![img](https://camo.githubusercontent.com/e1d8e676734366a9aabeae1c7c7929138166db5d0e1d7f598067342025e94063/68747470733a2f2f63646e2e6a7364656c6976722e6e65742f67682f4f534a6f62732f6f736a6f62732d73797374656d2f7374617469632f7370696b652f2545382542342541442545342542392542302545352539352538362545352539332538312e737667)

秒杀系统只是在电商系统上增加了一些特定条件。一般分三个步骤：

1. 客户通过客户端下单
2. 如果下单成功则进入支付阶段，否则返回购买失败
   1. 客户端发送下单请求给服务端
   2. 服务端查询数据库
      - 若该商品库存大于零，将库存减一并且返回下单成功
      - 若该商品库存等于零的话，返回下单失败
3. 进入支付阶段后，如果在一定时间内支付成功则返回购买成功，否则返回购买失败



问题的出现：“当商品库存只剩下 1 件而有多位客户同时下单的时候，每个下单请求在查询的时候都发现库存大于零，并且将库存减 1 返回下单成功”。

总结而言，**超卖问题的原因在于事务查询和更新库区期间，库存已经被其他事务修改了**。我们知道大部分并发问题都可以通过锁机制或者队列服务来解决，因此提供一下几种方案：

## 锁机制

### 悲观锁（强锁，排他锁）

![悲观锁](https://camo.githubusercontent.com/b7008c4c410ab4e0e3dfdc5bb0b80b54cbbb10b0a16a09ee6ad5a54f5c1e91f9/68747470733a2f2f63646e2e6a7364656c6976722e6e65742f67682f4f534a6f62732f6f736a6f62732d73797374656d2f7374617469632f7370696b652f2545362538322542322545382541372538322545392539342538312e737667)

悲观锁成功解决了商品超卖问题，不过它的缺点也比较明显：1）处理性能不高，当一件商品有多位客户同时下单的时候，每个请求需要等待排他锁，也要较长才知道是否下单成功。2）容易发生死锁：在实际工程中，下单操作不只涉及了库存修改，还可能涉及其他业务功能，由于悲观锁下每个请求都轮流持有锁，应用层的代码处理不好的话会更容易发生死锁。



### 乐观锁（事务锁、version锁）

![乐观锁](https://camo.githubusercontent.com/69d44af2e3a186e544e033a0be5ac979c9fb2ff6c2c57b7f7a4b7ea739198f29/68747470733a2f2f63646e2e6a7364656c6976722e6e65742f67682f4f534a6f62732f6f736a6f62732d73797374656d2f7374617469632f7370696b652f2545342542392539302545382541372538322545392539342538312e737667)

乐观锁因为并不需要等待锁，所以在事务竞争较少的情况下比悲观锁有更好的性能，缺点是事务竞争较多的情况下，由于经常需要回滚事务导致性能反而较差。



### 分布式锁（zookeeper、redis实现）——在现代架构中比较好用的一种

![分布式锁](https://camo.githubusercontent.com/f074b5bfd603253ca3a82e049c154218462da3a7cdd334011bd43c47b9e3f826/68747470733a2f2f63646e2e6a7364656c6976722e6e65742f67682f4f534a6f62732f6f736a6f62732d73797374656d2f7374617469632f7370696b652f2545352538382538362545352542382538332545352542432538462545392539342538312e737667)

分布式锁的优点是**将功能进行分离**（计数的和真实记录的分开），分布式组件负责解决并发安全的问题，数据库负责数据存储。不过缺点在于 1）分布式锁的正确实现并不简单，错误的实现方式容易引起其他一致性的问题。2）分布式锁在高并发下也会产生锁竞争的问题，性能不佳。3）由于引入了新的组件，要考虑分布式组件的可靠性，以及崩溃之后的恢复机制。



### 消息队列（电商场景最佳，不过kafka本身也是为这个设计的）

![消息队列](https://camo.githubusercontent.com/38f0ad9c4ad112be4d4f83e8e31c1b40706054339090c8fcd05b60b70476ee32/68747470733a2f2f63646e2e6a7364656c6976722e6e65742f67682f4f534a6f62732f6f736a6f62732d73797374656d2f7374617469632f7370696b652f2545362542362538382545362538312541462545392539382539462545352538382539372e737667)

**消息队列的优点对业务进行了解耦**，除了数据库之外，其他对下单请求感兴趣的业务系统，例如数据分析，日志记录等都可以订阅下单请求的消息。缺点在于 1）因为消息队列可能会崩溃，消息发送也可能失败，所以要考虑消息只消费一次，不会因为重复消费导致重复下单。2）由于引入了新的组件，要考虑消息队列的可靠性，以及崩溃之后的恢复机制。



# 秒杀系统的演进

秒杀系统和电商系统有两个核心区别：

1. 双十一也有极大的流量，但是双十一的商品种类很多，所以流量会分布到不同的商品中。而秒杀系统中，商品的种类和库存都比较少，导致大部分流量集中在少量商品中。
   1. 这意味着：当一款商品库存只有 10 件却有 1 万名用户下单的时候，只有前 10 名客户可以下单成功，其他用户都浪费时间在队列等待以及无意义地查询库存，既牺牲了用户体验也增加了消息队列以及数据库的压力
2. 秒杀系统由于商品稀缺，价值高。同一位客户可能会对同一商品多次提交下单请求，而且恶意刷单的请求比较多，所以系统接收到的无效请求及非法请求较多。
   1. 这里可以快速通过CDN缓存解决很大问题
3. 由于库存过少，有大量的请求（例如非法用户的请求，超过秒杀活动开始一定时间的请求）其实是没有机会抢到商品的，所以没有必要到达服务器，更不用说数据库了。
   1. 背后巨大的服务压力





## 限流机制（限流 + 负载均衡）

- 限流器分为前端限流与后端限流：

  - 前端限流包括验证答题，防止重复点击按钮等常见机制。
  - 后端限流使用限流算法进行流量限制，简单情况下可以使用固定限流算法，例如秒杀商品的库存是 10 件，只要限流器接收到 10 * k（k 可以根据业务进行调整）个请求之后，就停止接受该商品的所有请求。这样无论有多少个下单请求，最终到达服务器的单个商品请求数量都不超过 10 * k。实际工程中，因为有客户可能会出现支付超时导致释放库存的情况，系统需要通知限流器接受新的请求。

- 负载均衡负责将下单请求通过负载均衡算法转发到最合适的服务器。

  - 🔄 静态算法（不考虑服务器状态）

    | 算法                         | 描述                         | 适用场景                |
    | :--------------------------- | :--------------------------- | :---------------------- |
    | **轮询 Round Robin**         | 依次将请求分发给每台服务器   | 服务器性能相近          |
    | **加权轮询 Weighted RR**     | 按权重比例分发，性能强的多分 | 服务器性能差异较大      |
    | **随机 Random**              | 随机选择服务器               | 请求量大且均匀          |
    | **加权随机 Weighted Random** | 按权重随机选择               | 性能差异 + 随机分布     |
    | **IP Hash**                  | 对客户端 IP 取哈希，固定路由 | 需要会话保持（Session） |
    | **URL Hash**                 | 对请求 URL 取哈希            | CDN / 缓存命中率优化    |

    ------

    📊 动态算法（根据服务器实时状态）

    | 算法                           | 描述                         | 适用场景          |
    | :----------------------------- | :--------------------------- | :---------------- |
    | **最少连接 Least Connections** | 选当前连接数最少的服务器     | 请求耗时差异大    |
    | **加权最少连接**               | 结合权重 + 最少连接          | 性能不均 + 长连接 |
    | **最短响应时间**               | 选响应时间最短的服务器       | 对延迟敏感的业务  |
    | **自适应负载**                 | 综合 CPU、内存、连接数等指标 | 复杂生产环境      |

- 安全验证组件分为前端安全验证以及后端安全验证：

  - 后端安全验证包括黑名单校验，IP 地址校验等机制。
  - 前端安全验证包括：客户端账户验证（确保客户有资格参考秒杀活动），客户端版本安全验证（防止反编译以及修改客户端代码），秒杀接口动态生成（防止使用刷单脚本）等机制。



## 热门资源隔离

既然大部分流量集中在少量商品中，我们能不能针对这些商品进行特殊处理呢？这样既可以防止秒杀活动影响其他业务功能，也可以针对热门商品进行资源分配，答案是可以的，首先我们需要识别出热门商品，这里有两种常见的方法：

- 静态识别：包括京东在内的一些电商平台，客户在参加秒杀活动之前需要先进行预约，只有预约过的客户才能参考秒杀活动。这样系统可以提早识别热门商品以及进行流量预估。
- 动态识别：通过实时数据分析系统在秒杀活动前统计出现在较多客户浏览的热门商品，针对预估结果进行资源分配。

识别出热门商品之后，我们可以将热门商品的资源进行隔离，并且设置独立的策略，例如

- 使用特殊的限流器，由于秒杀系统的库存很少，在下单请求开始阶段就可以随机丢弃大部分请求。
- 使用单独的数据库，下单请求的处理速度受限于消费者的处理速度，也就是数据库的处理速度。我们可以对热门商品进行分库分表，这样可以将请求处理的压力分摊到多个数据库中。下图中，我们将秒杀系统的一些组件独立开来：

# Q&A：



### Q：“对秒杀商品进行分库分表之后可能导致某个分表库存为零，但其他分表还有库存”，这是什么场景，如何解决？

先说根本原因，主要就是这四类：

| 原因               | 说明                                 |
| :----------------- | :----------------------------------- |
| **路由不均匀**     | 哈希路由导致某些分片热点             |
| **用户行为集中**   | 特定用户群体被路由到同一分片         |
| **分片策略不合理** | 按用户ID分片导致活跃用户集中         |
| **缺乏跨分片协调** | 各分片独立扣减，无法感知其他分片库存 |

这其实对应着三个问题，一个是这个问题的产生是路由不均衡；另外一个是在这个不均衡发生之后，有什么快速解决方案；以及最佳实践是什么样子。



1、**路由不均衡**

```
情况一：请求路由不均（流量倾斜）
用户请求 → 路由策略 → 大量打到同一分片 → 该分片库存耗尽

情况二：库存分配不均（初始分配问题）
总库存 1000 → 分片0: 700, 分片1: 100, 分片2: 100, 分片3: 100
即使流量均匀，分片0也会最先耗尽
```



2、应急手段 —— **感知** + **补救**

```
感知：监控各分片库存水位，发现某分片趋近于零
补救：
  ├── 停止向该分片路由（隔离）——停止向库存为零的分片路由新请求；（直接摘掉分片0，流量只打1/2/3）
  └── 从其他分片动态调拨库存（再平衡）——从1/2/3各抽一部分库存补充到分片0，恢复均衡 ——调拨过程必须加分布式锁，防止调拨期间超卖
```

这是**事后补救**，有时延，不优雅



3、最佳实践——核心思路是**把"分片库存"和"全局库存"解耦**

```
Redis 全局原子计数（唯一扣减入口）
        ↓
扣减成功后，随机路由到任意分片写DB
        ↓
分片只负责"记录"，不负责"判断是否售罄"
```

这样分片是否均衡**根本不影响超卖和假售罄判断**，因为库存的"真相"始终在 Redis 全局计数器里



##### 继续追问，**Redis 全局计数与数据库**会不会出现缓存和数据库一致性的问题呢？

几种场景：

```
场景一：写DB时服务宕机
Redis -1 成功，DB写入请求还没发出去，进程挂了

场景二：写DB时网络超时
Redis -1 成功，DB写入超时失败，但不知道到底写没写进去

场景三：写DB时数据库挂了
Redis -1 成功，DB不可用，写入直接失败
```

**方案一：同步写DB，失败回滚 Redis（简单但有性能损耗）**

```
public boolean deductStock(String itemId) {
    // 第一步：Redis 原子扣减
    Long remaining = redisTemplate.opsForValue()
        .decrement("stock:" + itemId);
    
    if (remaining < 0) {
        // 超卖了，回滚
        redisTemplate.opsForValue().increment("stock:" + itemId);
        return false;
    }
    
    try {
        // 第二步：同步写DB
        stockMapper.deductStock(itemId);
        return true;
    } catch (Exception e) {
        // 第三步：DB失败，回滚Redis
        redisTemplate.opsForValue().increment("stock:" + itemId);
        return false;
    }
}
```

**方案二：异步写DB + 消息队列保证最终一致（推荐）**

```
Redis 扣减成功
        ↓
发送消息到 MQ（持久化）
        ↓
返回用户成功
        ↓
消费者异步消费消息 → 写入DB
        ↓
写入失败 → MQ重试，直到成功
```

但是这也有一个问题，发送 MQ 失败，Redis 少了1，MQ没有记录，DB也没有。可以最好是先发mq，再更新缓存，用事物保证一致性。

**方案三：定时对账兜底（任何方案都应该加）**

```
每隔一段时间
        ↓
统计DB中实际扣减记录数
        ↓
与Redis全局计数对比
        ↓
发现不一致 → 告警 + 自动修正
```

但是这里会出现锁的问题，

```
正常业务写入：
消费者不断消费MQ消息 → INSERT 扣减记录 → 并发很高

对账任务同时在跑：
SELECT COUNT(*) 统计扣减数量 → 全表扫描或大范围查询
        ↓
发现不一致，执行 UPDATE 修正
        ↓
和业务写入产生锁竞争！
```

几个快速解决方案，对账只读不写，修正异步化。（而且主从设计里面，读从库，写主库）

```
对账任务
    ↓
只做读操作，发现不一致
    ↓
不直接UPDATE，而是发一条"修正消息"到MQ
    ↓
由专门的修正消费者串行处理
    ↓
业务高峰期修正消费者暂停，低峰期再消费

并且这里对账走只读从库，修正走主库：
主库                    从库
  ↑                       ↑
业务写入              对账SELECT
MQ消费写入            （不加锁，不影响主库）
        ↓
        发现不一致
        ↓
        修正消息 → 低峰期写主库
```

这样对账任务本身不持有锁，修正操作时错峰执行，不影响业务





### Q：消息队列怎么保证消息有且仅生效一次（Exactly Once）？

> **核心难点**：网络不可靠 → 消息可能**丢失**或**重复**，两者难以同时避免

#### 生产者侧 — 保证消息不丢

| 手段                     | 说明                                                 |
| :----------------------- | :--------------------------------------------------- |
| **同步发送 + ACK确认**   | 收到 Broker 确认再继续                               |
| **失败重试**             | 网络抖动时自动重发                                   |
| **本地消息表**           | 消息写 DB 与业务操作同一事务，保证发出（这个很重要） |
| **事务消息（RocketMQ）** | 半消息机制，业务提交后才投递                         |

#### Broker 侧 — 保证消息不丢 —— 这个一般不考虑

接收生产者发送的消息，按主题（Topic）和队列（Queue/Partition）持久化存储（通常顺序写磁盘），并根据消费者订阅关系投递消息。 RocketMQ 中是“分拣中心”，在 Kafka 中每个节点即一个 Broker，在 RabbitMQ 中指单个消息代理实例（可集群部署）。

支持高吞吐（如 CommitLog 顺序写）、高可用（主从/多副本）、路由管理（向 NameServer 或 ZooKeeper 注册元数据）、消费位点（Offset）追踪。

位于“生产者 → ‌**Broker**‌ → 消费者”架构的中心，解耦系统、削峰填谷、保障异步可靠通信。‌‌

| 手段           | 说明                                               |
| :------------- | :------------------------------------------------- |
| **持久化存储** | 消息落盘再返回 ACK                                 |
| **副本同步**   | 多副本复制，防止单点宕机丢消息                     |
| **去重存储**   | 对 Producer 生成的唯一 ID 去重（Kafka 幂等生产者） |

#### 消费者侧 — 保证消息不重复生效（最关键）

**消费者幂等处理** 是实现 Exactly Once 的核心！

```
方案一：唯一消息ID去重
  消费前查 DB/Redis 是否已处理该 messageId
  ✅ 简单通用  ❌ 需要额外存储

方案二：数据库唯一索引
  业务数据设唯一键，重复插入直接报错忽略
  ✅ 天然幂等  ❌ 仅适合插入场景

方案三：乐观锁（版本号）
  UPDATE table SET val=x WHERE id=? AND version=?
  ✅ 适合更新场景  ❌ 需要版本字段

方案四：状态机 —— 这个设计很考验设计能力
  判断当前状态是否允许该操作（如订单状态流转）
  ✅ 业务语义清晰  ❌ 需要设计状态流转
```





总结而言，**生产者用事务/本地消息表保证不丢，Broker 持久化保证不丢，消费者用幂等处理保证不重复生效 —— 三者结合才能实现真正的 Exactly Once。**
