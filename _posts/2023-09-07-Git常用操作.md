# git常用操作

## 一、仓库配置类

### 1、添加远程库

添加一个新的远程仓库，可以指定一个简单的名字（一般远程库我们都叫他origin），以便将来引用,命令格式如下：

```shell
git remote add [shortname] [url]
git remote add origin  git@github.com:xxxxxx/ssssss.git		//常规操作
```



只要本地作了提交，就可以通过命令：

```
$ git push origin master
$ git push -u origin master //由于远程库是空的，我们第一次推送master分支时，加上了-u参数，Git不但会把本地的master分支内容推送的远程新的master分支，还会把本地的master分支和远程的master分支关联起来，在以后的推送或者拉取时就可以简化命令(直接用git pull或者git push，不需要参数)。
```

把本地`master`分支的最新修改推送至GitHub。

#### 1.1 确保提交代码时能知晓你的信息(name和email)需要配置

配置id信息：

```
git config --global user.name "dongshuo"
git config --global user.email "dongshuo03@meituan.com"
```

验证是否配合完成：

```shell
git config --list
```

生成ssh秘钥:

```shell
ssh-keygen -t rsa
```

进入cd ~/.ssh  查看目录下文件的id_rsa.pub 将该信息复制到代码平台，添加SSH公钥即可。



#### 1.2 删除远程库

根据名字删除，比如删除`origin`：

```
$ git remote rm origin
```

此处的“删除”其实是解除了本地和远程的绑定关系，并不是物理上删除了远程库。远程库本身并没有任何改动。要真正删除远程库，需要登录到GitHub，在后台页面找到删除按钮再删除。

#### 1.3 删除 远程分支/远程tag

在Git v1.7.0 之后，可以使用这种语法删除远程分支：

```
$ git push origin --delete <branchName>			//删除远程分支
$ git push origin --delete tag <tagname>		//删除远程tag
```

##### 1.3.1 删除 本地分支/本地tag/本地提交

```
git tag -d <your_tag_name> 删除一个标签
git branch -d dev  删除本地分支

git rm --cached <file>		//只想从暂存区删除文件，本地工作区不做出改变
git reset/revert		//删除提交，慎用reset
```

##### 1.3.2 删除/丢弃本地修改文件

```
git clean -nxdf（查看要删除的文件及目录，确认无误后再使用下面的命令进行删除）
git checkout . && git clean -xdf

```









## 二、分支类

让我们来看一个简单的分支新建与分支合并的例子，实际工作中你可能会用到类似的工作流。 你将经历如下步骤：

1. 开发某个网站。
2. 为实现某个新的用户需求，创建一个分支。
3. 在这个分支上开展工作。

正在此时，你突然接到一个电话说有个很严重的问题需要紧急修补。 你将按照如下方式来处理：

1. 切换到你的线上分支（production branch）。
2. 为这个紧急任务新建一个分支，并在其中修复它。
3. 在测试通过之后，切换回线上分支，<font color='red' size='5'>**然后合并这个修补分支**</font>，最后将改动推送到线上分支。

> 一定要记得merge到master最新的提交！！！！



### 0、新建分支

常用操作：

```
$ git checkout -b iss53
Switched to a new branch "iss53"
```

它是下面两条命令的简写：

```console
$ git branch iss53
$ git checkout iss53
```







### 1、修改分支名称

#### 情况1:本地还没有推送到远程

本地分支重命名(还没有推送到远程)

```
git branch -m oldName newName
```

#### 情况2:本地已经推送远程，且本地和远程的分支名目前是一致的

1. 重命名远程分支对应的本地分支

```
git branch -m oldBranch newBranch
```

2. 删除远程分支

- 当远程不再使用的话，建议删除。冗余的分支会让项目看起来不清晰简洁（<font color='red' size='4'>可选</font>，建议）：

```
git push --delete origin oldBranch
```

3. 然后在本地完成修改后，上传新命名的本地分支：

```
git push origin newBranch
```

4. 把修改后的本地分支与远程分支关联

```
git branch --set-upstream-to origin/newBranch
```

上面这个命令用的少，<font color='red'>**也可以这样，**</font>通过运行 `git push origin -u new-branch-name` 将上游分支重置为新本地分支的名称。

```
git push origin -u newBranch
```



### 2、拉取远程分支到本地

#### 情况1: 目前本地还没拉代码，直接拉分支代码

```
git clone -b ac git@gitlab.yopoint.vip:ac/YoPointSwift.git
```





#### 情况2: 本地已经拉取了代码，想拉取远程某一分支的代码到本地

- 拉取远程分支到本地(方式一)：

```
git checkout -b newBranch origin/newBranch
```

如果出现报错：`fatal: 'origin/newBranch' is not a commit and a branch 'newBranch' cannot be created from it`

 解决方法（下面两个都可以）：

```
git fetch origin //同步一下远程仓库
git fetch // 更新远程的所有更新
```



- 拉取远程分支到本地(方式二)：

```
git fetch origin newBranch
```

<font color='red' size='5'>注意：</font>

1. 区别创建分支命令，和拉取远程分支的命令：

```
git checkout -b ac_branch        这是在当前分支上创建一个ac_branch分支

git checkout -b ac_branch origin/ac_branch   这才是拉取远程分支ac_branch到本地
```

2. 本质上，方式1和方式2是做的类似的事情：

方式1做了三件事：

1. 拉取远程分支到本地
2. 在本地创建一个分支与远程分支对应
3. 自动切换到刚创建好的分支

方式2做了三件事：

1. 同步远程仓库（git fetch origin）
2. 拉取远程分支到本地
3. 在本地创建一个分支与远程分支对应

### 3、合并分支（merge）

> 在两种情况下，我们需要合并分支：
>
> 1. 在工作中，我们在分支上完成开发之后，一般需要将分支合并回主干，等待发布
> 2. 在分支开发的过程中，如果有人并行进行了开发（并且更新了master），导致我们的拉取的master分支已经落后最新的master，我们需要把最新的master与当前的分支进行合并

1. 对于第一种情况（合并分支到主干）：将 hotfix 分支合并回你的 master 分支来部署到线上

```
$ git checkout master
$ git merge hotfix
Updating f42c576..3a0874c
Fast-forward
 index.html | 2 ++
 1 file changed, 2 insertions(+)
```

这个时候master被快进到hotfix节点。

这个紧急问题的解决方案发布之后，你准备回到被打断之前时的工作中。 

你应该先删除 `hotfix` 分支，因为你已经不再需要它了：

- `master` 分支已经指向了同一个位置。 你可以使用带 `-d` 选项的 `git branch` 命令来删除分支：

```
git branch -d hotfix
Deleted branch hotfix (3a0874c).
```

同时，如果hotfix版本有问题，也可以通过commit id进行回退，也不需要创建分支。

2. 对于第二种情况：

```
$ git checkout tt1
$ git merge master
```

## 三、git commit和git push操作类

### 1、git commit将暂存区内容提交到版本库

下面四个常用的掌握就可以了：

```
git commit -m [message]			// 将暂存区内容提交到版本库

git commit [file1] [file2] -m [message]	//将某些文件提交到版本库 (可以跳过 git add，但必须是已被跟踪的文件)

git commit -am [message]			// 跳过 git add，将所有已被跟踪的文件更改(此时，文件处在 尚未暂存以备提交的变更 的状态)直接提交到版本库

git commit --amend -m <message>			// 修改上一次的提交信息 (删除当前最新的一条记录，重新提交)
```

#### 1.1 git 无法提交未被跟踪的文件

需要注意的是**，commit操作必须要求文件已被跟踪（索引），未被跟踪的文件没办法由暂存库提交版本库。**

如：在分支上新建了一个readme.txt文件，但是没有`git add .` ，这时候执行git commit会报错：

```shell
➜  gitTest git:(tt1) ✗ git commit -am "add readme.txt"
位于分支 tt1
未跟踪的文件:
  （使用 "git add <文件>..." 以包含要提交的内容）
	readme.txt

提交为空，但是存在尚未跟踪的文件（使用 "git add" 建立跟踪）
```

- <font color='red'>不规范操作（使用终端的touch直接创建文件）：</font>

```sh
➜  gitTest git:(tt1) ✗ touch 123.txt

➜  gitTest git:(tt1) ✗ ls
123.txt    readme.md  readme.txt


➜  gitTest git:(tt1) ✗ git status
位于分支 tt1
未跟踪的文件:
  （使用 "git add <文件>..." 以包含要提交的内容）
	123.txt
	readme.txt

提交为空，但是存在尚未跟踪的文件（使用 "git add" 建立跟踪）
```

- 规范操作（使用git add操作跟踪新文件）：

```shell
➜  gitTest git:(tt1) ✗ git add 123.txt
➜  gitTest git:(tt1) ✗ git status
位于分支 tt1
要提交的变更：
  （使用 "git restore --staged <文件>..." 以取消暂存）
	新文件：   123.txt

未跟踪的文件:
  （使用 "git add <文件>..." 以包含要提交的内容）
	readme.txt
```



#### 1.2 git 不会自动检索文件移动

所谓被跟踪，其实就是能被git索引到，因此，**如果你在仓库中对文件重命名则有可能导致错误，因为重命名后的文件没有被跟踪**。

- <font color='red'>不规范操作（直接使用重命名命令mv）：</font>
  - 此时，就需要手动把newFileName文件add到暂存库（`git add newFileName`）

```
$mv FileName NewFileName
$git status

# On branch master
# Changed but not updated:
#   (use "git add/rm <file>..." to update what will be committed)
#   (use "git checkout -- <file>..." to discard changes in working directory)
#
#    deleted:    FileName
#
# Untracked files:
#   (use "git add <file>..." to include in what will be committed)
#
#    NewFileName
no changes added to commit (use "git add" and/or "git commit -a")
```

- 规范操作（使用git来重命名）：

```
$git mv FileName NewFileName
$git status

# On branch master
# Changes to be committed:
#   (use "git reset HEAD <file>..." to unstage)
#
#    renamed:    FileName -> NewFileName
```

#### 1.3 git restore 回退暂存区文件/丢弃工作区的改动

如果要退回某个已经提交到暂存区的文件，可以使用`git status`查看提示。有两种方式：

-  `git reset HEAD <file>...`
- `git restore --staged <文件>...`

<font color='red'>**两个命令都可以，但是建议用restore**！！！</font>因为reset一般用于分支回，避免弄混。

比如，我修改了456.txt文件，

```
➜  gitTest git:(tt1) ✗ vim 456.txt
➜  gitTest git:(tt1) ✗
➜  gitTest git:(tt1) ✗
➜  gitTest git:(tt1) ✗ git status
位于分支 tt1
要提交的变更：
  （使用 "git restore --staged <文件>..." 以取消暂存）
	新文件：   456.txt

尚未暂存以备提交的变更：
  （使用 "git add <文件>..." 更新要提交的内容）
  （使用 "git restore <文件>..." 丢弃工作区的改动）
	修改：     456.txt

未跟踪的文件:
  （使用 "git add <文件>..." 以包含要提交的内容）
	readme.txt
```

采用`git restore 456.txt`将其回退，再`vim 456.txt`，会发现之前的更改已经丢弃。



#### 1.4 git restore, reset and revert的区别

这三个操作的名称是相似的，而且操作也比较类似。

- git restore是关于**从索引或另一个提交还原工作树中的文件**。 **此命令不会更新您的分支**。 该命令还可用于从另一个提交还原索引中的文件。
- git revert 是还原其他提交所做的更改，**并进行新的提交**。
- git reset是**移动顶端(tip)以便从分支中添加或删除提交**（关于更新分支的操作）。 此操作也更改提交历史记录。
  - git reset也可以用来还原索引，与git restore重叠。（我的理解是既然有了restore这样的专项命令，就避免混用吧）

<font color='red'>restore还是好理解，就是把暂存区的文件进行回退，相当于一个对文件的`ctrl + z`功能。但是revert和reset操作就很类似了。</font>

### 2、git push更新代码到仓库

只要本地作了提交，就可以通过命令：

```
$ git push origin master
```

把本地`master`分支的最新修改推送至远程仓库（origin）的master分支



如果是两个不同的项目，要把两个不同的项目合并，git需要添加一句代码——git pull，这句代码是在git 2.9.2版本发生的，最新的版本需要添加--allow-unrelated-histories（即，先把另一个项目拉取过来，再进行提交）：

```shell
git pull origin master --allow-unrelated-histories
git push -u origin master
```

#### 2.1 git push 的 -u 参数含义

> 一般将本地仓库推送到远程仓库的时候一般会使用 `git push` 命令。而作为新手，在网上看到一些教程有的会在 `git push` 的时候带上一个 `-u` 参数，而有的则没有。而推送的实际结果没有什么区别。就很好奇 `-u` 参数的作用到底是什么？

最简单也最官方的操作是，直接在终端运行一下`git push --help`，然后找到该命令的意义：

```
-u
–set-upstream

For every branch that is up to date or successfully pushed, add upstream (tracking) reference, used by argument-less git-pull(1) and other commands. For more information, see branch.<name>.merge in git-config(1).
————————————————
对于每个最新或成功推送的分支，添加上游（跟踪）引用，由无参数 git-pull(1) 和其他命令使用。 有关更多信息，请参阅 git-config(1) 中的branch.<name>.merge。
```

可以看到 `-u` 参数与这个变量相关：`branch.<name>.merge`

在git config中。对于这个变量又有描述如下：

Defines, together with branch.<name>.remote, the upstream branch for the given branch. It tells git fetch/git pull which branch to merge and can also affect git push (see push.default). When in branch <name>, it tells git fetch the default refspec to be marked for merging in FETCH_HEAD. The value is handled like the remote part of a refspec, and must match a ref which is fetched from the remote given by “branch.<name>.remote”. The merge information is used by git pull (which at first calls git fetch) to lookup the default branch for merging. Without this option, git pull defaults to merge the first refspec fetched. Specify multiple values to get an octopus merge. If you wish to setup git pull so that it merges into <name> from another branch in the local repository, you can point branch.<name>.merge to the desired branch, and use the special setting . (a period) for branch.<name>.remote.

主要就是说branch.<name>.merge与branch.<name>.remote一起定义给定分支的上游分支（upstream）。它告诉git fetch/git pull要合并哪个分支，还可以影响git push.

而upstream是指其他人将从中获取的主要存储库，例如您的GitHub存储库。-u选项自动为您设置上游，将您的仓库链接到一个中央仓库。这样，将来Git会“知道”您要推送到的位置以及您要从哪里提取的信息，因此您可以使用git pull或git push不使用参数。 当您git pull从分支进行操作而未指定源远程或分支时，git会查看 branch.<name>.merge 设置以了解从何处提取。而正是git push -u 命令为您要推送的分支设置此信息。

综上，**简单来说，带上`-u` 参数其实就相当于记录了push到远端分支的默认值，这样当下次我们还想要继续push的这个远端分支的时候推送命令就可以简写成`git push`即可。**（相当于打了个标，以后可以直接用git push来默认定位到这个远端分支）

`git push -u origin master`就相当于是执行了:

1. `git push origin master` 
2. `git branch --set-upstream master origin/master`。



> 参考：https://blog.csdn.net/Lakers2015/article/details/111318801

## 1、git fetch命令

**git fetch** 命令用于从远程获取代码库，该命令执行完后需要执行 **git merge** 让远程分支合并到你**所在的**分支。(相当于做同步操作)

常用：

```
git fetch <远程主机名> <分支名>		//取回origin主机的master分支
```

所取回的更新，在本地主机上要用”远程主机名/分支名”的形式读取。比如`origin`主机的`master`分支，就可以用`origin/master`读取。



比如，我们需要同步master上的一个修改（远程已经修改，但是本地还没有同步）：

1. 拉取远程分支

```
git fetch origin		//拉取远程项目库
git fetch origin master:tmp	////在本地新建一个temp分支，并将远程origin仓库的master分支代码下载到本地temp分支
```

有如下信息：

```
remote: Counting objects: 3, done.
remote: Compressing objects: 100% (2/2), done.
remote: Total 3 (delta 0), reused 0 (delta 0), pack-reused 0
Unpacking objects: 100% (3/3), done.
From github.com:tianqixin/runoob-git-test
   0205aab..febd8ed  master     -> origin/master
```

说明 master 分支已被更新！

2. 将更新同步到本地

```
git merge origin/master	 	//将远程master分支同步到本地master分支（默认就是master）
```

<font color='red' size='4'>**可见，拉取分支的最终目的，是为了把远程最新的修改同步到本地的某些分支上**</font>

## 2、git merge操作

`git merge`命令用于将两个或两个以上的开发历史加入(合并)一起。可以分支和主干之间相互合并，也可以分支和分支进行相互合并。（注意合并之后，被合并的分支并没有消失，可以按照自身需要选择删除）

合并分支`fixes`和`enhancements`在当前分支的顶部，使它们合并：

```shell
$ git merge fixes enhancements
```

将分支`dev`合并到当前分支中，**自动进行新的提交**：

```shell
$ git merge dev
```

将分支`maint`合并到当前分支中，但不要自动进行新的提交：

```shell
$ git merge --no-commit maint
```

注意：当您想要对合并进行进一步更改时，可以使用此选项，或者想要自己编写合并提交消息。应该**不要滥用**这个选项来潜入到合并提交中。小修补程序，如版本名称将是可以接受的。



### 2.1 遇到冲突时的分支合并

有时候合并操作不会如此顺利。 如果你在两个不同的分支中，对同一个文件的同一个部分进行了不同的修改，Git 就没法干净的合并它们。

如果你对 `#53 xxx问题`的修改和`#54 有关 hotfix 分支的修改`都涉及到同一个文件的同一处，在合并它们的时候就会产生合并冲突：

```
$ git merge iss53
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.
```

上面，我们在master上（已经合并了#54问题的处理分支），想要合并#53分支，出现冲突（conflict）。

使用 `git status` 命令来查看那些因包含合并冲突而处于未合并（unmerged）状态的文件：

```
$ git status
On branch master
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  (use "git add <file>..." to mark resolution)

    both modified:      index.html

no changes added to commit (use "git add" and/or "git commit -a")
```

可以看到两个提交，都修改了index.html文件。

出现冲突的文件会包含一些特殊区段，看起来像下面这个样子：

```html
<<<<<<< HEAD:index.html
<div id="footer">contact : email.support@github.com</div>
=======
<div id="footer">
 please contact us at support@github.com
</div>
>>>>>>> iss53:index.html
```

这表示 `HEAD` 所指示的版本（也就是你的 `master` 分支所在的位置，因为你在运行 merge 命令的时候已经检出到了这个分支）在这个区段的上半部分（`=======` 的上半部分），而 `iss53` 分支所指示的版本在 `=======` 的下半部分。 为了解决冲突，你必须选择使用由 `=======` 分割的两部分中的一个，或者你也可以自行合并这些内容。 例如，你可以通过把这段内容换成下面的样子来解决冲突：

```html
<div id="footer">
please contact us at email.support@github.com
</div>
```

上述的冲突解决方案仅保留了其中一个分支的修改，并且 `<<<<<<<` , `=======` , 和 `>>>>>>>` 这些行被完全删除了。 

**在你解决了所有文件里的冲突之后，对每个文件使用 `git add` 命令来将其标记为冲突已解决。** 一旦暂存这些原本有冲突的文件，Git 就会将它们标记为冲突已解决。

重新提交：

```
$ git add index.html
$ git commit -m "合并issue3分支"
# On branch master
nothing to commit (working directory clean)
```

**done！**

但是这种合并不是fast-forward合并，而是non fast-forward合并。

#### 2.1.1 使用rebase强制进行fast-forward合并

> 参考：[猴子都能懂的git入门](https://backlog.com/git-tutorial/cn/stepup/stepup1_4.html)

merge合并有两种情况：

1. 分支开出去之后，master部分没有更新，这时候执行merge操作就直接是fast-forward（快进）合并：**这种情况下，分支删除之后会丢掉分支信息。**（如下图，回看Head之前的信息，就不知道X/Y的修改是来自于bugfix分支的提交；假如有多人修改，就不知道具体是谁在哪个分支上引入了，这个分支也会丢失，在看不到该分支历史）

![fast-forward合并](https://backlog.com/git-tutorial/cn/img/post/stepup/capture_stepup1_4_2.png)

2. 分支开出去后，master部分有新的更新，这是后执行merge操作就变成了non fast-forward合并

![结合了两个修改的合并提交](https://backlog.com/git-tutorial/cn/img/post/stepup/capture_stepup1_4_4.png)

这种方式就可以查明每个分支的具体操作。

但是这样，在分支较多的情况下，会让 git tree变得很复杂。（

**所以，可以采用git rebase的方式进行操作：**

![使用rebase合并分支](https://backlog.com/git-tutorial/cn/img/post/stepup/capture_stepup1_4_7.png)

首先，rebase bugfix分支到master分支, bugfix分支的历史记录会添加在master分支的后面。如图所示，历史记录成一条线，相当整洁。

> Merge和rebase都是合并历史记录，但是各自的特征不同。
>
> - **merge**
>   保持修改内容的历史记录，但是历史记录会很复杂。
> - **rebase**
>   历史记录简单，是在原有提交的基础上将差异内容反映进去。
>   因此，可能导致原本的提交内容无法正常运行。
>
> 您可以根据开发团队的需要分别使用merge和rebase。
> 例如，想简化历史记录，
>
> - 在topic分支中更新merge分支的最新代码，请使用rebase。
> - 向merge分支导入topic分支的话，先使用rebase，再使用merge。

#### 3.1 删除远程标签时遇到已删除标签被其他同学同步上来

- 起因： 由于每次上线都会打一个标签，因此标签库存在多个标签。想要删除全部的无效标签。 结果执行完毕删除远程标签和删除本地标签后。 发现其他同事再次推送的时候， 删除的那些标签又莫名其妙的回来了。
- 原因: 这是因为其他同事的本地标签没有清理，这时候就必须要其他同事全部都要清理本地的标签。 (很显然这行不通，很难。)
- 解决办法：
  - 使用 `git tag -l | xargs -n 1 git push --delete origin` 命令一条一条的删除远程仓库. ( `xargs -n 1`表示将参数按1个分组，相当于`xargs -n1` )
  - 然后再用 `git tag -l | xargs git tag -d` 清理本地仓库。 (此方法楼主已经测试并且可用) 。

### 2、git清除未跟踪的文件

在利用 `git` 工作时，工程目录下经常会出现一些未跟踪文件，虽然 `git` 支持通过 `.gitingore` 文件添加一些忽略文件类型和文件目录。但有时需要清理一些临时文件和自动生成的文件，手动删除显得太麻烦，这时你可以利用 `git clean` 命令来帮你完成这项操作。

其中几个主要参数用法如下：

```text
-d   # 删除未跟踪目录以及目录下的文件，如果目录下包含其他git仓库文件，并不会删除（-dff可以删除）。
-f   # 如果 git cofig 下的 clean.requireForce 为true，那么clean操作需要-f(--force)来强制执行。
-i   # 进入交互模式
-n   # 查看将要被删除的文件，并不实际删除文件
```
