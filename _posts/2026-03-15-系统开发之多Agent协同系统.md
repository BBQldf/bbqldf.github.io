# 系统开发之多agent协同系统

> 多Agent系统(Multi-Agent System, MAS)是由多个自治智能体组成的分布式系统，每个Agent具备自主决策、环境感知和交互协作能力。现代MAS架构已从单一Agent架构演进为分层协同体系，典型结构包括：
>
> - **感知层**：负责环境数据采集和预处理
> - **决策层**：包含认知型、反应式和混合型Agent
> - **执行层**：任务具体实施单元
> - **协调层**：全局任务调度和冲突解决
>
> 常见的采用"管理Agent-专业Agent"的等级结构，管理Agent负责任务分解和协调，专业Agent专注特定领域任务执行



# 一、一句话总结

对比单agent架构，一句话总结，Multi-Agent是由多个相互作用的智能体（Agent）构成的分布式系统，通过协作、竞争或协商机制完成复杂任务，其中每个 Agent 具备**独立感知、决策和执行能力，同时通过通信协实现信息共享与协同优化。**

这里面提到的几个多智能体系统的核心组件，和Claude Code的官方文档基本一致，核心就是：智能体、环境

和互动机制。

* **Agent**

这些是系统内活跃的决策实体。每个智能体都具有一定程度的自主性，这意味着它可以独立工作、感知本地环境，并根据其目标和可用信息做出选择。智能体可以是软件程序、聊天机器人、实体机器人、无人机、传感器，甚至是人类。它们是具有特定角色和功能的独立实体。

* **Profile（Role）：**

角色是对智能体的详细描述，包括其特征、能力、行 和限制条件。在多Agent系统中，每个Agent都有自己的角色、技能和行动定义，都是为满足特定目标而量身定制的。例如在软件开发中，Agent可能担任产品经理和工程师的角色，每个角色都拥有对应专业知识并在指导开发过程。其实就是一个agent.md文件～

* **环境**

这是LLM-MA系统运行并进行交互的具体背景或设定。环境可以是虚拟的，例如模拟世界或网络，也可以是物理的，例如机器人智能体的工厂车间。它是智能体感知信息、执行动作以及接收反馈的基础，智能体通过观察与环境互动，理解周围情况，做出决策，并从行动的结果中学习。智能体执行动作后会从环境中获得反馈，这可以帮助智能体改进策略。并且一部分的环境是需要共享的。

* **通信协议和语言**

为了协同工作，智能体需要相互通信。通信协议是它们交换信息的规则。其中包括消息的格式（例如使用JSON 或 XML）以及消息的发送方式（例如使用 HTTP 或 MQTT）。agent之间是可以进行通信，并且完成多轮A2A的对话的。





# 二、什么是多agent

> 从现代AI系统开发的视角来看，常见的Agent架构类型主要围绕**“规划（Planning）”、“记忆（Memory）”和“工具使用（Tool Use）”**这三个核心组件来设计。

其实现在市面上有很多类型的single/muli-agent架构，比较常见的有6类（每一类都有其独特的使用场景，其实无关乎谁对谁错）：

1. **单智能体（Single Agent）**：无需与其他实体进行直接交互或通信，一个agent即可完成所有任务
   1. 适用于任务简单，few-shot/one-shot即可搞定（比如，今天天气怎么样？）
2. **网络型（Network）**：多个智能体以去中心化的方式直接交互，通信通常以点对点形式进行，支持信息、资源乃至任务的共享。
   1. 这一块用的少，管理通信开销并确保决策的一致性，仍是不小的挑战
3. **监督者型（Supervisor）**：存在一个专用的 “监督者智能体”，负责监督并协调一组下属智能体的活动。监督者充当通信、任务分配与冲突解决的中心枢纽。（<font color='red'>最常见的场景</font>）
   1. 主要就是有一个super-agent来负责规划，然后各个sub-agent去执行
   2. 我会在后面单独说一下这个场景的演化
4. **监督者工具型（Supervisor as a Tool）**：监督者的角色不再侧重于直接指挥与控制，而是更多地为其他智能体提供资源、指导或分析支持
   1. 这是目前claude code、codex等cli产品最常用的方式，但是有点不一样是，cc强调把其他能力做成plugins、skills
5. **层级型（Hierarchical）**：这种结构非常适合处理可拆解为子问题的复杂任务，且每个子问题可由层级中的特定层面负责。
   1. 类似于监督者性的一种升级演化版
6. **定制化型（Custom）**：允许构建独特的关系与通信结构，这些结构可精准适配特定问题或应用场景的需求。
   1. 往往源于三类需求：针对特定性能指标进行优化、应对高度动态的环境，或是将领域特定知识融入系统架构。

# 三、为什么多agent

但是事实上没有这么复杂：

## **ReAct 架构：**

核心是“思考-行动-观察”的闭环。Agent 接收到任务后，不会立刻给出答案，而是先**思考（Thought）**需要做什么，然后采取**行动（Action）**调用外部工具，拿到工具返回的**观察结果（Observation）**后，再次进入思考。这个循环会一直持续，直到 Agent 认为已经收集到足够的信息来回答用户。

- Agent 运行过程：

  ```
  用户指令：“帮我查一下我的顺丰快递到哪了？另外北京今天下雨吗，我出门要带伞吗？”
  
  Thought (思考)：用户问了两个问题，我需要先查快递，再查天气。
  Action (行动)：调用 Express_API(单号="SF12345")。
  Observation (观察)：API返回“已到达北京朝阳集散中心”。
  Thought (思考)：快递查完了，现在查北京天气。
  Action (行动)：调用 Weather_API(城市="北京")。
  Observation (观察)：API返回“晴，25度”。
  Thought (思考)：信息收集完毕，可以回答用户了。最终输出结果。
  ```

- 优点：开发成本极低、灵活度高，满足70%的一问一答的需求

- 缺点：

  - **上下文窗口爆炸**（因为每一次“思考-行动-观察”都会被追加到历史对话中，步骤一多，Prompt 就会变得极其冗长，导致模型遗忘前面的信息）
  - **容易陷入“死循环”**（如果 API 报错或者返回了意料之外的数据，基础模型很容易反复调用同一个工具，导致 Token 消耗殆尽）
  - **不适合长线任务**（让它写一本小说或者做一个复杂项目，它走到一半就会忘记最初的目标，然后开始“胡言乱语”，也就是产生“幻觉”）

## **Plan-and-Execute 架构图 (计划与执行)：**

当任务非常复杂（比如“帮我调研10个竞品并写一份对比周报”）时，单体 ReAct 容易陷入死循环。这时候就需要“先规划，后执行”的架构。

   - Agent运行过程：

     ```
     用户指令：“帮我写一份 2026 年全球新能源汽车电池市场的深度调研报告，要求包含核心玩家对比和未来趋势预测，字数不少于 3000 字。”
     
     规划器 (Planner) 拿到任务，拆解为 4 个子任务：
     任务 1：搜索 2025-2026 年全球电池市场规模数据。
     任务 2：搜索宁德时代、比亚迪、松下等核心玩家的最新财报和技术路线。
     任务 3：整理数据，撰写“市场现状”与“玩家对比”章节。
     任务 4：结合前文撰写“趋势预测”并汇总全文。
     执行器 (Executor) 依次接单。它在执行“任务 2”时，只专注于搜索核心玩家信息，根本不需要知道任务 1 和任务 4 的存在。
     评估器 (Evaluator) 检查任务 2 的结果，发现漏了“LG新能源”，打回给执行器要求补充。补充完成后，再推进到任务 3。
     ```

   - 系统被拆分为三个核心大脑：

     - **规划器**负责把大目标拆成 1、2、3 步；
     - **执行器**像一个没有感情的打工人，只负责完成当前的单步任务；
     - **评估器**负责检查结果，如果做错了就打回重新规划，做对了就继续下一步。

   - 优点：

     - **成功率高**（不要小看这一点哦！现在一个任务的完成度虽然很高了（最开始可不是～），但是这个是一次质的飞跃，大目标被拆解成了小目标，极大降低了单次大模型推理的难度）
     - **节省 Token，避免上下文污染**（执行器在干活时，不需要携带全局的冗长历史记录，只需要当前子任务的 Prompt，专注度极高）
     - **容错率高**（这种场景下面，一定会有一个评估器（这个时候还不能叫监督器），用来进行任务重试，不至于卡顿）

   - 缺点：

     - **延迟极高 (High Latency)**：因为需要反复规划、执行、评估，用户发完指令后，可能需要等几分钟甚至十几分钟才能看到最终结果
     - **API 成本高昂**：原本 ReAct 可能 3 次大模型调用就搞定了，这个架构可能需要调用 20 次大模型（亲身经历）
     - **规划器的单点故障**：如果一开始 Planner 拆解的计划就是错的（方向偏了），后面的 Executor 干得再好也是南辕北辙



**有了PAE架构之后，其实很明显的就会进化出一个想法，“打工人”能不能更加专业化一点？**（专业名词叫“关注点分离”）

## **Multi-Agent Supervisor 架构图：**

这是目前企业级系统开发（如 LangGraph、AutoGen 框架）中最推崇的架构。它模拟了人类公司的组织架构。

```
用户指令：“帮我用 Python 写一个贪吃蛇游戏，要有计分板，并且代码必须没有 Bug。”

Agent 运行过程：
主管 Agent (项目经理) 接收需求，分析后决定先让“程序员 Agent”写代码。
程序员 Agent 拥有专门写代码的 Prompt（比如“你是一个资深Python工程师...”），它写完代码后，提交给主管。
主管 Agent 拿到代码，转交给“测试员 Agent”。
测试员 Agent 拥有运行代码的工具（Python解释器），它运行后发现贪吃蛇撞墙没有判定游戏结束（报错了），于是把 Bug 报告发给主管。
主管 Agent 把 Bug 报告打回给“程序员 Agent”要求修改。
```

**优点：**

1. **专业的事交给专业的 Agent (关注点分离)**：每个 Agent 的系统提示词（System Prompt）和配备的工具都是高度定制的。程序员不需要懂测试，测试员不需要懂写代码，这让大模型的表现更加精准（避免了让一个模型既当爹又当妈导致的混乱）。
2. **极强的扩展性**：如果明天你想加一个“UI设计师 Agent”来画游戏的图标，只需要在主管的路由表里加一行配置即可，系统架构无需大改。
3. **突破单体能力天花板**：通过多角色的“辩论”或“交叉验证”，可以解决极其复杂的跨领域问题，输出质量远超单个大模型。

**缺点：**

1. **开发和调试难度地狱级**：多个 Agent 之间的通信协议、状态同步、死锁问题（比如程序员和测试员互相不服，陷入无限争吵）非常难处理。
2. **主管 Agent 的能力瓶颈**：整个系统极其依赖“主管”的智商。如果主管模型不够聪明，分发错了任务，或者无法正确总结各方意见，整个团队就会瘫痪。
3. **资源消耗巨大**：多 Agent 协同往往伴随着海量的内部对话（Token 消耗），在实际企业应用中，如果不加限制，账单会非常惊人。



在实际的系统开发中，架构的选择往往取决于**任务的复杂度**：

1. **简单问答与单步工具调用**：直接使用基础的 LLM + Function Calling。
2. **需要动态探索的中等复杂度任务**：选择 **ReAct** 架构。
3. **步骤明确的复杂长任务**：选择 **Plan-and-Execute** 架构。
4. **跨领域的企业级复杂系统**：必须采用 **多Agent分层协同（Supervisor）** 架构。

### 当前技术挑战

1. **动态环境适应**：Agent参数实时调整机制
2. **大规模协同**：百万级Agent无缝协作
3. **通信效率**：高并发低延迟通信
4. **安全与隐私**：分布式系统安全机制
5. **可解释性**：黑盒决策过程透明化
